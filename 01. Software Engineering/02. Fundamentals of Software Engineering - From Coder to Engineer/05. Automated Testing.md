# Chapter 5: Automated Testing

> "Quality is not an act, it is a habit."
> — Philosopher Will Durant, paraphrasing Aristotle

When a new year begins, plenty of people set goals around living a healthier lifestyle. Those goals never come from one healthy meal, or one trip to the gym — they come from consistently making good choices, most of the time. Producing high-quality, maintainable software works the same way: it's the *consistent practice* of good habits around writing, reviewing, and testing code — not any single heroic effort — that actually gets you there. And just like a healthy lifestyle, it isn't always easy, and excuses tend to set you back.

The path to consistency isn't smooth, either. Plenty of developers wrestle with real self-doubt — wondering whether they're making the right calls, or whether their approach will hold up to scrutiny. Writing high-quality software is a discipline built through regular practice — but over time, your testing efforts become a safety net that both your teammates *and* your future self will genuinely appreciate. This chapter covers why automated testing matters, the different kinds of tests you'll encounter, and how to actually write them.

## Benefits of Automated Testing

It's natural to feel a little skeptical about writing *extra* code just to verify your *existing* code — it can feel like unnecessary overhead, or even a bit of a resume-padding exercise. It's neither. Automated testing is a genuinely valuable skill, and a real investment in the health of your codebase.

### Acts as Documentation

Joining an existing project can be overwhelming (Chapter 6 covers this directly). Good documentation is genuinely a luxury — if you're lucky, you can lean on lead developers or domain experts for a high-level overview before diving in, but that's not always available, and documentation is often sparse or nonexistent. Your best fallback, in that situation, is a project with genuinely well-written tests.

**Imagine this scenario:** you're asked to fix a payment-processing bug related to a specific card type. Without documentation or tests, you'd be stuck combing through unfamiliar, likely complicated code, or leaning entirely on colleagues for guidance.

**Now imagine instead finding a solid test suite for the payment system** — a `PaymentProcesserTests` class with clearly named tests like:

- `shouldValidateValidCreditCardNumber`
- `shouldProcessCreditCardTransaction`
- `shouldProcessOrangePay`

These test names alone tell you a lot about what the module is supposed to do, and point you toward the exact code involved. A modern IDE lets you jump straight from a test to the specific service it exercises (see Chapter 2's code-reading strategies). A *failing* test points you almost directly at the likely source of a bug.

Suppose you also find a test named `shouldFailWhenCreditCardTypeIsOrangePay` — that single name tells you, immediately, that "Orange Pay" was never actually meant to be supported. That's exactly the kind of insight that lets you fix the real issue quickly, update the front end accordingly, and even improve the documentation as you go.

This is the core idea: **tests double as documentation**, helping you get oriented in a codebase far faster than you could otherwise. Investing in a strong test suite pays off in code quality, fewer production issues, and faster onboarding for the next person (possibly future-you).

### Improves Maintainability

Writing genuinely maintainable code is a skill that takes years to build (see Chapter 3). Writing tests *first* is a bit like thinking before you speak — it forces you to plan and structure your ideas before committing to an implementation. A beginner's main goal is usually just "make it work" — a fine starting point, but writing tests pushes you to notice design problems much earlier than you otherwise would.

**Consider this example — a class responsible for publishing blog posts (details omitted for brevity):**

```java
public class BlogPostController {
    public void publish(Post post) {
        // save blog post to the database (flip isPublished to true)
        // log blog post has been published
        // send email to subscribers about the new post
    }
}
```

At a glance, this seems perfectly reasonable. But try to actually write a test for this `publish` method, and the cracks appear almost immediately — this one method is quietly responsible for:

- Talking to a database
- Logging
- Sending emails

That's a direct violation of the **single responsibility principle**: "a class should have only one reason to change." A change to how emails get sent has no business also risking a change to how blog posts get saved — yet here, they're tangled together in one method.

**Just by thinking about how you'd test this code, you naturally uncover its design flaw.** The fix: pull each responsibility (persistence, logging, email) into its own dedicated class — each independently testable, and each far easier to maintain going forward.

Writing tests doesn't just verify that code *works* — it actively pushes you toward better design, surfacing structural problems well before they calcify into real production issues.

### Boosts Your Confidence

Of all the benefits of automated testing, this one might matter most: **the confidence to code freely.** Software development is inherently iterative — you write, experiment, and refactor, constantly. Without tests, you tend to code defensively, avoiding anything that might risk breaking something unseen — and that hesitation can quietly discourage genuinely good, creative solutions. A solid test suite acts as a safety net, fundamentally changing how you approach the work: you can make changes, try new approaches, and refactor boldly, trusting that your tests will catch any real problems.

Changing and refactoring code is inherently stressful — you rarely have a *complete* mental model of the whole application, and side effects can be genuinely hard to predict in advance. Working without tests is a bit like climbing a mountain without ropes: thrilling if nothing goes wrong, catastrophic the moment something does.

**A robust test suite lets a team:**

- Refactor with genuine confidence
- Confirm that changes haven't broken existing functionality
- Catch potential problems before they ever reach production

**Writing and maintaining a real, comprehensive test suite also helps you:**

- Build a deeper understanding of how the system actually behaves
- Catch bugs early, while they're still simple and cheap to fix
- Reduce the odds of accidentally reintroducing an old bug (a "regression")
- Confirm features actually meet their intended specification, and keep code quality consistent over time

This doesn't just improve code quality — it directly builds your confidence as a developer. You can add features, refactor, and make changes with real assurance that your tests will flag anything that actually goes wrong. Confidence in coding comes from a mix of knowledge, experience, and the right tools — automated tests are one of the most powerful tools in that mix, throughout your entire career.

### Leads to Consistency and Repeatability

Developers used to rely heavily on hours of manual testing — following a written checklist of steps meant to stress-test a new feature. Even when the checklist was followed faithfully, manual testing is inherently error-prone, hard to repeat exactly, and extremely labor-intensive. Human beings simply aren't built to execute the exact same sequence of steps, over and over, without any variation creeping in.

**Automated tests, by contrast, are perfectly consistent and repeatable.** A test script runs the exact same steps every single time, with no accidental omissions. These tests can run in seconds, at the click of a button, or automatically as part of every build. **Regression tests** specifically confirm that existing functionality keeps working correctly as the codebase continues to change.

With a solid test suite in place, you get consistent, repeatable verification regardless of environment — and because automated tests can run frequently and cheaply, they provide continuous protection against old bugs quietly resurfacing, while also catching *new* bugs introduced by fresh changes. When building a new feature, this lets you skip expensive, error-prone manual testing almost entirely, and lean on automation instead.

## Types of Automated Testing

Now that the *why* is covered, let's look at *what* you should (and shouldn't) actually be testing. Automated testing spans several distinct categories — UI tests, end-to-end tests, integration tests, and unit tests. A common, useful way to think about this is the **testing pyramid**, originally created by Mike Cohn.

**The three main layers, bottom to top:**

- **Unit tests** — cover individual components or functions in isolation from the rest of the system, confirming each piece works correctly entirely on its own.
- **Integration tests** — verify how several components or modules actually work together as a cohesive unit.
- **End-to-end (E2E) tests** — cover the entire application, starting from the user interface and running all the way down through the backend.

**The pyramid shape reflects a real, fundamental trade-off.** As you move *up* the pyramid, tests get slower to run, more expensive to maintain, and more prone to breaking for unrelated reasons. A unit test might run in microseconds; an end-to-end test might take several minutes. Multiply that by hundreds or thousands of tests, and the difference becomes enormous — which is exactly why the pyramid has far more unit tests at the base than end-to-end tests near the top.

### Unit Tests

Unit tests form the base of the pyramid, and typically make up the largest share of any healthy test suite. Think of a unit test as a **contract** your code has to fulfill — if the code's actual behavior changes, the corresponding unit test should break, immediately flagging that something shifted.

Unit tests examine one specific function or component in complete isolation, confirming that one piece behaves correctly entirely on its own. They act as your first line of defense against bugs, alongside static analysis and code review. They should be fast to write, and fast to run — giving you near-instant feedback while you're actively developing. When a unit test fails, it typically points you directly at a specific function, or even a specific line — making debugging fast and precise. This combination of speed and precision is exactly what makes unit tests so valuable during day-to-day feature development and continuous integration — they let you iterate quickly, with real confidence.

### Integration Tests

An integration test verifies how several different components or modules actually work together as a cohesive whole. Where a unit test deliberately isolates one piece of code, an integration test deliberately examines the *interactions between* several pieces — a broader scope than a unit test, but still narrower than testing the entire system end to end.

Integration tests matter because some bugs only ever appear once individually-correct components are actually wired together — a bug that no unit test, however thorough, could ever catch on its own. This step is essential for confirming the system holds together as a whole, not just piece by piece.

Because of their broader scope, integration tests generally take longer to run than unit tests, and tend to run less frequently as a result. When one fails, it usually takes more investigation to pin down exactly *which* interaction between components actually caused the problem.

### End-to-End Tests

An end-to-end (E2E) test covers the *entire* application — starting at the user interface, and running all the way through to the backend systems. These tests simulate real user behavior, confirming the whole system genuinely works together the way a real user would experience it.

Because of how much ground they cover, E2E tests are typically slow, and demand real time and resources to run — which is exactly why you'll generally find far fewer of them than unit or integration tests, often run on a scheduled basis rather than on every single commit. They can also be **brittle** — more prone to false failures triggered by something trivial (a minor UI element changing) rather than an actual functional problem. A failing E2E test often demands significant debugging effort just to pin down what actually went wrong.

Despite these costs, E2E tests are genuinely invaluable — they provide the strongest possible confidence that the whole system, end to end, actually works together the way real users will experience it.

### What Mix of Tests Should You Actually Write?

The right balance varies project to project — it depends on the application's complexity, its technology stack, and how business-critical it is. Your organization may also have its own standards or benchmarks for coverage and test quality. The right move is tailoring your testing strategy to your specific project's needs, while still keeping a genuinely balanced mix across all three layers of the pyramid.

### What You Should *Not* Test

Comprehensive testing matters — but so does being deliberate about what's actually worth testing. A few guidelines worth internalizing:

- **Don't test language features or framework internals.** Your focus belongs on *your own* code, not on verifying that the language or framework itself works correctly — that's not your job to re-verify.
- **Avoid testing purely generated code** — getters, setters, builder methods, auto-generated data transfer objects (DTOs), and similar boilerplate.
- **Avoid testing private methods directly.** Focus instead on the *public* interface that calls those private methods — if the public behavior is correct, the private implementation supporting it is implicitly covered too.
- **Avoid tests that depend on real external services.** Use mocks for unit tests, and test doubles for integration tests, instead of reaching out to a live third-party system every time your test suite runs.

A well-designed test suite is thorough *and* maintainable — genuinely comprehensive, without becoming unwieldy. Focusing your testing effort on your own code, and skipping what doesn't need it, is how you get there.

## Code Coverage

**Code coverage** measures the percentage of your code that actually gets executed when your test suite runs. A helpful mental image: think of code coverage like the fog-of-war map in a video game. The whole map starts dark; as you explore, the areas you actually walk through become visible. Tests that exercise a given section of code "light it up," proving you've actually been there. High coverage means you've explored most of the map; low coverage means blind spots remain — exactly the kind of place bugs like to hide.

Most modern IDEs, and most languages, support code coverage tooling directly — letting you check your own coverage instantly as you work. Many CI/CD pipelines can also enforce a *minimum* coverage threshold, failing the build (and blocking a PR from merging) if coverage drops below it. Coverage tools analyze a test run and produce reports showing exactly which lines, branches, and functions actually executed — you might see something like "85% line coverage" or "72% branch coverage."

**A crucial limitation to understand: higher coverage does not automatically mean better testing.** Some organizations require a minimum threshold — 80% or 90% coverage, say — before code can ship. Well-intentioned, but this can quietly backfire.

**It's entirely possible to write a test that executes code without meaningfully verifying anything at all** — effectively gaming the coverage number. A test that calls a method but never actually asserts anything useful still *counts* toward coverage, while providing essentially zero real protection against bugs.

**Rather than chasing 100% coverage as a vanity metric, treat it as a diagnostic signal.** Low coverage in genuinely critical business logic is worth real attention. High coverage on simple data classes or configuration objects, on the other hand, might just mean you're spending effort testing things that never really needed it in the first place. Focus on writing tests that verify *meaningful behavior*, and let coverage numbers guide your attention — not dictate it. The actual goal isn't hitting an arbitrary percentage; it's building genuine confidence in the code you're shipping.

## Writing Tests

Time to get hands-on with the actual mechanics of writing tests.

### Getting Started

Whatever language, framework, or meta-framework you're using, you almost certainly have solid testing tools available. This section uses **Java** as a concrete example — the syntax will differ in your language of choice, but the underlying ideas transfer directly.

The most popular testing framework for Java is **JUnit** — added to a project via your build tool's dependency configuration (for example, a Maven POM file). Once JUnit is set up, you're ready to start writing tests.

**Two general philosophies for *when* to write tests:**

- **Test-first**, most commonly seen as **test-driven development (TDD)**, follows a "red-green-refactor" cycle: write a failing test first ("red"), write just enough code to make it pass ("green"), then improve the implementation without changing its behavior ("refactor").
- **Test-last** means writing the tests *after* the functionality already exists.

Both are legitimate, and the right choice often depends on project needs and team preference. Either way, every approach ultimately relies on **assertions** to verify that expected outcomes actually occurred.

> **A note on AI:** writing tests is a genuinely strong use case for AI assistance. Feeding an AI tool examples from your existing test suite helps it pick up your team's preferred style, naming conventions, and standards. AI can help generate test cases, flag edge cases you might have missed, produce routine boilerplate, and even help refactor existing tests for readability — all of which can meaningfully speed up testing while staying consistent with your team's existing patterns.

### Assertions

An **assertion** is a function that verifies some condition is actually true during a program's execution. Java has several assertion libraries, each with its own feature set.

JUnit 5 ships with a wide range of built-in assertions, all available as static methods on `org.junit.jupiter.Assertions`. Here's `assertEquals`, verifying a simple addition — the first argument is the *expected* value, the second is the *actual* value:

```java
@Test
void shouldAdd2Numbers() {
    int expected = 3;
    int actual = 2 + 1;
    assertEquals(expected, actual);
}
```

If `expected` and `actual` match, the assertion — and therefore the whole test — passes. If they don't match, the assertion (and the test) fails. It's worth spending a little time browsing your assertion library's documentation to see the full range of assertions available to you.

### Writing Unit Tests

Remember: unit tests are isolated, and run fast. Imagine a class that performs a couple of basic math operations, each needing its own test:

```java
package com.fose;

public class Operations {
    public double add(double a, double b) {
        return a + b;
    }

    public double subtract(double a, double b) {
        return a - b;
    }
}
```

**Before writing your first test, pay attention to file location.** This class lives in the `com.fose` package. Your test class should live under `/src/test/java`, in that *same* package — this convention is what lets your test framework and build tool automatically discover and run it.

**Following a test-last approach**, start by scaffolding empty test methods for everything you plan to cover:

```java
package com.fose;

import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

class OperationsTest {
    @Test
    void add() {
    }

    @Test
    void subtract() {
    }
}
```

**Next, you need an instance of `Operations` to actually test.** This is commonly called the **system under test (SUT)** — the specific component your current test is focused on:

```java
class OperationsTest {
    private final Operations sut = new Operations();
}
```

**With the SUT in place, fill in the actual test logic** — and make a real effort to cover meaningful edge cases, not just the obvious happy path (you don't want your *users* discovering those edge cases for you):

```java
package com.fose;

import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

class OperationsTest {
    private final Operations sut = new Operations();

    @Test
    void add() {
        assertEquals(5, sut.add(2, 3));
        assertEquals(0, sut.add(-1, 1));
        assertEquals(-5, sut.add(-2, -3));
    }

    // Additional tests omitted for brevity
}
```

**Notice the third assertion**, `add(-2, -3)` — deliberately testing negative numbers, not just the "nice," obvious positive case. Covering a range of realistic inputs like this, rather than just one clean example, is exactly what separates a genuinely useful test from a superficial one.

### Mocking

**Mocking** is a technique for creating a fake, controllable stand-in for a real dependency — sometimes called a "stunt double" — specifically so you can test a piece of code in true isolation.

In this context, a **dependency** is simply any other class the code you're testing (the SUT) relies on to do its job. Mocking is what makes isolated unit testing possible — but it becomes genuinely tricky in *integration* testing, where the whole point is verifying how real dependencies actually behave together. Juggling multiple real dependencies, keeping them in the right state, and managing their interactions can quickly get complicated — a bit like trying to keep several balls in the air simultaneously.

**Example: a `UserService` with two real dependencies** — `UserRepository` (handles reading/writing users to the database) and `EmailService` (handles sending emails):

```java
package com.fose;

public class UserService {
    private UserRepository userRepository;
    private EmailService emailService;

    public UserService(UserRepository userRepository, EmailService emailService) {
        this.userRepository = userRepository;
        this.emailService = emailService;
    }

    public void registerUser(String username, String email) {
        // register user
    }
}
```

**How do you write a genuine *unit* test for `registerUser`, given these two dependencies?** If you use the real `UserRepository` and `EmailService`, you're no longer writing a unit test — you've drifted into integration-test territory, whether you meant to or not.

**This is exactly where a mocking framework comes in.** Instead of a real `UserRepository`, you substitute a *mock* — something that looks like a `UserRepository` from the outside, but isn't actually one, and won't perform any real database operations. It only does whatever behavior you explicitly program it to simulate. This lets you test `UserService` completely in isolation — its dependencies get their own, separate unit tests elsewhere.

**Here's a `UserServiceTest` using Mockito**, a widely used Java mocking framework (the concepts translate directly to equivalent tools in other languages):

```java
@ExtendWith(MockitoExtension.class)
class UserServiceTest {

    @Mock
    private UserRepository userRepository;

    @Mock
    private EmailService emailService;

    @InjectMocks
    private UserService userService;

    @Test
    public void testUserRegistration() {
        String username = "newuser";
        String email = "newuser@example.com";

        when(userRepository.existsByUsername(username)).thenReturn(false);

        userService.registerUser(username, email);

        verify(userRepository).save(any(User.class));
        verify(emailService).sendWelcomeEmail(email);
    }
}
```

**Walking through this line by line:**

- `@Mock` on `userRepository` and `emailService` — tells Mockito to generate fake, controllable versions of these two classes.
- `@InjectMocks` on `userService` — tells Mockito to automatically construct a real `UserService`, wiring in the two mocks above as its dependencies.
- `when(userRepository.existsByUsername(username)).thenReturn(false)` — configures the mock: "when this exact method gets called with this exact input, pretend the answer is `false`" (i.e., simulate "this username doesn't already exist").
- `userService.registerUser(username, email)` — actually runs the method under test.
- `verify(userRepository).save(any(User.class))` — confirms that `registerUser` genuinely called `save()` on the (mock) repository, with *some* `User` object, as part of doing its job.
- `verify(emailService).sendWelcomeEmail(email)` — confirms a welcome email was genuinely triggered, with the correct email address.

Notice that this test never touches a real database or sends a real email — it only confirms that `UserService` *correctly orchestrates* its dependencies, exactly the kind of narrow, fast, isolated check a true unit test is meant to provide.

### Writing Integration Tests

Now consider testing how `UserService` genuinely interacts with its real dependencies, rather than mocked stand-ins. Start with a new test class, `UserServiceIntTest` — the `Int` in the name clearly signals "this is an integration test," distinct from the earlier unit test.

Since `UserService` depends on both `UserRepository` and `EmailService`, you'll need *real* instances of each. In a real application, a dependency-injection framework often provides these automatically — but here, they're constructed by hand for clarity:

```java
class UserServiceIntTest {

    private UserRepository userRepository;
    private EmailService emailService;
    private UserService userService;

    @BeforeEach
    void setUp() {
        userRepository = new UserRepository();
        emailService = new EmailService();
        userService = new UserService(userRepository, emailService);
    }
}
```

**`@BeforeEach` runs this setup fresh before every single test** in the class, ensuring each test starts from a clean, predictable state.

With genuine (non-mocked) dependencies wired up, you can now write integration tests that exercise the real interaction between them:

```java
@Test
void shouldNotRegisterUserWithUsernameOfUser() {
    // Test invalid user registration
}

@Test
void shouldRegisterValidUser() {
    // Test valid user registration
}
```

*(Note: the original source material had these two test descriptions swapped relative to their comments — corrected here so each comment matches what its test name actually implies.)*

### Writing End-to-End Tests

E2E testing here focuses on the backend API directly, and requires a genuinely running environment to test against — in this example, Spring Boot with an embedded Tomcat server. The goal: confirm a real server actually starts, actually receives requests, and returns the correct responses.

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@AutoConfigureMockMvc
class UserRegistrationE2ETest {

    @Autowired private MockMvc mockMvc;
    @Autowired private UserRepository userRepository;
    @Autowired private EmailService emailService;
    @Autowired private ObjectMapper objectMapper;

    @BeforeEach
    void setUp() {
        userRepository.deleteAll();
    }

    @Test
    void testUserRegistration() throws Exception {
        String username = "testuser";
        String email = "testuser@example.com";
        User request = new User(username, email);

        // Perform the request
        mockMvc.perform(post("/api/users/register")
                .contentType(MediaType.APPLICATION_JSON)
                .content(objectMapper.writeValueAsString(request)))
                .andExpect(status().isOk());

        // Verify that the user was saved in the database
        Optional<User> savedUser = userRepository.findByUsername(username);
        assertTrue(savedUser.isPresent());
        assertEquals(email, savedUser.get().email());

        // Additional verifications (welcome email sent) omitted for brevity
    }
}
```

**What this test actually does, step by step:**

1. **`@SpringBootTest(webEnvironment = ... RANDOM_PORT)`** — spins up a real, running instance of the application on a randomly assigned port, rather than mocking anything away.
2. **`@AutoConfigureMockMvc`** and the `@Autowired MockMvc mockMvc` field — provides Spring's built-in support for simulating real HTTP requests against the running server.
3. **`@BeforeEach` / `userRepository.deleteAll()`** — wipes the database clean before every test, so each one starts from a known, predictable state.
4. **`mockMvc.perform(post(...))`** — actually sends an HTTP `POST` request to `/api/users/register`, with a JSON body built from a `User` object, and confirms the response status is `200 OK`.
5. **The final block** — queries the *real* database afterward, confirming the new user genuinely got saved, with the correct email address attached.

This test walks the *entire* path — from an actual HTTP request, through the real application logic, all the way down to a real database — confirming the whole system genuinely works together end to end, not just in isolated pieces.

## Wrapping Up

This chapter covered the full landscape of automated testing and its central role in software development — how tests double as documentation, improve code quality, and act as a genuine safety net for fearless refactoring. From unit tests to end-to-end tests, each layer plays a distinct, necessary role in confirming your software actually works as intended.

Just like maintaining a healthy lifestyle, writing good software comes down to consistent, positive habits, not one-off heroics. A robust test suite is a bit like having a trainer and a clear workout plan — it gives you the confidence to move forward, experiment, and make changes without constant fear of quietly breaking something. That confidence is what lets you keep improving your codebase and adapting to new requirements, without being held hostage by fear of the unknown.

As you move forward in your career, work toward making testing a genuine habit. It can feel like extra effort at first — but the long-term payoff, in code quality, maintainability, and your own professional growth, is enormous. Quality isn't a one-time act — it's a consistent practice. Embracing automated testing isn't just about catching bugs; it's about becoming a more effective, confident, and skilled engineer.

## Putting It into Practice

Testing isn't a skill you master overnight — it's built through consistent, repeated practice. Pick one or two of the following that resonate with your current situation, rather than trying to adopt all of them at once:

- **Learn from open source.** Pick a library you already use and admire. Read through its tests. What do they teach you about the library's functionality that you didn't already know? How do their testing habits differ from your own? Consider adopting a few of their strategies yourself.

- **Use tests as documentation.** The next time you're assigned to work in an unfamiliar part of a codebase, look for existing tests *first*. If they're missing or inadequate, write new ones as you learn — turning your own exploration into something the next person can benefit from too.

- **Improve existing codebases incrementally.** If your project's test coverage is genuinely lacking, don't try to fix it all at once. Challenge yourself (and your team) to write just one new test a day.

- **Start new features with tests.** The next time you're assigned a new feature, begin by writing its tests first. This alone tends to clarify requirements and steer you toward better design from the very start.

- **Address bugs with tests.** When fixing a bug, first write a test that reproduces it — a test that should *fail* against the current, broken code. This is also often a natural opportunity to refactor the surrounding code for clarity along the way.

- **Advocate for testing.** If you find a bug that proper tests would have caught, add the test that would have caught it, and use the moment as a genuine learning opportunity for the team — including a conversation with your manager about how better coverage could prevent similar issues going forward.

- **Just practice.** The only real way to improve at writing tests is to keep writing them. Code katas (introduced in Chapter 3) are a great structured, low-stakes way to build this specific skill deliberately.

Building a genuinely strong test suite is an ongoing process, not a finish line. Every small step toward better testing habits compounds into real, lasting improvements in your codebase's quality and maintainability.
