# Chapter 2: Reading Code

> "Code is read much more often than it is written."
> — Guido van Rossum, creator of Python

Despite how coding is usually taught, software engineers spend far more time *reading* code than *writing* it. Most beginner courses jump straight into writing — core syntax, language idioms — without ever teaching you to *read* first, the way you'd never try to learn Polish or Portuguese by skipping straight to writing essays in it. And while almost every school project starts from a blank slate, working developers are almost always operating inside code that's taken years to become what it is today (a topic explored further in Chapter 6).

With the rise of AI-assisted and chat-based coding tools, reading code is only going to become *more* important, not less. Whether you like it or not, you will constantly work with code you didn't write. The good news: there are concrete techniques to help you get your bearings quickly when you land in unfamiliar territory. This chapter covers why reading code is genuinely hard, and gives you practical tools to make it easier.

## The Challenge of Working with Existing Code

However you learned to code, you probably spent most of that time in **greenfield development** — starting completely from scratch, free of any baggage from prior work. In your actual career, though, opportunities to build something from a totally blank editor are rare. Most of your time as a working engineer will be spent on **brownfield development** — working within the boundaries of an existing codebase, dealing with legacy code as a daily reality.

### Legacy Code by Any Other Name…

Reading old code is a task most developers would rather avoid, and the very term "legacy code" is rarely used as a compliment. Definitions vary wildly — some call it code without proper tests, some call it code with *too many* tests, some just mean "code I didn't personally write."

But it's worth resisting the urge to dismiss an existing application just because it's old. If a product has delivered real business value for years, and kept earning continued investment, that's genuinely something worth respecting. A more generous framing: call it **heritage code**, or simply **existing code**. (Chapter 6 covers this idea in much more depth.)

When you're asked to solve a problem inside existing code, you're actually solving **four separate problems at once**:

1. **Understanding the business problem itself.** The domains software engineers work in keep getting more demanding — "software eating the world" means engineers increasingly tackle genuinely complex business challenges, since a lot of the easy problems have already been solved.

2. **Seeing the problem through the eyes of whoever came before you** — often the hardest part of the job. Every developer has their own habits and tendencies. Maybe your predecessor leaned heavily on a functional programming style you're less comfortable with, or used a design pattern or library that's new to you. Agree with their choices or not, there's rarely only one valid way to solve a given problem.

3. **Recognizing when the existing abstraction level is simply wrong.** Maybe the code is modeled too generically, or misses an important nuance of the business domain. Maybe two distinct concepts got conflated, or a favorite pattern got forced onto a problem where it didn't really fit. Sometimes a real refactor is genuinely needed just to make the code understandable again.

4. **Digging through layers of accumulated technical debt**, much like an archaeologist peeling back layers of history. Languages and best practices evolve constantly — what was considered the "right" approach when this code was written may now be considered an anti-pattern. You can often roughly "carbon-date" a piece of code just by noticing which frameworks or language features are (or aren't) being used. Frameworks like Spring, for example, have evolved for years, frequently supporting multiple valid approaches to the same problem at once — and "the right way to do it" keeps shifting underneath you.

It doesn't help that you're almost always dealing with **patches layered on top of patches.** Maybe an earlier developer didn't fully understand the problem, or wasn't aware of a newer language feature that would have simplified things enormously. Maybe someone leaned heavily on an AI coding assistant and generated a huge volume of code quickly, without much scrutiny. Combine that with the usual "fix it fast" pressure, and you can easily lose an entire afternoon just deciphering a single method.

Working with existing code is genuinely technically challenging — which explains why so many developers dread it. But the difficulty isn't only technical. Some of it comes from how our own brains work against us.

### Cognitive Biases

When reading someone else's code, it's easy to unfairly judge it against your own work. (You don't write *bad* code, right?) More than once, even the authors of this book have grumbled some unflattering variation of "what idiot wrote this?" — only to discover, moments later, that the "idiot" in question was themselves, a few years earlier. And honestly, if you look back at code you wrote years ago and feel a little embarrassed by it, that's actually a *good* sign — it means you've genuinely grown since then.

Two specific cognitive biases work against you here:

**The IKEA effect** — the tendency to overvalue things you built yourself. One study found people were willing to pay 63% more for a product they'd assembled themselves compared to an identical, professionally assembled version. (Think of pick-your-own-apple orchards: you pay a premium to do the picking yourself — not because it's cheaper, but for the experience, and the sense of ownership over choosing the best fruit.) In software, this shows up as developers having strong, often unshakeable opinions about the "right" way to do things — usually whatever way they'd have done it themselves.

**The mere-exposure effect** — the tendency to prefer whatever you're already familiar with. This is a big driver of the language dogmatism you'll see among developers: an instinct to treat "the first language I learned" as the natural baseline for everything else. (A telling anecdote: when Java introduced lambda expressions, someone on a Java mailing list asked why the language needed these "new-fangled Lambdas" — not realizing lambdas aren't a new concept at all, and were actually part of Java's *original* design plans.)

This kind of tool provincialism is exactly what Paul Graham describes in his essay "Beating the Averages." Graham argues programming languages exist on a kind of power continuum — but it's genuinely hard to *see* why one language is more powerful than another from inside your own experience. He illustrates this with a hypothetical language called "Blub," and an imagined productive "Blub programmer." Looking *down* the power continuum, the Blub programmer only sees languages missing features they use constantly, and can't imagine why anyone would settle for something so limited. Looking *up* the continuum, they see a pile of unfamiliar features Blub doesn't have — and, since they've never needed those features to be productive in Blub, they can't imagine why anyone else would need them either.

**The practical takeaway:** watch for these biases both in yourself and in your teammates. If a colleague seems oddly, stubbornly attached to a specific tool or approach, it might be worth gently wondering (to yourself, at least) whether the IKEA effect or the "Blub paradox" is at play. And turn that same lens on your own assumptions too — it's very easy to be the Blub programmer without realizing it.

## Approaching Unfamiliar Code

You'll encounter existing, unfamiliar codebases constantly throughout your career — so how do you get up to speed without losing your mind?

**Start with your teammates.** A basic project walkthrough should be part of any proper onboarding.

**Read the documentation.** Many projects have a README that helps orient you; others rely on wikis or internal sites for a concise overview. Look for **architecture decision records (ADRs)** too — these capture invaluable context, especially the "why" behind a decision, which tends to vanish the moment everyone's attention moves on to the next fire drill. A few minutes with good docs can save you hours in the debugger.

**Read the project's coding standards**, if they exist — this primes you to recognize the patterns you'll keep running into as you explore.

**If documentation is out of date, update it as you learn.** If it doesn't exist at all, consider starting your own as you go — this is the Golden Rule from Chapter 1 in action: building this documentation helps *you* learn the project right now, and helps every developer who comes after you.

As you document, favor lightweight, low-effort notes that answer a handful of core questions:

- What does this service actually do?
- How does it work, at a high level?
- What does it depend on?
- How do you actually run it?

You might be thinking: *documentation always drifts out of sync with the code eventually.* And that's a fair concern — but it doesn't have to be inevitable. The most reliable way to keep documentation honest is to let your **tests** do a lot of that documenting for you. Behavior-driven tests, written well, can effectively become **executable documentation** — a topic covered in more depth in Chapter 5.

> ### Metrics Can Mislead
>
> **Code coverage** — how much of your codebase actually runs during your test suite — can be a genuinely useful metric. But there's no silver bullet in software, and it's entirely possible to have serious problems even at 100% coverage.
>
> A friend of the authors once joined a project plagued by regressions on nearly every release. Getting up to speed on the code, he asked the tech lead whether there were tests. The tech lead proudly answered, "Yes — we're right around 92% code coverage." Impressed, but still puzzled by the constant regressions, he kept digging.
>
> Looking closer at the actual test code, he noticed something alarming — a pattern that, at first, he assumed was just a rare outlier. It wasn't. It turned out to be endemic across the whole codebase. He went back to the tech lead: "I couldn't help but notice your tests don't actually have any assertions in them." The tech lead's response? Repeating the 92% coverage statistic.
>
> **The lesson:** be skeptical of any single metric — numbers can mislead badly if you don't look at what's actually behind them. But don't let that skepticism curdle into dismissing the underlying *practice* either — a metric that's become empty ceremony (tests that run but check nothing) isn't a reason to abandon testing; it's a reason to look harder at what you're actually measuring. Teams should periodically challenge their own habits and metrics, and be willing to change course when something clearly isn't delivering the value it's supposed to.

## Software Archeology

Once you've talked to the team and skimmed whatever documentation exists, it's time to actually open your editor and start digging — practicing a bit of "software archeology."

**Look at the code's structure.** How is it organized? Some languages have first-class packaging constructs; others rely purely on convention. Is this a monolith, or a distributed system with dozens (or hundreds) of separate services? What business/domain concepts show up directly in the code? Read the tests — what do they tell you about the intended behavior? With that in hand, does a given class's *purpose* actually make sense to you?

**If the intent still isn't clear, dig further.** Modern editors make it trivial to see everywhere a given function is called from, letting you work *backward* from usage to understanding. Callers of a function are often the fastest way to understand what it's actually for. Following that trail can lead you all the way back to an entry point — an HTTP endpoint, for instance — connecting a specific user action all the way down to the code that handles it.

**Then, run the application.** What does it actually do? Pick something concrete — a specific button in the UI, a specific parameter in an API call — and trace it back into the code. Skim the open issue list; picking one specific bug or feature and following its trail through the code is often a great way to build understanding. Look for a clear landmark — if you know a particular action should update the database, go find exactly where that happens. Step through the code with your debugger: did it behave the way you expected? Did it take you down a completely different path than you assumed? What you're really doing, piece by piece, is building a **mental model** of the system — loading the application into your own head.

### A Worked Example: Spring PetClinic

To make this concrete, consider the **Spring PetClinic** sample application — a well-documented, widely used example Java/Spring app. Even without deep Java or Spring expertise, exploring it is fairly approachable.

After cloning and running the app, you'll notice it can search for pet owners. Poking through the owner-related templates, you'll spot one helpfully named `findOwners.html`, which references an `/owners` endpoint. Searching the codebase for `/owners` turns up plenty of matches — but a bit of intuition points you toward this annotation:

```java
@GetMapping("/owners")
```

...found on the `processFindForm` method inside `OwnerController`.

**Try it yourself:** set a breakpoint on that method, trigger a search from the browser, and watch what happens in your debugger. If your first guess was wrong, that's fine — just repeat the process. Eventually, you'll make the right connection, and each step you retrace deepens your actual understanding of how the pieces fit together.

### Using Your IDE to Explore

Lean on your IDE, not just as a code editor, but as an exploration tool:

- **Jump to related methods** across files, instead of manually digging through the folder tree.
- **Collapse method bodies** to shrink the visible surface area and get a bird's-eye view of a file.
- **Use the class outline view** many IDEs provide, showing you a class's structure — and just reading the method names alone often tells you a lot about a module's purpose.

**One important caution: don't assume the code does what its name implies.** As code evolves, variable and method names often drift away from what they actually do — nobody goes back and renames things every time behavior shifts slightly. Don't rush past this; confirm your assumptions rather than just trusting a name. Cutting corners here tends to create *more* work later, not less — take the time it takes, so it takes less time overall.

**Watch out for misleading exception handling, too.** More than once, developers have run into exception-handling code built on incorrect assumptions about what could actually go wrong. Be especially cautious around code that catches very broad, high-level exceptions — convenient for whoever wrote it, but often a sign that real error conditions are being obscured rather than properly handled.

**Use IDE analysis tools where available.** Tools like IntelliJ IDEA can visualize a project's internal dependencies, giving you a clearer sense of how different pieces connect to each other. Modern tooling is genuinely powerful here — let it do some of the heavy lifting for you.

### Using Your Source Control History

Your version control tool is also a research tool, not just a backup system. Look at a file's change history — what gets modified frequently? What do the commit messages reveal about *why* things changed?

**A practical Git command to find your codebase's "hotspots"** — the files changed most often over the last year:

```bash
git log --pretty=format: --since="1 year ago" --name-only -- "*.java" \
  | sort | uniq -c | sort -rg | head -10
```

**What this command actually does, piece by piece:**

- `git log --pretty=format: --since="1 year ago" --name-only -- "*.java"` — lists just the filenames (no commit messages or metadata) of every `.java` file touched in a commit over the past year. Each commit's changed files get printed, one filename per line.
- `| sort` — sorts those filenames alphabetically, which groups identical filenames together — a required step before counting duplicates.
- `| uniq -c` — collapses consecutive duplicate lines into one, prefixed with a count of how many times each filename appeared (i.e., how many times that file was touched).
- `| sort -rg` — sorts those counts numerically, in *reverse* (highest first).
- `| head -10` — trims the output down to just the top 10 most frequently modified files.

**Why this is useful:** files that change constantly are often either the most actively developed part of the system, or — just as informatively — a persistent source of bugs and rework. Either way, they're usually a great place to focus your early attention.

You can also use `git blame` to see, line by line, who last touched a given piece of code, and when — useful both for tracking down context and for figuring out who on your team might actually remember *why* something was written a certain way. Most IDEs expose this same history visually, if you'd rather avoid the command line. However you dig in, don't hesitate to just ask a teammate directly — sometimes that's simply the fastest path to an answer.

### Generating Diagrams from Code

Hand-maintained architecture diagrams tend to drift out of date as an application evolves — but you can often generate fresh diagrams directly *from* the current code, rather than relying on something someone drew (and then forgot to update) months ago. Many IDEs can generate a class diagram with a simple keyboard shortcut; standalone tools like **Umbrello**, **Doxygen**, or **Structurizr** can do the same thing more thoroughly. It's even worth considering adding an automated step to your build pipeline that regenerates these diagrams every time code is committed, so they never go stale in the first place.

## Effective Code-Reading Strategies

With a general lay of the land, it's time to dig into the actual code-reading process itself.

### Leveraging IDE Features

Your IDE is far more than a place to type code — it's a genuine exploration and comprehension tool, arguably even *more* valuable when you're navigating unfamiliar code than when you're writing something new.

**Adopt a continuous-learning mindset.** Modern IDEs have so many features that trying to learn all of them at once is a losing battle. Instead, aim to learn just one new trick a week. JetBrains' IntelliJ IDEA, for instance, has a built-in "Tip of the Day" feature that surfaces a new shortcut, tool, or technique each time you open the IDE — covering everything from keyboard shortcuts to refactoring tools, debugging features, and plugins. Learning your IDE's shortcuts specifically tends to be the highest-leverage place to start, since it compounds every single day you use the tool. (If you miss a day's tip in IntelliJ, you can always find it again under **Help → Tip of the Day**.)

> **A simple habit worth building:** keep a personal "IDE Tricks" document — jot down new shortcuts or features as you discover them, and revisit it periodically to actually build the muscle memory, rather than forgetting the trick a week after learning it.

**Use code navigation tools deliberately.** A few specific features are worth actively practicing:

- **Find usages/references** — instantly see every place a class, method, or variable is actually used in the codebase. Extremely useful when you're trying to understand how some abstraction gets used *in practice*, not just in theory. In IntelliJ, for instance, clicking a small gutter icon next to an interface shows you every class that implements it; right-clicking a method and selecting "Find Usages" shows every call site.

- **Jump to definition** — instantly move from where something is *used* to where it's actually *defined*, following the code's logical thread. For example, Command-clicking (or Ctrl-clicking on Windows/Linux) a type like `ArrayList` jumps straight to its source and documentation. This is especially valuable for understanding third-party libraries, framework internals, base classes and interfaces, and shared utility methods scattered throughout a codebase.

- **Call hierarchies and dependency views** — specialized views that visualize *relationships* between pieces of code:
  - **Call hierarchy** — shows both what calls a given method, and what that method itself calls.
  - **Type hierarchy** — shows inheritance relationships between classes and interfaces.
  - **Dependency diagram** — visualizes how entire modules or packages depend on one another.

  In IntelliJ, right-clicking a method and choosing "Show Call Hierarchy" reveals both its incoming calls (who calls it) and outgoing calls (what it calls) — often the fastest way to understand a method's real role within a larger system.

### Code Analysis Features

IDEs don't just help you *navigate* code — they can actively *analyze* it as you work, surfacing insights and teaching you patterns along the way.

**Automated inspections and suggestions.** As you write or edit code, your IDE may proactively suggest refactors. These are only suggestions — you're never obligated to accept them — but they're often worth reviewing anyway, since they frequently point toward idiomatic patterns you might not have known about.

**Example — iterating over a list the "traditional" way:**

```java
List<Book> books = library.getBooks();
for (Book book : books) {
    System.out.println(book.getTitle());
}
```

This code works perfectly well, and there's nothing technically wrong with it. But an IDE like IntelliJ might suggest a more idiomatic, streamlined alternative:

```java
library.getBooks().forEach(book -> System.out.println(book.getTitle()));
```

**What changed:** the explicit `for` loop is replaced with a call to `forEach()` — a method available on Java collections that applies a given action (here, a lambda expression printing the book's title) to every element, without you needing to manage the loop's mechanics yourself. Functionally identical — but more concise, and closer to how experienced Java developers tend to write this kind of code today. Suggestions like this don't just clean up your code; they teach you the idioms of the language and frameworks you're working in.

**Code structure visualization.** Understanding how code is organized — whether at the level of an entire project, or a single class — is another place your IDE can help enormously. A project structure view can immediately tell you, for instance, whether you're looking at a monolithic application organized "package by feature," rather than you having to infer that from scratch.

Similarly, if you want a quick sense of what methods exist on a given class — say, an `OwnerController` in a sample project called PetClinic — you don't need to manually scroll through the whole file or trust that some external API documentation is actually up to date. Most IDEs offer a **structure view** that lists a class's methods at a glance, giving you an immediate, accurate sense of that class's purpose and capabilities.

## Reading Tests for Insight

Tests are another powerful form of documentation — arguably a more *trustworthy* one, since a badly-written test will typically fail rather than silently rot. Well-written tests reveal not just *what* the code does, but *why* it behaves the way it does. Where the implementation tells you *how* something works, a good test tells you how it's *supposed* to work.

### Tests as Living Documentation

Unlike ordinary docs, which can silently drift out of date, tests generally have to stay accurate — otherwise they fail your CI pipeline. That makes them a genuinely reliable, current source of truth about how the system is meant to behave.

Tests document expected inputs and outputs, how components interact, and overall system behavior. Reading tests *before* diving into an implementation lets you understand what a component is *supposed* to do, before you get lost in *how* it actually does it — which makes the implementation itself far easier to follow once you get there.

**Example — a pricing test:**

```java
@Test
public void shouldApplyBulkDiscountForLargeOrders() {
    // Given
    Customer regularCustomer = new Customer(CustomerType.REGULAR);
    Order bulkOrder = new Order(regularCustomer, 500.00);

    // When
    double finalPrice = pricingService.calculateFinalPrice(bulkOrder);

    // Assert
    assertEquals(450.00, finalPrice, 0.01);
}
```

**What this test tells you, before you've looked at a single line of `pricingService`'s implementation:** a *regular* (non-premium) customer placing a $500 order ends up paying $450 — a 10% bulk discount. It confirms the discount is tied to *order size*, not customer type, since the customer here is explicitly `REGULAR`. With that expectation firmly in mind, diving into `calculateFinalPrice()`'s actual implementation becomes much easier — you already know exactly what result you're looking for it to produce.

### Understanding Workflows and Processes

Tests — especially integration tests — often map directly onto real user stories, making them a great window into broader business workflows, not just isolated units of logic. Look specifically for tests that verify an entire process end-to-end; their setup code often mirrors real-world usage scenarios, giving you valuable context about how the system actually behaves once deployed.

**Example — a product review workflow:**

```java
@Test
public void submitProductReviewShouldUpdateRatingAndNotifyAdmin() {
    // Given
    Product product = productService.findProductById("XYZ789");
    User user = userService.findByUsername("janedoe");
    Review review = new Review(user, product, 4, "Great quality product!");

    // When
    ReviewResult result = reviewService.submitReview(review);

    // Then
    assertTrue(result.isSuccessful());
    assertEquals(4.2, productService.findProductById("XYZ789").getAverageRating(), 0.01);
    verify(notificationService).sendReviewNotification(anyObject(), eq("XYZ789"));
}
```

**What this reveals:** submitting a review does more than just save the review itself — it also recalculates and updates the product's average rating, *and* triggers a notification to an administrator. Without reading a single line of `reviewService`'s actual implementation, this one test already sketches out the entire feature's shape: three distinct outcomes (success confirmation, rating recalculation, and an admin notification), all expected from a single action.

### Discovering Edge Cases and Boundaries

An **edge case** is a situation sitting right at the extreme boundary of what a program expects — valid, but unusual, input or conditions that tend to be exactly where hidden assumptions break down. In an existing, mature codebase, you'll often find a disproportionate number of tests covering these — usually because a real bug was discovered in production at some point, and a test was written afterward specifically to make sure it never quietly resurfaces.

Pay close attention to test names like `shouldHandleEmptyList`, `shouldRejectInvalidInput`, or `shouldTimeoutAfterTenSeconds` — these directly mark the system's known limits and past failure points.

**Example — expired credit card handling:**

```java
@Test
public void shouldRejectPaymentWhenCreditCardIsExpired() {
    // Given
    CreditCard expiredCard = new CreditCard("4111111111111111", "05/20");
    Order order = new Order(new Customer(), 50.00);

    // When
    PaymentResult result = paymentService.processPayment(order, expiredCard);

    // Then
    assertFalse(result.isSuccessful());
    assertEquals(PaymentFailureReason.EXPIRED_CARD, result.getFailureReason());
}
```

**What this reveals — something you might easily miss just reading the implementation cold:** the system explicitly checks for and gracefully handles expired credit cards, returning a specific, well-defined failure reason (`EXPIRED_CARD`) rather than, say, crashing or returning a generic error. With that expectation in hand, you can go straight into `processPayment()` looking specifically for the logic responsible for catching this exact case.

With a solid grasp of how to explore a project and read its code effectively, the next step is learning how to safely *change* it — the focus of the next chapter.

## Practice Makes Perfect

At the end of the day, extend yourself some grace here. Modern codebases are frequently enormous and sprawling — no single person understands the entire thing, and that was never a realistic goal in the first place. Your understanding will keep growing over time; just repeat this process every time you run into a new corner of the project. It can feel intimidating at first, but every developer has been exactly where you are. You'll be fine.

**How do you actually get better at reading code?** As unglamorous as it sounds: practice reading it. There's an enormous supply of well-written, publicly available open source code, across virtually every language, freely available to explore. There's no real shortcut here — improvement comes from repetition, and it genuinely does get easier, and faster, the more you do it.

## What About AI?

Software tooling keeps evolving — from bare-bones, shell-based text editors, to modern IDEs with built-in refactoring tools and intelligent code completion. Writing code has gotten easier over time in some real sense — though the size and complexity of the applications being built has grown right alongside it, so the net difficulty may be roughly a wash.

With tools like GitHub Copilot and general-purpose generative AI, some have speculated that developers might soon become unnecessary altogether. Predictions of the developer's demise have a long, consistently wrong track record — from COBOL, to fourth-generation programming languages, to countless drag-and-drop "no-code" tools — none of them have meaningfully shrunk demand for software engineers at scale.

These tools absolutely have a place in your toolkit — asking an AI assistant what a chunk of unfamiliar code does can be a genuinely useful starting point. But you still need the surrounding context. AI doesn't inherently know the *purpose* behind a given class; you still need to actually read the code, dig through documentation, and talk to your teammates. AI tools don't (yet) have a full, reliable understanding of an entire enterprise codebase, and the risk of confidently wrong answers — "hallucinations" — is real. AI can help you move faster, but you're still ultimately responsible for understanding the nuance and relationships in code spread across multiple repositories.

As one of this book's authors likes to put it: when working with AI, **you are the pilot, not the passenger** — trust, but verify. AI can reason reasonably well about code in isolation, but it may not reliably understand an enterprise-scale codebase without producing confidently incorrect answers along the way. Modern applications are frequently spread across many repositories and dependencies — it's still on you, the developer, to understand how all of that actually fits together, so that AI tooling can genuinely accelerate your work rather than quietly leading you astray.

## Wrapping Up

Arguably, coding gets taught backward: you learn to *write* before you ever learn to *read* — and yet reading someone else's code will end up occupying a huge share of your actual career. You may never enjoy working in existing code quite as much as building something fresh, but it comes with the job. Rather than avoiding it, learn to embrace it — there's real professional growth waiting there. Stay aware of your own cognitive biases. Don't be afraid to dig into an unfamiliar codebase and get your hands dirty; you'll learn something every time. And as your own understanding deepens, leave the code a little better than you found it — easing the path for whoever comes next. That next person might just be you.

## Putting It into Practice

**There's no substitute for reading more code.** Fortunately, there's an enormous supply of open source projects available to practice on. Set aside a couple of hours to read through a framework you already use at work (or one you've been curious about), or dig into a project that genuinely interests you. If you're not sure where to start, browse GitHub's trending repositories for inspiration. Apply the techniques from this chapter as you go. Revisit the same project, or try a completely new one, a couple of months later — was it noticeably easier the second time around? Keep at it consistently, and this skill will keep improving.

**Working with existing code is a skill you build deliberately**, and open source software again gives you a huge playground for practicing it. Contributing to an open source project is an excellent way to build this muscle — and it's genuinely far less intimidating to get started than most people expect. Pick a project, and spend a few real hours working inside it.
