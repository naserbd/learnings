# Chapter 3: Writing Code

> "Any fool can write code that a computer can understand. Good programmers write code that humans can understand."
> — Martin Fowler, British software developer, author, and international public speaker on software development

Writing code is, without question, a core part of software engineering. And while the *mechanics* of coding are widely taught, the nuances of writing genuinely *good* code are not. Just because you *can* write code to solve a problem doesn't mean you *should* — sometimes the right move is not writing new code at all.

As AI and agentic coding tools become more common, a software engineer's day-to-day work is shifting — less raw, hands-on typing, and more reviewing, correcting, and refining code an AI tool generated. But here's the catch: to be genuinely good at *reviewing* code, you first have to be genuinely good at *writing* it yourself. Counterintuitive as it sounds, writing clean, thoughtful code is more valuable than ever in an AI-assisted world, not less.

Developers, for better or worse, tend to have strong personal opinions about what makes code "good" or "bad" — but metrics and tooling can offer real, objective insight beyond gut feeling. Tests are some of the best documentation you can produce. Good code reviews keep a team from relying on error-prone patterns or unnecessarily clever tricks. At the end of the day, code should be written for **humans** to read.

However it gets taught, programming is fundamentally a *communication* activity — and not communication with the compiler. The computer will happily execute any syntactically valid code you throw at it; that says nothing about whether another human being can actually follow what you were trying to accomplish. The best engineers write code aimed squarely at the *human reader* — concise, well-organized, and clear about its intent. Code that's easy to read is code that's easy to maintain, and that's ultimately what determines a software project's long-term success.

The very first thing most people learn in software development is *how to code* — probably starting with "Hello, World!" What often gets skipped, whether in a university course, a boot camp, or self-taught material, is *what actually makes code good*, *how to write it that way*, and — just as important — *when writing new code isn't even the right move*. This chapter aims to fill that gap.

## Don't Reinvent the Wheel

Before you start typing out line after line of (undoubtedly excellent) code, take a moment to check whether this problem has already been solved. Developers routinely write far more code than they need to, re-solving problems someone else already figured out. Before writing something fresh, look around:

- **Is there a library that already does this?** (Just remember — adopting a library isn't free; check its own dependencies, and follow your organization's policies around third-party code.)
- **Does your primary language already have a built-in feature for this?** Languages evolve constantly (recall "The Capitalization Assignment" story from Chapter 1) — a five-minute search can save you hours of unnecessary work.
- **Does your application framework already solve this?** If it feels like there *should* be an easier way, there very often is.

Ask yourself honestly: are you really the first person in the world to ever need this exact piece of code? Almost certainly not.

### A Practical Decision Tree

Christopher M. Judd teaches the following checklist in his boot camps (adjust it to fit your own language and frameworks):

1. Is this already done somewhere else in the current codebase?
2. Does the standard library (e.g. the JDK, for Java) already handle it?
3. Does an existing framework (e.g. Spring Framework or Spring Boot) already solve this?
4. Does a solution exist in a widely-used utility library (e.g. Google Guava)?
5. Does a solution exist in another common utility library (e.g. Apache Commons)?
6. Do any other libraries *already present in the project* solve this?
7. Does an open source library exist that solves this?

**If every answer is "no,"** you're clear to write the code yourself — and be sure to write tests alongside it as you go. **If you answered "yes" to any of these,** congratulations — you just saved yourself real time. Use what's already there.

**One important caveat:** yesterday's best practice can quietly become tomorrow's anti-pattern. Just because an approach was correct five or ten years ago doesn't mean it's still correct today. Don't be afraid to ask *why* something is done a certain way, and to challenge the status quo when it seems warranted. It's easy to mindlessly copy a pattern you found online, or to trust AI-generated code without really understanding *why* it works. Software moves fast — if you never stop to look around with fresh eyes (recall "The Value of a Fresh Set of Eyes" from Chapter 1), you risk missing something genuinely important. Languages, frameworks, and best practices are always evolving; staying current is simply part of the job. Ask good questions, and keep an eye on the horizon.

## What Is Good Code?

Sometimes there genuinely isn't a library or built-in feature to lean on — you'll have to write code yourself. So: what actually makes code *good*?

To paraphrase a famous (if unrelated) legal quote: good code can be hard to define precisely, but you know it when you see it. "Good" versus "bad" code is genuinely subjective territory — but with enough experience, you develop a real sense for it. Developers sometimes describe this instinct as code having a "smell." An overly long method, for instance, is usually a red flag — though not automatically; sometimes there really isn't a simpler way to express the logic, and you have to actually read it to know. Few developers will openly admit to writing bad code — but anyone who's ever opened an editor has written *some*, at some point.

**Metrics can give you real, objective insight here.** For example, **cyclomatic complexity** measures the number of distinct paths through a piece of code — more paths generally means more complexity, and more complexity generally means the code would benefit from refactoring. Static analysis tools like PMD, SonarQube, and JSHint can catch certain classes of bugs and bad practices automatically, before they take root. Tools like SonarQube and CodeScene can offer genuinely valuable insight into a codebase's overall health. Your organization likely already has something like this in place — if you're not sure, ask around. If it truly doesn't, consider being the one who pushes to introduce it.

### Adding Metrics to Existing Projects

Ideally, a project adopts tools like these from day one — "start clean, stay clean" (assuming the team actually fixes what gets flagged). But often you don't get that luxury, and you're introducing a linter or analyzer into a codebase that's been growing, unchecked, for months or years.

**It's tempting to install a tool like PMD and flip on every rule at once. Don't.**

- Some rules directly contradict each other — satisfying one can automatically violate another.
- Turning on too many rules at once typically produces an unmanageable flood of warnings, which is actually counterproductive. It's demoralizing for a team to face thousands of open warnings, where a new commit's few extra warnings vanish into the noise, and fixing a couple dozen of them barely moves the needle.

**A better approach:** discuss with your team and agree on a small, manageable starting set of rules (finding that consensus alone may take real discussion!). Turn those on. Fix what surfaces as a team. Once that's under control, turn on a few more rules, and repeat. After a few months of this steady, incremental process, you'll have a much richer, more comprehensive rule set running — and genuinely cleaner code to show for it.

**Metrics should never be followed blindly** — some complexity genuinely can't be avoided, and applying a rule of thumb without judgment can sometimes make things worse, not better. Apply common sense alongside any metric.

That said, you can put a well-known psychological effect to good use here: the **Hawthorne effect** — the tendency for people to change their behavior simply because they know they're being observed. If you want more of your codebase covered by tests, for instance, prominently displaying your current code coverage number can genuinely move that number, just through visibility alone.

> ### Story: Increasing Code Coverage Through Metrics
>
> *Told by Nate.*
>
> Years ago, I joined a project that had been actively developed for a few months. I was pleasantly surprised to find actual tests already existed — less pleasantly surprised to discover nearly all of them were being skipped in the build, because they "kept failing."
>
> Undeterred, I ran a code coverage report — a measurement of how many lines of code actually get executed when the tests run. The number, unsurprisingly, was in the single digits. But instead of quietly fixing it myself, I posted the number publicly, and talked about it — a lot. In nearly every standup and retrospective. Whenever the number ticked upward, I made a point of publicly praising whoever was responsible. When someone tackled tests for a particularly gnarly part of the codebase, I had them walk the team through how they did it.
>
> Slowly, the coverage number climbed, and the codebase itself became noticeably less fragile. It took several months, but by the time I moved on to a different project, coverage was closing in on 70%. Not perfect — nowhere near "done" — but a dramatically better place than where we started. And it all began with something as simple as running a coverage report, and then actually talking about it, consistently, out loud, in front of the team.

**Metrics can also be misused** (recall the earlier "Metrics Can Mislead" discussion). Some organizations, for instance, have mistakenly tried to evaluate individual engineers by raw lines of code written, deleted, or modified — a genuinely bad idea. Consider another metric: **external coupling**, or how many other classes depend on a given class. High coupling generally signals a refactoring opportunity — but not always; that class might be a deliberate **facade**, intentionally hiding a more complex set of classes behind a simpler interface. Every metric needs real context to be meaningful.

Just because something *can* be measured doesn't mean the number alone tells you much. Try to tie your metrics back to actual project goals, and pay closer attention to the *trend* than to any single snapshot — are things getting better or worse over time? Favor short observation windows, and stay genuinely willing to adjust course.

Adding a whole new tool to your pipeline isn't your only option, either — many IDEs surface code-quality metrics directly, and plenty of editor plugins exist to help you write better code as you type. Wiring analysis tools directly into your editor (rather than only running them at build time) shortens the feedback loop considerably — you find out about a violation the moment you type it, not hours later when a build breaks. Your team may already have a preferred setup; if you're unsure, just ask a colleague.

## Less Is More

> "The goal of software design is to create chunks or slices that fit into a human mind. The software keeps growing, but the human mind maxes out, so we have to keep chunking and slicing differently if we want to keep making changes."
> — Kent Beck

Given the sheer size of some codebases, you might assume developers get paid by the character. While some problems genuinely require millions of lines of code, in most cases, smaller is better. Less code means less that has to be loaded into someone's brain at once. Many projects eventually grow past the point where any single developer can hold the whole thing in their head — one of the real forces behind the rise of microservices and function-as-a-service architectures.

Sprawling, poorly bounded codebases ("big balls of mud") tend to come with dictionary-length setup guides and build processes measured in weeks, not minutes. New developers can spend weeks or months just becoming *productive* in code like this. The smaller the codebase, the faster a new developer can genuinely understand it and start contributing (Chapter 6 covers this in more depth). Here are some concrete things to watch for when trying to keep code size under control.

### The Zeroth Law of Computer Science

A huge portion of the advice software engineers give each other about taming complex code boils down to one underlying principle: **high cohesion, low coupling.**

- **Cohesion** measures how closely related the things inside a single unit of code actually are. High cohesion means related things stay together. A "notification" function that also happens to contain print-formatting logic is a clear example of *low* cohesion — two unrelated jobs crammed into one place.
- **Coupling** measures how interdependent different modules or routines are on each other. Tightly coupled code is fragile — a change in one part can unexpectedly break something seemingly unrelated elsewhere. Changing how notifications work shouldn't have any effect on the printing module.

High cohesion and low coupling generally produce code that's more readable, and easier to maintain and evolve over time. Many well-known design patterns are really just different strategies for achieving exactly this, at different levels of abstraction — arguably, microservices at their best are simply "high cohesion, low coupling," applied at the level of entire services.

### Beware Boilerplate Code

**Boilerplate code** — repetitive, mostly mechanical code — should be avoided, even when your editor or framework auto-generates it for you. You may not have to *type* it, but you'll still be carrying it around for the entire lifespan of the project. Classes should generally be short — a few pages at most (though "short" varies by language, since some are naturally more verbose than others). As a rough gut-check: if you find yourself scrolling a lot to see a whole class, it's probably grown too large.

### Favor Composition over Inheritance

Most languages let classes inherit from other classes — a genuinely powerful feature, and there are absolutely legitimate *is-a* relationships in the real world (a cat *is-a* mammal). But inheritance is very commonly overused. Some developers reach for inheritance purely as a shortcut for code reuse — but **reuse should be a side effect of good design, not the reason for choosing it.**

**A concrete example — cars and trucks.** Suppose you model a domain involving vehicles. You might define a `Vehicle` superclass that both `Car` and `Truck` inherit from, including a combustion engine — reasonable enough, since (at the time) every car and truck genuinely has one:

```
class Vehicle {
    Engine engine
    Integer num_wheels
    Integer num_doors
    Function brake {}
    Function accelerate {}
}

class Truck < Vehicle {
    Number tow_capacity
}

class ElectricVehicle < Vehicle {
    Number range
    # wait… EVs don't have engines…
}
```

**What went wrong here:** electric vehicles come along, and suddenly the assumption baked into `Vehicle` — "every vehicle has a combustion engine" — breaks down. An `ElectricVehicle` genuinely *is-a* vehicle, conceptually — but it doesn't fit the mold the superclass assumed everyone would share. Because `engine` was hardcoded onto the shared parent class, every subclass is now stuck inheriting an assumption that no longer universally holds.

**A more flexible alternative: composition.** Instead of baking an engine directly into the vehicle hierarchy, give each vehicle a *propulsion system* as a separate, swappable component it "has," rather than something it inherits:

```
class Vehicle {
    PropulsionSystem propulsion
    Integer num_wheels
    Integer num_doors
    Function brake {}
    Function accelerate {}
}

class CombustionEngine implements PropulsionSystem { ... }
class ElectricMotor implements PropulsionSystem { ... }
```

Now `Vehicle` doesn't assume *anything* about what powers it — a gas-powered car and an electric vehicle can both be perfectly valid `Vehicle` instances, simply composed with a different `PropulsionSystem`. This is more flexible, and doesn't require rewriting the class hierarchy the moment a new kind of vehicle shows up. This isn't a blanket ban on inheritance — just a strong general preference for composition when reuse, not a genuine *is-a* relationship, is the actual motivation.

### Favor Short Methods

Aim for genuinely short methods — single-digit line counts, as a rough target. Like a well-designed Unix command-line tool, a method should do exactly one thing, do it well, and combine cleanly with other small methods to accomplish bigger goals. **A method name containing a conjunction** ("and," "or," "but") is a strong signal it's doing too much — split it apart. Method names should be clear, concise, and unclever. If you're genuinely struggling to name a method, that difficulty is often itself a signal the method is trying to do too much — try breaking it into smaller pieces and see what happens. Be descriptive. Eliminate duplicated logic, even in small doses. Simplify — and then simplify again.

**Overall goal:** smaller, more manageable code, because that's what actually improves both developer understanding and day-to-day productivity.

## Write Code to Be Read

Keep the next developer in mind — the one who'll eventually need to read, understand, and maintain what you're writing today. This is the Golden Rule from Chapter 1, applied directly to code: how can you write this in a way that genuinely makes life easier for whoever comes next? What do you actually need to communicate to them?

There's no shortage of rules of thumb about the "correct" length for a function — arbitrary line counts, reuse heuristics, and so on. Martin Fowler offers a more useful lens: pay attention to the gap between **intent** and **implementation**. In other words — how long does it take you to understand *what* a function is doing, just from looking at it? Ideally, you should be able to read a function's *name* and immediately understand what it does, without ever needing to read its body. Following this principle honestly tends to produce very short functions — sometimes just a single line — because the goal is genuinely making the intent unmistakable. The class name gives you overall context; the method names inside it tell you what the class actually does.

**A useful metaphor: the newspaper "inverted pyramid."** A well-written news article leads with the most important facts, then moves progressively into supporting detail, and finally deeper background — letting the reader stop wherever their interest runs out, whether that's after the first paragraph or all the way to the end. Good code can follow the same shape: a class name works like a headline; skimming its method signatures gives you the gist; and diving into any individual method's implementation, or a call it makes to another class, is entirely optional, available only if you actually want that level of detail.

### Naming Things Is Hard

Every developer has stared blankly at their editor, stuck on what to call a variable, method, or class. That struggle sometimes signals a genuine gap in understanding the underlying problem — but it's also just... genuinely hard, which is exactly why `foo`, `bar`, and `foobar` show up so often in example code everywhere. Coming up with a good name takes real time — don't rush it. (Once again: take the time it takes, so it takes less time overall.) It's worth the investment. Don't hesitate to ask a teammate for input — sometimes just explaining what you're building out loud is enough to surface the perfect name on your own.

Your IDE can help here too. Many developers work backward from a declaration, letting the editor suggest a name based on the value itself — for instance, starting with something like `new ArrayList({"Red", "Green", "Blue"})` and letting postfix completion suggest a sensible variable name from there. AI coding tools can offer inspiration in a similar way.

Don't be afraid to rename poorly-named code once you understand it better — modern editors make safe, project-wide renaming genuinely painless. Just be cautious about constantly renaming *core domain concepts* — if a central term in your business domain keeps changing, that's usually a sign of a deeper misunderstanding, not just a naming problem.

**A useful trick worth trying: the "gibberish game."** Often, the very first word you reach for to describe a new concept isn't actually the best fit — but it does shape how you think about the problem while you're first working through it. So, while you're still exploring a new domain, try deliberately making up a nonsense word for a concept instead of forcing a "real" name too early. Once you understand the concept better, go back and replace the placeholder with something meaningful — you'll often land somewhere noticeably clearer than whatever your very first instinct would have produced. Next time you're genuinely stuck naming something, try dropping in a nonsense placeholder and circling back to it later, once the fog has cleared a bit.

## The Problem with Code Comments

Every programming language gives developers a way to talk directly to other developers, rather than to the compiler — the humble **code comment**. It might feel like liberal commenting is simply good practice — but arguably, heavy commenting is itself a code smell.

**Comments violate the DRY principle** (Don't Repeat Yourself). DRY violations usually bring to mind copy-pasted code or duplicated logic — but comments can just as easily be a form of the same problem. Very often, what actually happened is: you wrote the code, and then turned right around and wrote a *second*, parallel description of the same thing in English.

**Comments are also a real maintenance burden.** When you modify the code, will you reliably remember to update the comment sitting next to it too? In practice, comments frequently drift out of sync with the code they describe — leaving yet another layer of confusing, contradictory sediment for whoever reads this months or years later. Your time is generally better spent making the *code itself* easier to read, rather than writing a parallel narration of what you did. Expressive, readable languages genuinely help here — and it's often worth avoiding a language's more obscure "magic" features, since cleverness tends to demand more explanation, not less. Good method and variable names beat comments almost every time — if you feel the urge to write a comment, first try renaming the thing you were about to explain. Very often, that alone removes the need for the comment entirely.

**Arguably the single least useful kind of comment: the giant changelog block** sitting at the top of a file — sometimes hundreds or even thousands of lines of tracking numbers, dates, author names, and change descriptions. It might be mildly interesting as an archaeological curiosity once in a great while, but the sheer repetition adds far more noise than value day-to-day. Let your version control system handle that job — it already does it far better.

**None of this means comments should never be used.** When you're doing something genuinely non-obvious, and you've already tried and failed to simplify the code itself, a comment explaining the situation is legitimately helpful. Good comments tend to focus on **why**, not **what** — explaining a choice between two possible algorithms, or clarifying a piece of logic that's inherently complex, regardless of how it's written. That said: if code genuinely needs an explanation to be understood, that's often itself a sign the code should be rewritten, not just annotated.

Sometimes a comment is really a note to your future self — a flag that a hack works, even though you don't fully understand *why* yet. Some developers even leave comments as a kind of tongue-in-cheek warning to future maintainers:

```
// Dear maintainer:
//
// Once you are done trying to
// "optimize" this routine,
// and have realized what a terrible
// mistake that was,
// please increment the following
// counter as a warning
// to the next person:
//
// total_hours_wasted_here = 42
```

Whether comments like this are genuinely a good idea is debatable — plenty of the famous examples you'll see floating around are likely more legend than fact. Still, there's nothing wrong with occasionally leaving honest, useful advice behind for your future self.

## Tests as Documentation

If comments aren't the right tool for documenting your code, what is? **Write tests.** Tests — especially ones written in a clear, fluent style — function as executable specifications that evolve *alongside* the actual production code, rather than drifting away from it the way plain-text docs tend to. Documentation of any kind — comments, READMEs, formal specs — tends to start diverging from reality almost immediately after it's written. Tests written *as* you write the code let you refactor with real confidence later, and act as reliable signposts for whoever reads the code after you. (Testing is covered in much more depth in Chapter 5.)

Adding tests to *existing* code is also a great way to capture and share what you're personally learning about how that code actually behaves — turning tacit, in-your-head knowledge into something the rest of the team can benefit from. As you rename a poorly-named method, trim some dead code, and add a test that pins down the real behavior, you're actively leaving that code better than you found it.

Some developers insist they still need heavy documentation comments for anyone consuming their service or API. Fair enough — those comments are less problematic than the earlier kind — but they're still not your only, or even best, option. Tests remain a more durable form of documentation. **Consumer-driven contracts** — an approach where the consumers of your service define tests describing exactly what behavior they depend on — let you clearly communicate what your service does, while giving you real confidence to keep iterating on it. As long as those contract tests still pass, you can freely evolve your implementation without fear of silently breaking something downstream. Consumers, in turn, gain genuine confidence in your service, since they have concrete, runnable tests simulating its expected behavior — letting *them* change their own code without fear of quietly introducing a new bug on their end either.

Consumer-driven contracts are a genuinely valuable part of building reliable, resilient systems. Most major languages and frameworks have solid tooling for this already — Spring Cloud Contract, and various platform-specific implementations of Pact, among others — worth adopting rather than reinventing.

## Avoid Clever Code

Software is hard, and the domains it lives in are genuinely complex — but not all complexity is created equal. In his widely cited essay "No Silver Bullet," Fred Brooks draws a useful distinction between two kinds:

- **Essential complexity** — complexity that's simply inherent to the problem itself: the real nuance of business rules, the inherent challenge of coordinating with a team, the constantly shifting nature of any living codebase. There's genuinely nothing you can do to eliminate this — it's just part of the job.
- **Accidental complexity** — complexity developers *add* unnecessarily, through noisy technology choices, overly heavy tooling, or needlessly clever code. Unlike essential complexity, this kind is genuinely avoidable, and worth actively hunting down and removing.

Software isn't immune to snake-oil sales pitches, either — plenty of companies promise products or processes that will supposedly revolutionize how you build software. Healthy skepticism is warranted. Real improvements absolutely exist, but as the scientific method reminds us, extraordinary claims deserve extraordinary evidence. Stay vigilant about hunting down and removing *accidental* complexity wherever you find it (Chapter 6 goes into this in more depth).

**Languages and frameworks often contain "error-prone forms"** — syntax that's technically valid, but structured in a way that quietly invites future mistakes. Consider this Java snippet — can you spot the trap?

```java
if (condition)
    doFoo();
    doBar();
```

In Java, curly braces around a single-statement `if` block are technically optional. Some developers genuinely prefer omitting them — it's less visually noisy, and saves a bit of vertical space. **But look closely: despite the indentation, `doBar()` is not actually inside the `if` block at all.** It runs *unconditionally*, every single time, regardless of `condition` — the indentation is visually misleading, since Java doesn't care about whitespace the way Python does. If a future developer adds a second statement expecting it to be covered by the same `if`, and simply matches the existing indentation without adding braces, they've just introduced a real bug — one that's easy to miss in review, precisely because it *looks* correct at a glance.

**The fix — always use braces, even for single-statement blocks:**

```java
if (condition) {
    doFoo();
}
doBar();
```

This makes the actual scope of the `if` statement unambiguous, regardless of how it's indented or how the code evolves later.

**The broader lesson:** actively avoid these kinds of error-prone patterns in whatever language you use. Just because *you* clearly understand a subtlety doesn't mean it's equally well understood across your whole team. Don't hesitate to update (or create) coding standards that explicitly call these patterns out — and lean on static analysis tools in your deployment pipeline to catch them automatically before they ever reach production.

## Code Reviews

Code reviews can range from a quick, informal "can you glance at this method?" to a multi-hour walkthrough involving several developers. Whatever form it takes, code review is a genuinely excellent way to learn, share experience, and spread knowledge across a team. More eyes on code is a good thing in general — part of why some organizations lean on pair programming too.

**A few practices that consistently improve code reviews, formal or not:**

**Don't be snarky. Avoid sarcasm.** Asking for feedback can already be genuinely stressful, and criticism — however well-intentioned — is easy to take personally. *How* you phrase feedback matters as much as the substance of it. Stay empathetic toward your teammate. It can be tempting to use a review as a chance to show off some obscure bit of technical trivia — resist that urge. The goal is improving the code, not demonstrating your own expertise.

> "The only way to make something great is to recognize that it might not be great yet. Your goal is to find the best solution, not to measure your personal self-worth by it."
> — Jonas Downey, software designer, developer, and writer

**Focus your attention where it actually matters.** Style and formatting issues matter, but your review time is far better spent elsewhere — automate style enforcement with tooling, and let the computer handle it. Spend your actual attention on the things a computer genuinely can't catch: Are the method and variable names clear? Is the code readable at a glance? Is there duplication? Is logging, tracing, and metrics coverage adequate? Are interfaces consistent with the rest of the codebase? Did the developer fall into an error-prone pattern? Is the underlying model actually correct? Did they reach for the wrong abstraction?

### Avoid the Checkbox Code Review

In some organizations, "code review" has degenerated into little more than a required checkbox in the source control system before a merge is allowed. The underlying intention is reasonable — but in practice, this often results in one developer nudging another to "just review it real quick," which mostly means checking a box on their teammate's behalf. Even with good intentions, this defeats the entire purpose of review. **Working in smaller batches** — smaller, more frequent changes rather than huge, sprawling ones — makes genuine, careful review much more realistic to actually do well.

Pull requests (PRs) are a common way teams try to enforce code quality. They're certainly more comfortable than an hours-long video call arguing over code — but they don't automatically build team trust or cohesion either. Some developers treat PRs as an opportunity to nitpick, reopen old arguments, or stake out turf. PRs are also vulnerable to the dreaded **LFTM response** ("looks fine to me") — sometimes genuinely appropriate, but often just another flavor of the checkbox review in disguise.

Text-based comments also strip away tone and body language that a face-to-face conversation naturally carries. A comment you didn't intend to sound harsh can easily land that way for the person reading it. It's not unusual for less experienced developers, in particular, to genuinely dread getting a PR reviewed. Practice empathy here — ask yourself how you'd want to be treated in the same position, and act accordingly. Senior team members especially should model this behavior for everyone else.

> ### It Is Hard to Be Criticized
>
> Developers put real personal investment into their work, which makes feedback naturally hard not to take personally sometimes. Code review is never an opportunity to embarrass someone for not knowing about a new language feature, or for not immediately spotting a simpler solution. No one is perfect; everyone makes mistakes. Code review is about building better software — it's about the *code*, not the *coder*. Keep it professional, stay humble, and ask genuinely helpful questions. Feedback tends to land better when it's sandwiched between honest positives, so make a point of calling out what's genuinely good too.
>
> Share your own past experiences generously — personal stories carry real weight, and tend to dissolve people's natural resistance to change far more effectively than abstract rules do. Offer help based on things you've personally run into before. Be careful with sweeping, absolute statements — make sure you actually have the full picture before declaring that something "won't work." You might be missing context, constraints, or history you're simply not aware of yet. Stick to discussing the code itself.
>
> "Every single one of us is doing the absolute best we can given our state of consciousness."
> — Deepak Chopra, Indian-American author and alternative medicine advocate
>
> If something about someone's code genuinely concerns you, don't hesitate to talk to them directly, one-on-one. People tend to get defensive in group settings — a quick private conversation often resolves things far more smoothly. Don't ambush a teammate publicly; nobody wins in that scenario. Remember: reviews are an opportunity to *learn*, and an opportunity to *teach* — for everyone involved.

### Fostering Trust

If checkbox reviews and rubber-stamp "LFTM" comments aren't the answer, what is? Whatever your specific process looks like, don't lose sight of its actual purpose: sharing experience, learning together as a team, and steering clear of problematic patterns — confusing idioms, deprecated approaches, and the like. Encourage teammates to genuinely ask questions and offer real, constructive feedback. Good code review builds a real sense of **collective code ownership**, and real trust across the team. Something as simple as highlighting an interesting bug-of-the-week, or taking a few minutes to share a lesson learned, can have an outsized impact.

A regular, informal space — even just a few minutes during a Friday standup — where people share an interesting problem they solved that week is a genuinely powerful, low-effort learning tool. It's easy to assume everyone already knows how to handle a particular situation; actually talking it through with the team spreads that knowledge much further than silent assumption ever will.

It should go without saying, but: treat your teammates with genuine respect. Be kind, do what's right, do what actually works. Don't be afraid to pause and honestly reassess your own approach, and ask whether there's a better way. Whether or not your team formally follows Agile, regularly adapting and adjusting is always worthwhile — if something isn't working, change it.

## Learning New Languages

Getting better at writing code fundamentally requires... writing code. But you can meaningfully accelerate your own growth by deliberately learning *new* programming languages.

Think about how learning a foreign human language actually works: for months, even years, you translate everything back into your native tongue in your head. Eventually — but only eventually — you start thinking, even dreaming, directly in the new language. It takes real time. Programming languages are surprisingly similar — just with far stricter grammar and a much smaller vocabulary to learn.

**The key is immersion.** If you genuinely want to learn French, moving to Paris will get you there far faster than an app on your phone. Picking up a new programming language works the same way: build something real in it — ideally something that solves an actual problem bothering you at work or at home. Follow the language's community online. Listen to podcasts, or watch conference talks focused specifically on it. Better yet, attend a local meetup, or the next relevant conference.

Developers often get emotionally attached to whatever language they learned first — worth being cautious about. Programming languages are ultimately just tools. A hammer beats a screwdriver for driving in a nail, but the reverse is true for a screw — different languages genuinely fit different problems better. (Many embedded systems, for instance, are written in C specifically because it's portable, reliable, and can be tightly optimized for specific hardware.) Make it a habit to regularly explore languages outside your usual toolkit. Several well-respected voices in the industry recommend picking up a genuinely new language roughly every year or two.

## Programming Is Fundamentally About Communication

Long before university computer science departments existed as their own thing, programming often lived inside math departments — and many people still assume strong mathematical aptitude is a prerequisite for succeeding in software. (Some universities still use math exams as part of recruiting for CS programs!) But despite that lingering assumption, research consistently shows that **language aptitude** — not mathematical skill — is the far better predictor of how quickly someone picks up a new programming language. Certain specialized domains genuinely lean math-heavy, sure, but the *craft* of programming itself really doesn't.

Programming is, first and foremost, a communication activity — and not communication with the compiler. The computer will happily run any syntactically correct code you give it; that guarantees nothing about whether a human being can actually follow what you meant to accomplish. The best engineers keep the *human reader* front and center, the same way any good writer keeps their audience in mind.

**Why bother learning a language you might rarely, if ever, use at your day job?** Because learning a genuinely new language reshapes how you think about *all* code you write afterward — even the language you use every single day. When picking a new language to explore, try to choose something meaningfully different from what you already know. An experienced Java developer, for instance, gains relatively little by only exploring other C-like languages (C# is a perfectly fine language to learn, to be clear — just not a very *different* one). Consider something with a genuinely different paradigm instead: a dynamic language like Ruby, or a functional language like Haskell. Even a fairly shallow exploration of a language well outside your comfort zone will noticeably shift how you approach programming afterward — sometimes deepening your appreciation for your usual language, sometimes changing how you actually write code in it.

Learning new languages gets easier the more you do it. Your very first language starts you at zero — nothing to compare it to. By your third, fourth, fifth language, you start noticing recurring patterns: "oh, this idiom is basically the same as that thing in Java," or "this structure looks like it was borrowed straight from Ruby."

Early in your career, it makes sense to focus most of your energy going deep on whatever language and framework you use daily. But don't neglect exploring beyond that — doing so isn't just intellectually refreshing, it genuinely makes you a stronger developer overall.

## Wrapping Up

Writing code is inescapably part of the job. Avoiding error-prone patterns and unnecessary cleverness can be the entire difference between a codebase people genuinely enjoy working in, and one they actively dread touching. From code reviews to static analysis tooling, there are many concrete ways to help yourself (and your team) write consistently better code. Favor tests over heavy commenting. When reviewing others' work, lead with empathy. And never lose sight of the core principle running through this whole chapter: code should be written to be read by humans — holding onto that idea will take you a long way toward earning that elusive "good code" label.

Writing genuinely good code is an ongoing, never-quite-finished learning process. Stay open to picking up new skills, and to sharpening the ones you already have. Staying current with evolving best practices is a real, durable driver of long-term success in this field.

## Putting It into Practice

There's no shortcut here — improving as a developer requires actually writing code. As the old joke goes: a pedestrian on 57th Street sees a musician stepping out of a cab and asks, "How do you get to Carnegie Hall?" Without missing a beat, the musician replies, wearily: "Practice."

**Build deliberate practice into your routine.** Every couple of months, block out a couple of hours for a **code kata** — a term borrowed from martial arts, where a kata is a fixed sequence of movements practiced repeatedly until it becomes second nature. A code kata applies the same idea to software: a small, self-contained problem you solve deliberately, specifically to sharpen skills you might not exercise much in your regular day-to-day work.

You can use code katas in plenty of ways: solve the same one in two or three different languages; pair up with a friend or colleague to work through one together; deliberately solve the same kata two or three *different* ways. Revisit a kata you solved months (or years) ago, and give it a fresh code review — what would you do differently today, knowing what you know now?

**Lean on the open source world again.** Block out a couple of hours to contribute to an open source library you already use, or one you'd like to start using. Not sure where to begin? Browse GitHub's trending repositories for inspiration. Contributing is a genuinely excellent learning environment, and getting started is far less intimidating than it might seem.

You don't even need to contribute to learn something valuable — simply pick a project and spend a couple of hours reading through its code. What do you genuinely like about it? What would you change? Run a static analysis tool against it — what does it reveal? If you spot a real, clear issue, don't hesitate to raise it, or even submit a fix yourself.

**Stay current, deliberately.** Keeping up with change is a core, ongoing part of a healthy software career — make it a habit to periodically refresh your knowledge of your core languages and frameworks. You won't always organically stumble onto new features through daily work alone, so set aside a few hours once or twice a year specifically to see what's changed. Most major technologies have active advocates and communities worth following, so you can keep loosely current without a huge time investment. Not every shiny new feature will be relevant to what you're building today — but digesting a small, steady trickle of updates is far more manageable than trying to catch up on years of change all at once.

**Consider volunteering your skills.** Many nonprofits and local charities are genuinely in need of technical help. Volunteering gives you a chance to practice your craft — potentially even a language or framework you're actively trying to learn — while supporting a cause that matters to you.
