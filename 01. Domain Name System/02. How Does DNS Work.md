# How DNS Works

## 1. The Domain Namespace

DNS is, at its core, a **distributed database indexed by domain names**. Every domain name is a *path* through a giant inverted tree called the **domain namespace**. This tree has exactly one root, at the top — just like a Unix filesystem has exactly one root directory (`/`). DNS calls its root simply "the root."

Structurally, the tree can branch any number of ways at each node, and it can go up to **127 levels deep** (a limit essentially no real deployment ever approaches).

**Why this matters:** everything else in DNS — zones, delegation, resolution, caching — is a consequence of this one design choice: name the whole world as a single hierarchical tree, and let pieces of that tree be handed off to different administrators. Keep that image in your head; every later section is really just elaborating on it.

---

## 2. Domain Names

- Each **node** in the tree carries a **text label**, up to 63 characters, containing no dots.
- The **root's label is null** (zero length).
- A node's **full domain name** is the sequence of labels read from that node *up* to the root, separated by dots.

This produces a subtlety that trips people up constantly: because the root's label is empty, a name that "ends in the root" technically ends in a trailing dot — `www.oreilly.com.` — even though we usually just write `www.oreilly.com`. When the root is written by itself, it's shown as a single dot: `.`

This trailing dot has real operational meaning:

- A name **ending in a dot** is an **absolute** name — unambiguously anchored to the root. This is also called a **fully qualified domain name (FQDN)**.
- A name **without** a trailing dot may be interpreted as *relative* to some other domain — the same way a filename without a leading slash is relative to the current directory, not to `/`.

If you've ever been bitten by a resolver silently appending a search-domain suffix to an unqualified name, this is the mechanism behind it — the software treated your input as relative, not absolute.

**Uniqueness rule:** sibling nodes (children of the same parent) must have different labels. This is the *entire* mechanism that guarantees a domain name uniquely identifies one node in the tree — and note carefully what it does *not* require: labels only need to be unique among their siblings, not globally. `hobbes.pa.ca.us` and `hobbes.lg.ca.us` can coexist because `pa` and `lg` are different parents — exactly the way `/bin` and `/usr/bin` can both exist, but you can't have two `/usr/bin` directories.

---

## 3. Domains

A **domain** is simply a subtree of the namespace, and its name is the name of the node at the top of that subtree. The `purdue.edu` domain is the subtree rooted at the node `purdue.edu`.

Because subtrees nest inside each other, **a single domain name can belong to multiple domains simultaneously**. `pa.ca.us` is part of the `ca.us` domain *and* part of the `us` domain — it's just further down in both.

**Where are the actual hosts?** This is worth dwelling on because it's the point most people gloss over: domain names *are* the index into host information. A "host" in DNS isn't a separate kind of object — it's a domain name that happens to have host data (addresses, mail routing, etc.) attached to it. A domain "contains" a host simply because that host's domain name falls inside the domain's subtree. The hosts inside one domain are related **logically** — by organization or geography — not necessarily by network, physical location, or hardware. Ten machines in ten different countries can all sit in the same domain.

**Don't confuse DNS domains with:**
- **NIS domains** — NIS names look hierarchical but the hierarchy is cosmetic; hosts in an NIS domain can't navigate to data in other NIS domains the way DNS zones refer to each other.
- **NT domains** — account/security management, unrelated to DNS.
- **Active Directory domains** — these *are* closely tied to DNS domains (covered elsewhere, but flagged here as the one genuine relative in this list).

**A domain name can be both a domain and a host simultaneously.** `hp.com` is both the name of HP's whole corporate domain *and* the name that resolves to HP's main web server. Which meaning you get depends entirely on context: mailing someone `@hp.com` triggers mail-routing lookups (MX records); `ssh`-ing to `hp.com` triggers a host-address lookup (A/AAAA records). Same name, different record types, different meaning.

A domain can have its own subtrees, called **subdomains**. The test for "is X a subdomain of Y" is purely textual: X's name ends in Y's name. `la.tyrell.com` ends in `tyrell.com`, so it's a subdomain of `tyrell.com` — and by the same logic, also a subdomain of `com`.

**Level terminology**, used constantly in DNS conversation:
- **Top-level domain (TLD)** — a child of the root.
- **Second-level domain** — a child of a TLD.
- ...and so on, one level at a time.

---

## 4. Resource Records

The actual *data* attached to a domain name lives in **resource records (RRs)**.

Records are grouped into **classes**, where a class corresponds to a network/software family:
- **Internet class (`IN`)** — TCP/IP-based, and overwhelmingly the one that matters. This is what the rest of this document — and virtually every DNS deployment on Earth — uses.
- **Chaosnet class** — historic, essentially unused today.
- **Hesiod class** — mostly confined to MIT.

Within a class, records come in **types** — one type per kind of data (address, mail routing, name-server pointer, etc.). Different classes can define different types, though common ones (like "address") tend to recur across classes. Each type has its own fixed record syntax that every RR of that class-and-type must follow.

---

## 5. The Internet Domain Namespace

Theoretically, DNS imposes almost no rules on what labels mean — you could name your subdomains `A` through `Z` and DNS wouldn't stop you. But the *real* Internet namespace has accumulated strong **traditions** (not enforced rules — traditions that have been broken plenty of times) that make names decodable if you know the conventions.

### Top-Level Domains — the original seven

| TLD | Purpose | Examples |
|---|---|---|
| `com` | Commercial orgs | `hp.com`, `sun.com`, `ibm.com` |
| `edu` | Educational orgs | `berkeley.edu`, `purdue.edu` |
| `gov` | Government orgs | `nasa.gov`, `nsf.gov` |
| `mil` | Military orgs | `army.mil`, `navy.mil` |
| `net` | Originally network infrastructure orgs (`nsf.net`, `uu.net`); opened to any commercial org in 1996 |
| `org` | Originally non-commercial orgs (`eff.org`); restrictions also removed in 1996 |
| `int` | International orgs | `nato.int` |

A special case: **`arpa`** — originally where every ARPANET host lived during the transition off host tables, later emptied out into organizational TLDs, but still in active use today (this becomes important later — it's the domain that makes reverse lookups possible; see Section 10).

These seven are now called **generic top-level domains (gTLDs)**, to distinguish them from **country-code TLDs**.

### Country-code TLDs

To internationalize the namespace, geography was allowed as an alternative organizing principle. Country-code TLDs follow **ISO 3166** — the standard two-letter country abbreviations. One notable exception: Great Britain's ISO code is `gb`, but by long tradition the UK uses `uk` instead.

### The 2000/2005 gTLD expansion

In late 2000, **ICANN** (the organization managing DNS overall) added seven new gTLDs to relieve namespace pressure. ICANN splits these into two categories:
- **Unsponsored gTLDs** — open, generic: `biz`, `info`, `name` (individuals), `pro` (professionals).
- **Sponsored TLDs (sTLDs)** — chartered for a specific community, with a sponsoring organization overseeing policy: `aero` (aviation), `coop` (cooperatives), `museum` (museums).

In early 2005, ICANN added two more sponsored TLDs — `jobs` (HR industry) and `travel` (travel industry) — with others (`cat`, `mobi`, `post`) under evaluation at the time; `mobi` was the one that made it to actual root delegation.

### Further down the tree — organizational patterns vary by country

- **Australia (`au`)** mirrors the original US organizational scheme: `edu.au`, `com.au`.
- **UK-style countries** put the organizational marker *before* the country code: `co.uk` (corporations), `ac.uk` (academic).
- **The US (`us`) domain, historically** — this one's a great illustration of namespace design choices having real administrative consequences. It originally had 50 subdomains, one per US state, using postal abbreviations, subdivided by city, then by host. This entire structure was abandoned in 2002 when Neustar took over `us` management and opened it up like `com`/`net`.

### Reading real domain names — worked examples

This is the payoff of learning the traditions above — you can now decode names on sight:

- **`lithium.cchem.berkeley.edu`** → `edu` = US educational institution → `berkeley.edu` = UC Berkeley → `cchem` = College of Chemistry's subdomain → `lithium` = a specific host (likely one of many named after elements).
- **`winnie.corp.hp.com`** → `hp.com` = Hewlett-Packard → `corp` = corporate HQ subdomain → `winnie` = an arbitrarily-named host.
- **`fernwood.mpk.ca.us`** → `ca.us` = California (old-style US domain) → `mpk` = Menlo Park (only decodable if you know Bay Area geography — traditions only help if you know the local convention).
- **`daphne.ch.apollo.hp.com`** → included specifically to show names aren't limited to four labels: `apollo.hp.com` = the old Apollo Computer domain, absorbed when HP acquired Apollo → `ch.apollo.hp.com` = Apollo's Chelmsford, MA site → `daphne` = a host there.

## 6. Delegation

Delegation is **the** mechanism that achieves DNS's founding design goal: decentralized administration. The analogy is exactly workplace delegation — a manager breaks a big project into pieces and hands each piece to a different employee, who then owns that piece fully.

Concretely: an organization administering a domain can carve off subdomains and hand each one to a different organization. The delegated organization now owns *all* the data in that subdomain — free to modify it, and free to delegate further subdomains of its own. Critically, **the parent domain doesn't keep a copy of the delegated data** — it keeps only pointers (nameserver records) telling queriers where to go to get it.

Not every organization delegates its entire domain. A domain can have some subdomains delegated away while still directly hosting other names itself. Acme Corp might delegate `rockaway.acme.com` and `kalamazoo.acme.com` to those divisions, while keeping scattered sales-office hosts directly under `acme.com` itself, ungrouped.

## 7. Nameservers and Zones

Programs that store namespace information are **nameservers**. A nameserver typically holds *complete* information about some portion of the namespace called a **zone**, loaded from a local file or transferred from another nameserver. When it holds a zone's complete data, it's said to have **authority** for that zone. One nameserver can be authoritative for many zones.

### Zone vs. domain — the distinction that everything else depends on

This is subtle but load-bearing, so slow down here:

> **A zone contains everything a same-named domain contains, *except* whatever has been delegated away into child zones.**

Concretely: the `edu` *domain* includes `berkeley.edu`, `purdue.edu`, `nwu.edu`, and everything beneath them. But the `edu` *zone* — the file the `edu` nameservers actually load — contains almost none of that. It contains mostly delegation pointers to `berkeley.edu`'s nameservers, `purdue.edu`'s nameservers, and so on. The actual `berkeley.edu` data lives in the **separate** `berkeley.edu` zone, managed by Berkeley.

And this recurses: `berkeley.edu` itself is broken into further zones — `cc.berkeley.edu`, `cs.berkeley.edu`, `ce.berkeley.edu`, `me.berkeley.edu` — each potentially served by an entirely different set of authoritative nameservers than the parent `berkeley.edu` zone, even though some overlap is common.

**The Canada example makes the "ragged edge" clear.** The `ca` (Canada) domain has provincial subdomains `ab.ca`, `on.ca`, `qc.ca`. If those three are delegated to provincial nameservers:
- The `ca` **domain** contains `ca` + `ab.ca` + `on.ca` + `qc.ca` (everything).
- The `ca` **zone** contains only the `ca`-level data itself — mostly pointers to the three delegated zones.

Now suppose two more provinces, `bc.ca` and `sk.ca`, exist in the namespace but **haven't been delegated yet** (maybe those provincial authorities aren't ready to run their own nameservers, but the `ca` administrators still want the names to exist for consistency). In that case the `ca` zone has a **ragged bottom edge**: it directly contains `bc.ca` and `sk.ca` data (since those aren't delegated), but not `ab.ca`, `on.ca`, or `qc.ca` (since those *are* delegated). A zone's boundary is defined entirely by where delegation happens to have occurred — not by any fixed depth in the tree.

**Why nameservers load zones instead of domains, stated plainly:** if a root nameserver loaded the root *domain* instead of the root *zone*, it would need to load the entire namespace of the Internet. Loading only the zone means loading only what hasn't been handed off to someone else.

**If your domain has no subdomains, this distinction is invisible** — your zone and your domain contain identical data. The zone/domain gap only opens up once delegation starts happening beneath you.

---

## 8. Delegating Subdomains

Mechanically, delegating a subdomain means: assign authority for it to a *set* of nameservers (plural — redundancy matters here, foreshadowing Section 9). Your own zone's data, instead of containing the actual records for that subdomain, contains **NS records** — pointers naming the nameservers that are authoritative for it. When your nameserver gets a query that falls inside the delegated subdomain, it answers with a **referral**: "ask these other servers instead."

---

## 9. Types of Nameservers

DNS defines two roles a nameserver can play for a given zone:

- **Primary master** — reads the zone's data straight from a local file on its own host.
- **Secondary master** (now more commonly called a **slave**) — pulls the zone's data from another authoritative nameserver, called its **master server**. That master is *usually* the primary, but doesn't have to be — a slave can load from another slave. The act of pulling this data over is a **zone transfer**.

Both primaries and slaves are **equally authoritative** for the zone — "slave" is a role name, not a statement about data quality or trust. DNS built in this primary/slave split specifically to make administration tractable: once your primary's zone data is correct, you don't hand-copy it to every other nameserver — you just point slaves at the primary, and zone transfers keep them in sync automatically.

**Why you want slaves at all:** redundancy (survive a nameserver outage), load distribution, and locality (a nearby nameserver answers faster than a distant one).

One more nuance: labeling a *nameserver* (not a zone) as "primary" or "slave" is technically imprecise, since a single nameserver can be primary for one zone and slave for another simultaneously. In practice, most nameservers lean overwhelmingly one way or the other, so the label is used loosely to mean "primary/slave for most of what it serves."

---

## 10. Zone Datafiles

The files a primary master reads its zone data from are **zone datafiles** (often just "datafiles"). Slaves can also persist their transferred data to datafiles locally — so that if a slave restarts, it can reload from its own backup copy first, check whether it's still current, and only pull a fresh zone transfer if needed. This has two benefits: it avoids unnecessary transfers, and it gives the slave something to serve even if the master happens to be down at restart time.

Datafiles are made of resource records describing every host in the zone, plus any delegation markers for child zones. BIND also supports an `$INCLUDE`-style directive to pull in other datafiles — directly analogous to C's `#include`.

---

## 11. Resolvers

**Resolvers are the client side of DNS.** Any program that needs namespace information — `ssh`, `ftp`, a web browser, a mail server — goes through a resolver to get it. The resolver's job, precisely, is:

1. Construct and send the query to a nameserver.
2. Interpret the response — either resource records or an error.
3. Hand the result back to whatever program asked for it.

In BIND, the resolver isn't even a separate running process — it's a set of **library routines** linked directly into client programs. It is deliberately dumb: it can build a query, send it, wait, and retry on timeout — and that's about it. It does **not** know how to follow referrals or navigate the namespace itself. This is called a **stub resolver**, and it's the reason virtually all of the resolution intelligence lives in the nameservers, not the clients. (Historically, some DNS implementations built smarter resolvers capable of following referrals themselves — but that's not the model the Internet actually runs on.)

---

## 12. Resolution

**Resolution** is the process of a nameserver tracking down an answer it doesn't already have. Because the namespace is a single inverted tree, a nameserver needs exactly one bootstrap fact to reach *any* point in that tree: the names and addresses of the **root nameservers**. From the root, every other answer is reachable by successive referral.

### Root nameservers

The roots know, at minimum, which nameservers are authoritative for every top-level zone (some roots are even directly authoritative for certain gTLDs themselves). Given any query, a root can point you to the right TLD servers; the TLD servers can point you to the right second-level zone servers; and so on, each hop getting "closer" to the answer.

Because the roots are the mandatory entry point whenever no other information is cached, they are structurally critical — if all of them were simultaneously unreachable, **all resolution on the Internet would fail**. That's why the Internet runs **13 logical root nameservers**, deliberately spread across networks and geographies (at the time of writing: one on a commercial backbone (PSINet), one on the NASA Science Internet, two in Europe, one in Japan). In reality, each of those 13 "logical" servers is often many physical machines behind load balancing or shared-unicast — but logically, there are 13. Query volume at the roots is enormous — tens of thousands of queries per second per server in informal measurements — yet the system holds up well in practice, largely because of caching (Section 13).

### A worked resolution trace: `girigiri.gbrmpa.gov.au`

1. Local nameserver asks a **root** nameserver for the address → referred to the **`au`** nameservers.
2. Local nameserver asks an **`au`** nameserver the same question → referred to **`gov.au`** nameservers.
3. Local nameserver asks a **`gov.au`** nameserver → referred to **`gbrmpa.gov.au`** nameservers.
4. Local nameserver asks a **`gbrmpa.gov.au`** nameserver → gets the actual answer.

Notice: the local nameserver asked the *exact same question* at every hop — it never asked "who is authoritative for `gov.au`?" as a separate query. It simply kept re-asking for `girigiri.gbrmpa.gov.au`'s address, and let each server refer it one level closer.

## 13. Recursion vs. Iteration

This is the piece that explains *why* the resolution trace above looks the way it does — and why one nameserver in that chain did all the work while the others didn't.

- A **recursive query** obliges the queried nameserver to return either the actual answer or a definitive error (no such data, no such name) — it is **not allowed** to just hand back a referral. If it doesn't know the answer itself, it must go find it by querying other nameservers on the client's behalf.
- An **iterative (nonrecursive) query** allows the queried nameserver to simply return the best information it already has — which might just be a referral to someone closer to the answer, with no obligation to chase it further.

**Why the resolver always sends a recursive query:** stub resolvers (Section 11) aren't smart enough to follow a referral themselves. So the resolver's single query to its local nameserver *must* be recursive — it's offloading all the referral-chasing work onto that nameserver.

**What the local nameserver does with that obligation:** by convention/politeness, current implementations resolve recursion by issuing their *own* queries as **iterative** ones to the servers further up/out the chain, manually following each referral themselves rather than passing the recursive burden further along. (The one common exception: a nameserver configured to use a designated **forwarder**, to which it does send a recursive query itself.)

**Efficiency shortcut — "closest known" nameservers:** when a nameserver must resolve a recursive query itself, it doesn't necessarily start at the root. It checks what it already knows, starting from the most specific ancestor domain and working upward: does it know the nameservers for `girigiri.gbrmpa.gov.au`? If not, `gbrmpa.gov.au`? If not, `gov.au`? If not, `au`? The one guaranteed stopping point is the root zone, since every nameserver is preloaded with root server addresses. This is precisely why a `berkeley.edu` nameserver resolving `waxwing.ce.berkeley.edu` doesn't need to bother the root at all — it can jump straight to what it already knows.

**A crucial detail about which query gets forwarded:** at every step, the nameserver forwards the *exact same original query* — never a synthetic query like "who is authoritative for `ce.berkeley.edu`?" This matters for two reasons: (1) there might not even *be* separate nameservers for that intermediate domain — `ce.berkeley.edu` might just be part of the `berkeley.edu` zone with no independent delegation — and (2) some higher-up nameserver might, unexpectedly, already have the specific answer cached, and a narrower synthetic query would miss that.

### Iteration, from the other side

When a nameserver receives an **iterative** query, its job is much lighter: check local data (including cache) for the answer; if found, return it; if not, return the best referral it has — the names/addresses of the nameservers closest to the queried name that it happens to know about. It does *not* go query anyone else on the requester's behalf. It's also worth noting: a referral response includes *all* the nameservers the responding server knows for that zone — picking which one to try next is left entirely up to whoever is doing the iterating.

### Choosing among equally-authoritative nameservers

When several nameservers are all authoritative for the same zone (say, the 13 roots), how does a BIND nameserver pick which one to actually query? It uses **round-trip time (RTT)**: every query it sends starts an internal stopwatch, and every response stops it, recording how fast that particular remote server answered. Next time it needs to query servers for that zone, it simply favors whichever one has historically responded fastest.

Before any real RTT data exists for a given server, it's assigned an artificially low placeholder RTT — lower than any real value could be — which forces the nameserver to try every authoritative server at least once, in effectively random order, before it starts genuinely favoring the fastest one. It's a simple mechanism, but it lets a BIND nameserver "lock on" to nearby, fast servers automatically, with zero external monitoring infrastructure required.

### The whole picture, end to end

A single resolution, seen as a whole: the resolver sends **one recursive query** to its local nameserver (call it A). A then issues a sequence of **iterative queries** — to B, then (after B's referral) to C, then (after C's referral) to D — each time getting either a referral (pointing further down the tree) or, finally, the actual answer from D. A then returns that answer to the original resolver. Along the way, **every single response A receives — referral or final answer — updates A's RTT table** for that responding server, sharpening future server-selection decisions.

## 14. Mapping Addresses to Names (Reverse Lookups)

Everything above assumes you're going *name → data* (e.g., name → address). But the reverse direction — *address → name* — is genuinely useful: readable log files, and authorization checks (classic Unix `.rhosts`/`hosts.equiv` checks compare against reverse-mapped names).

With a flat host table, reverse lookup was trivial: linear scan for the address, return the matching name. **DNS's data is indexed by name**, though, so a literal reverse lookup would seem to require scanning every domain name in existence — clearly unworkable at Internet scale.

**The clever fix:** build a *parallel* branch of the namespace where the labels *are* IP address octets, so reverse lookups become ordinary name→data lookups. This branch is the **`in-addr.arpa`** domain (this is the "still in active use" callback from Section 5's mention of `arpa`).

Mechanically: `in-addr.arpa` can have up to 256 subdomains (one per possible value of the IP address's first octet), each of which can have up to 256 subdomains of its own (second octet), and so on — bottoming out four levels down, where a resource record attached to the final octet gives back the full domain name for that address.

**The address is written backward in the tree**, because domain names are always read leaf-to-root. If `winnie.corp.hp.com`'s address is `15.16.192.152`, its reverse-mapping node is `152.192.16.15.in-addr.arpa`.

**Why backward, specifically — this is a genuinely elegant design point, not an accident:** IP addresses are hierarchical too, but in the *opposite* direction from domain names. Domain names get *less* specific left to right (`daphne.ch.apollo.hp.com` — most specific label first). IP addresses get *more* specific left to right (`15.16.192.152` — network number first, host-within-network last). By putting the *first* IP octet at the *top* of the `in-addr.arpa` tree, delegation lines up with how network numbers are actually allocated: the `15.in-addr.arpa` zone (everything starting with `15.`) can be delegated as a single unit to whoever administers network `15/8`. Had the octets been ordered the other way, `15.in-addr.arpa` would mean "every address *ending* in 15" — scattered across every network on the Internet, and utterly undelegatable as a coherent administrative unit.

## 15. Caching

Full recursive resolution, as traced above, can involve several hops — yet DNS in practice is fast. **Caching** is the main reason why.

Every time a nameserver resolves a recursive query on someone's behalf, it necessarily *learns* things along the way: every referral it receives tells it "these servers are authoritative for this zone, at these addresses" — and it stores that. When it finally gets the actual answer at the end of the chain, it caches that too.

**Negative caching**, which BIND implements: if an authoritative server responds "this name doesn't exist" or "this record type doesn't exist for this name," the querying nameserver caches *that* fact too, temporarily — so repeated queries for a nonexistent name don't repeatedly walk the whole tree.

**Compounding effect:** once a nameserver has resolved `eecs.berkeley.edu`, it has cached not just that answer but the `berkeley.edu` nameservers' identities along the way. A subsequent query for a *different* name in the same tree — say, `baobab.cs.berkeley.edu` — skips the root entirely: the nameserver recognizes `berkeley.edu` as the closest known ancestor and starts there directly, cutting the resolution chain down significantly.

**Two compounding benefits, worth naming explicitly:**
1. Faster answers for the client (fewer hops).
2. Reduced load on the root nameservers specifically — since cached knowledge of intermediate zones means most queries never need to touch the roots at all. This is *the* mechanism that keeps the 13 root servers viable despite carrying the entire Internet's namespace as ultimate fallback.

### Time to Live (TTL)

Cached data can't live forever, or updates at the authoritative source would never propagate — remote nameservers would just keep serving stale cached answers indefinitely. So the zone's administrator sets a **TTL** on the data: the maximum time any nameserver is allowed to hold it in cache before discarding it and re-fetching from the authoritative source. TTL applies to **negative** caching too — a "doesn't exist" answer also has to expire, in case the record gets added later.

**The TTL trade-off, stated as sharply as the source material states it:**
- **Small TTL** → higher consistency (changes propagate fast across the network) but **more load** on your authoritative nameservers and **slower average resolution** for your zone's data (more frequent re-fetches).
- **Large TTL** → faster average resolution (data stays cached longer, fewer round trips) but **longer-lived inconsistency** if you change something — remote caches keep serving the old answer until their TTL finally expires.

Setting TTL is, in the end, a direct trade-off between **performance** and **consistency** — and it's a decision the zone administrator makes deliberately per-record, not something DNS decides for you.

## How the pieces connect — the full chain

To tie all fifteen sections together in one pass: the **namespace** (§1–3) is a tree; **resource records** (§4) hang data off nodes in that tree; the **Internet's actual namespace** (§5) follows organizational and geographic naming traditions on top of that raw tree structure; **delegation** (§6) is how administration of subtrees gets handed off; **zones** (§7–8) are the concrete, delegation-bounded units nameservers actually load and serve; **primary/slave nameservers and datafiles** (§9–10) are how that zone data gets created, replicated, and kept redundant; **resolvers** (§11) are the dumb clients that kick off the whole process; **resolution, recursion, and iteration** (§12–13) are the algorithm by which an answer gets tracked down across that delegated tree, hop by hop from the root; **reverse mapping** (§14) is a clever reuse of the exact same name→data indexing to solve the opposite problem; and **caching with TTLs** (§15) is what keeps the whole system fast enough to be usable at Internet scale, by exploiting everything a nameserver incidentally learns while doing its job.
