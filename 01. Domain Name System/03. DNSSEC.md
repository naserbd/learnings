# DNSSEC Teaching Guide

---

# 1. What is DNSSEC?

Let's take this text apart piece by piece and build up a real mental model, not just a definition to memorize.

## "DNS was designed in a day and age when the Internet was a friendly and trusting place"

This is the historical root of the whole problem. DNS was designed in the early 1980s (RFC 1034/1035, 1987) for a small, cooperative research network. There was no adversarial threat model baked into the protocol — no one designing DNS in 1983 was thinking "what if someone forges a response to redirect a bank's website." The protocol trusts that whatever answer arrives, on the wire, matching the query ID, is the truth.

That assumption became dangerous once DNS became critical infrastructure for a global, adversarial internet — most famously demonstrated by the **Kaminsky attack** (2008), which showed how trivially an off-path attacker could forge DNS responses and poison a resolver's cache by guessing a 16-bit query ID and racing the real answer.

## "The protocol itself provides little protection against malicious or forged answers"

Plain DNS (over UDP, typically) has almost no built-in authenticity check. The only things tying a response to a query are:
- Source IP/port matching (spoofable)
- The 16-bit Query ID (guessable, especially with off-path attacks)
- The question name/type matching

None of these cryptographically prove the response came from the real authoritative server or wasn't altered in transit. A resolver just has to *trust* what shows up.

## "DNSSEC ... adds digital signatures into DNS data"

This is the core mechanism. DNSSEC doesn't change *how* DNS answers are looked up — it adds **new resource record types** that carry cryptographic signatures alongside the existing data (A, AAAA, MX, etc.). The key record types you'll meet later in this guide:

- **RRSIG** — the digital signature over a set of records
- **DNSKEY** — the public key used to verify signatures
- **DS** — a hash of a child zone's key, stored in the parent zone (this is what builds the chain of trust)
- **NSEC/NSEC3** — used to prove a name *doesn't* exist (authenticated denial of existence)

## "each DNS response can be verified for integrity ... and authenticity"

Two distinct security properties, worth separating clearly because people conflate them:

- **Integrity** = the data wasn't tampered with in transit. A hash/signature check catches even a single bit-flip.
- **Authenticity** = the data genuinely originated from the zone's legitimate owner, not an impostor pretending to be `example.com`.

DNSSEC gives you both via public-key cryptography: the zone owner signs their records with a private key; anyone can verify with the corresponding public key.

## "In the ideal world when DNSSEC is fully deployed, every single DNS answer can be validated and trusted"

Important nuance: this is aspirational. DNSSEC deployment today is still partial — plenty of zones are unsigned, and plenty of resolvers don't validate. DNSSEC is not "on" globally; it works zone-by-zone and resolver-by-resolver. An unsigned zone simply provides no DNSSEC protection — it's not a security failure, it's the absence of the feature for that name.

## "DNSSEC does not provide a secure tunnel; it does not encrypt or hide DNS data"

This is one of the most common misconceptions among people new to the topic, so it's worth dwelling on. **DNSSEC ≠ encryption.** Anyone snooping traffic can still see every query and answer in plaintext. DNSSEC only proves *authenticity and integrity* — it says nothing about *confidentiality*.

If you want confidentiality of DNS traffic, that's a different, unrelated technology: **DNS over TLS (DoT)** or **DNS over HTTPS (DoH)**. Don't let students merge these concepts — they solve completely different problems and are often deployed together but are independent layers.

## "It operates independently of an existing Public Key Infrastructure (PKI). It does not need SSL certificates or shared secrets."

DNSSEC has its **own** trust model, separate from the Web PKI (the certificate authorities that issue TLS certs for HTTPS). There's no "DNSSEC Certificate Authority" issuing certificates. Instead:

- Trust is established through a **chain of trust** that mirrors the DNS hierarchy itself: the root zone signs the TLD's key hash (DS record), the TLD signs the second-level domain's key hash, and so on down.
- The root's own trust anchor is distributed out-of-band (built into resolver software/configuration) — this is the one place where trust has to start somewhere, similar in spirit to a root CA, but it's a completely separate hierarchy from Web PKI.
- No shared secrets are involved (unlike, say, TSIG which uses a shared symmetric key for securing zone transfers) — DNSSEC uses purely asymmetric (public/private key) cryptography.

## "It was designed with backwards compatibility in mind, and can be deployed without impacting 'old' unsecured domain names"

DNSSEC records are just additional RRs sitting alongside the classic ones. A non-DNSSEC-aware resolver simply ignores RRSIG/DNSKEY/DS records it doesn't understand and gets the normal answer as before. This incremental-deployability was a deliberate design goal — you didn't need to flag-day the entire internet to roll it out. This is also why deployment could happen zone-by-zone over two decades rather than needing a coordinated global cutover.

## The Three Components

The text sets up a really useful mental framework — think of DNSSEC as touching three distinct roles, each with a different job:

| Component | Who runs it | What they do under DNSSEC |
|---|---|---|
| **Recursive server** (resolver) | ISPs, enterprises, public resolvers (e.g. Google 8.8.8.8, Cloudflare 1.1.1.1) | Enable **validation** — cryptographically check signatures on every answer before trusting/caching it |
| **Authoritative server** | Domain owners / zone operators | **Sign** their zone data — generate RRSIGs, publish DNSKEY/DS records |
| **Application/stub resolver** | Every end-user device/app | Ideally also validates, or at minimum trusts the "AD" (Authenticated Data) flag its upstream recursive resolver sets |

Notice the asymmetry: **signing** and **validating** are two completely separate operational responsibilities performed by two different parties, and one is meaningless without the other existing somewhere in the chain. If nobody signs, there's nothing to validate. If nobody validates, signing gives you no actual protection against a determined attacker — the signatures are there but nobody's checking them.

This guide (per the text) focuses on the first two — recursive and authoritative — since those are where the meaningful operational work (and BIND configuration) happens.

---

**Quick mental checkpoint before moving on:**
- DNSSEC = authenticity + integrity, **not** confidentiality
- Its own trust hierarchy, **not** tied to Web PKI/SSL
- Deployed incrementally, zone by zone, resolver by resolver
- Three players: **signer** (authoritative), **validator** (recursive), **consumer** (application)

# 2. What Does DNSSEC Add to DNS?

## Primer on Public Key Cryptography

### "Public Key Cryptography works on the concept of a pair of keys, one is made available to the world publicly, and one is kept in secrecy privately."

This is **asymmetric cryptography**, and it's the foundation everything else in this chapter builds on. Unlike symmetric cryptography (one shared secret key, like a house key you'd hand to a trusted friend), asymmetric crypto uses a **mathematically linked pair** of keys:

- **Private key** — never leaves the owner's possession. This is the secret.
- **Public key** — freely distributed to anyone, published even.

The mathematical relationship (based on problems like integer factorization for RSA, or elliptic curve discrete logarithm for ECDSA) means: something done with one key in the pair can only be undone/verified with the *other* key in the pair. Neither key alone reveals the other.

### "Think of it as a cleverly designed lock, where one key locks, and one key unlocks."

This is a nice simplified analogy for a *general* audience, but let's sharpen it for engineers because DNSSEC specifically uses public-key crypto for **signing**, not **encrypting**. It's worth distinguishing two different use-cases of asymmetric crypto that beginners often conflate:

| Use case | Who uses private key | Who uses public key | Purpose |
|---|---|---|---|
| **Encryption** (e.g., someone sending you a secret) | Sender encrypts *with your public key* | You decrypt *with your private key* | Confidentiality |
| **Digital signing** (DNSSEC's use case) | Signer signs *with their private key* | Everyone else verifies *with signer's public key* | Authenticity + integrity |

DNSSEC uses the **signing** model exclusively. The zone owner's private key produces a signature over the DNS data; anyone holding the corresponding public key can mathematically check "yes, this signature could only have been produced by the holder of the matching private key, and the data hasn't changed since it was signed." That's why the guide's framing of "locking" (private key, kept secret, used to sign) vs. "unlocking" (public key, published, used to verify) is the correct direction for DNSSEC — the private key locks (signs), the public key unlocks (verifies) — but nothing is actually being *encrypted/decrypted* here in the confidentiality sense. Keep that distinction crisp in your head, because it's the single most common point of confusion for engineers new to DNSSEC.

### "In DNSSEC, we give out the unlocking public key to the rest of the world, while keeping the locking key private."

Concretely:
- The **private key** lives only on (or near) the authoritative name server / signing infrastructure of the zone owner. It's used offline or in a signer process to generate RRSIGs.
- The **public key** is published *inside the DNS itself*, as a DNSKEY record, so any resolver in the world can fetch it and verify signatures — no separate PKI, no out-of-band certificate distribution needed.

---

## The Six New Resource Record Types

DNSSEC's entire mechanism is implemented by layering **six new RR types** on top of classic DNS.

### RRSIG — Resource Record Signature

> "just about every DNS answer... will come with at least one RRSIG"

Key detail worth internalizing early: **RRSIG doesn't sign individual records — it signs an entire RRset** (all records of the same name+type, e.g., all the A records for `www.example.com`). This matters operationally: if `www.example.com` has three A records (round-robin), one RRSIG covers all three together as a set, not three separate signatures.

Each RRSIG carries metadata alongside the raw signature bytes: which key/algorithm signed it, an **inception** and **expiration** timestamp (DNSSEC signatures are time-bound and must be periodically re-signed — this is a major operational difference from most PKI certs which last months/years; DNSSEC signatures are often re-generated every days-to-weeks), and the original TTL of the covered set.

### DNSKEY — the public key record

> "two categories of keys ... ZSK ... and KSK"

- **ZSK (Zone Signing Key)** — signs the bulk of ordinary zone data (A, MX, CNAME, etc.). Because it signs a large, frequently-changing volume of data, it's typically rotated more often and kept "closer to the signing process" operationally.
- **KSK (Key Signing Key)** — signs only the DNSKEY RRset itself (i.e., it signs the keys, not the zone data). Because the KSK is what the parent zone's DS record ultimately points to (forming the chain of trust), it's treated as more sensitive/critical and typically rotated less frequently, with more ceremony.

Think of it as a two-tier signing hierarchy *within a single zone*: KSK vouches for the ZSK, ZSK vouches for everything else. This separation lets an operator rotate the high-volume ZSK routinely without having to touch the parent zone's DS record every time — only KSK rotations require coordinating a DS update with the parent.

### DS — Delegation Signer

> "the parent zone can 'vouch' for its child zone"

This is the record that literally *stitches* the chain of trust together across zone boundaries. A DS record does **not** contain a public key itself — it contains a **cryptographic hash** of the child zone's KSK (its DNSKEY record). It lives in the **parent** zone (e.g., the `.com` zone holds the DS record for `example.com`).

The logic: a validating resolver walking down from the root can, at each delegation point, fetch the child's DNSKEY, hash it, and compare that hash against the DS record the parent already vouched for. If they match, trust is transitively extended one level down. This DS-to-DNSKEY hash check, repeated recursively from the root downward, *is* the chain of trust.

### NSEC / NSEC3 / NSEC3PARAM — Authenticated Denial of Existence

> "proving that something really does not exist"

In plain DNS, if you query for a name that doesn't exist, you get NXDOMAIN — but there's no data to *sign*, because there's nothing there. So how do you cryptographically prove a negative — that `nonexistent.example.com` truly has no records, rather than an attacker forging a fake NXDOMAIN to censor a real answer? DNSSEC's answer: a zone pre-signs statements of the form "there is nothing between name X and name Y in this zone" by walking the sorted list of all names in the zone and chaining them together (NSEC), or a privacy/enumeration-resistant hashed variant of the same idea (NSEC3, with NSEC3PARAM carrying the hashing parameters). The concept to hold onto: **DNSSEC can authenticate absence, not just presence** — that's a nontrivial cryptographic feat compared to just signing existing data.

---

## Quick mental model to carry forward

- **RRSIG** = the signature (proves *this specific data* is authentic/intact)
- **DNSKEY** = the public key (used to check RRSIGs) — split into **ZSK** (signs data) and **KSK** (signs keys)
- **DS** = the parent's hash-based vouch for the child's KSK (the glue of the chain of trust)
- **NSEC/NSEC3/NSEC3PARAM** = proof that *nothing* exists, when nothing exists

# 3. How Does DNSSEC Change DNS Lookup?

## Traditional (Insecure) DNS Lookup, Recapped

> "a recursive name server receives a query from a client... tracks down the authoritative name server(s)... sends the query... waits for the answer."

This is the baseline: **iterative resolution**. The recursive resolver starts at the root, gets referred to `.org`, gets referred to `isc.org`'s name servers, and finally asks `isc.org` directly for `www.isc.org`. In plain DNS, once an answer comes back matching the query, the resolver just accepts it. No cryptographic scrutiny.

## What Changes With Validation Enabled

> "a validating recursive name server... will ask for additional resource records in its query"

This is the mechanical trigger: the resolver sets the **DO (DNSSEC OK) bit** in its EDNS0 extension on every outgoing query. This is a signal, not a guarantee — it tells the authoritative server "if you have DNSSEC data for this, please include it." A non-DNSSEC-aware authoritative server simply won't have anything extra to send back, and the resolver falls back to unvalidated behavior for that zone.

> "the validating resolver will perform cryptographic computation to verify the authenticity... and even ask the parent zone as part of the verification"

**Validation is not a single check — it's a recursive, upward-climbing chain of checks.** Verifying one answer requires verifying the key that signed it, which requires verifying *that* key was legitimately vouched for by a parent, which requires verifying the parent's own key, and so on — all the way up to the root.

---

## The 12-Step Walkthrough: Resolving `www.isc.org`

### Round 1 — Verifying the answer from `isc.org` (Steps 1–4)

**Step 1:** Resolver asks `isc.org`'s name server for the A record of `www.isc.org`, with the DO bit set — "please include DNSSEC proof."

**Step 2:** `isc.org` responds with the A record **and** an RRSIG — the digital signature covering that A RRset.

**Step 3:** To check that RRSIG, the resolver needs the actual public key that produced it. So it separately queries `isc.org` for its DNSKEY record.

**Step 4:** `isc.org` responds with its DNSKEY(s) — and notably, *those keys are themselves signed too* (the ZSK's use is vouched for by an RRSIG generated with the KSK). At this point the resolver can mathematically confirm: "the A record in step 2 really was signed by the private key matching this public key, and nothing was altered in transit."

**The catch:** this only proves internal self-consistency. An attacker who fully compromised `isc.org`'s name servers could generate their *own* matching key pair and sign fake answers with it — steps 1–4 would look perfectly valid on their own. Self-signed data can never prove *who* it is — only that it's internally consistent. You need an outside party to vouch for the key itself. That's exactly the problem DS records exist to solve.

### Round 2 — Verifying `isc.org` via its parent, `.org` (Steps 5–8)

**Step 5:** Resolver asks the parent zone, `.org`, for the verifiable information it holds about its child `isc.org` — this is the **DS record**, that hash of `isc.org`'s KSK.

**Step 6:** `.org` sends back that DS record. The resolver hashes the DNSKEY it got in step 4 and compares it against this DS value. Match → this proves `.org` (an independent party) vouches for exactly this key.

But — same problem, one level up: what if the attacker also compromised `.org`'s servers? So the resolver repeats the exact same pattern:

**Step 7:** Resolver asks `.org` for its DNSKEY, to verify the signature on the DS data received in step 6.

**Step 8:** `.org` responds with its DNSKEY (and signature). Resolver verifies the step-6 data is authentic — but again, only self-consistently proven so far.

### Round 3 — Verifying `.org` via *its* parent, the root (Steps 9–12)

**Step 9:** Resolver asks the root zone for the DS record it holds on `.org`.

**Step 10:** Root responds with that DS record. Resolver hashes `.org`'s DNSKEY (from step 8) and compares — match confirms root vouches for `.org`'s key.

**Steps 11–12:** Same closing-the-loop pattern once more — resolver asks root for *its own* DNSKEY, root responds, and the resolver can now verify the DS data received in step 10 was authentically signed by root.

---

## 3.1 Chain of Trust — Where Does It End?

> "what about the root server itself? Who do we go to verify root's keys? There's no parent zone for root."

**The chain of trust cannot be infinite, and it cannot bootstrap itself from nothing.** Every "ask the parent" step in rounds 1–3 pushed the trust problem one level up. At the root, there's nowhere left to push it — the buck stops there.

The resolution: the resolver is configured, **out-of-band, ahead of time**, with the root zone's public key (or a trust anchor derived from it) — this is called the **trust anchor**. It's not fetched dynamically and verified via some DNS mechanism; it's installed into the resolver's configuration directly.

So the final check isn't "ask someone" — it's "compare what root just sent you (steps 11–12) against the copy you already had on file." If those match, the entire chain — root → `.org` → `isc.org` → the actual A record — is now cryptographically proven, and the answer can be trusted.

> "In the ideal, fully deployed world of DNSSEC, all validating resolvers only need to trust one key: the root key."

**A single trust anchor (the root key), installed once, transitively extends trust across every signed zone on the internet** — no matter how deep the delegation chain, the pattern from rounds 1–3 just repeats one more level for every extra layer of delegation.

---

## Mental Model to Carry Forward

```
For a given zone Z:
  1. Get the answer + RRSIG from Z
  2. Get Z's DNSKEY, verify the RRSIG
  3. Get the DS record for Z from Z's parent
  4. Hash Z's DNSKEY, compare to the DS record → confirms parent vouches for Z
  5. Repeat this whole unit one level up, treating "parent" as the new Z
  ...until you reach the root, whose key you already trust from local config
```

- This whole climbing process is called **chain of trust**.
- The pre-installed root key that terminates the chain is the **trust anchor**.

# 4. Why Is DNSSEC Important?

## 1. Be a Good Netizen

> "By enabling DNSSEC validation... you're protecting your users... by signing your zones... you are making it possible for other people to verify your zone data."

**Signing and validating are two separate, independently valuable actions, and neither one protects you against the same threat as the other.**

- If you run a **recursive resolver** and enable **validation**, you protect *your users* from being fed forged/tampered answers for *any* signed domain they look up.
- If you run **authoritative servers** and **sign your zone**, you protect *your domain's visitors* — anyone anywhere in the world who queries your zone through a validating resolver.

Signing your own zone does nothing to protect *you* as a resolver operator, and enabling validation does nothing to protect *visitors to your domain*. They're complementary, not substitutes. DNSSEC's value for any single zone only exists if *someone* downstream is validating, and validation only matters if *someone* upstream has signed.

## 2. Compliance

> "You may not even get a say whether or not you want to implement DNSSEC, if your organization is subject to compliance standards that mandate it."

The US government's 2008 mandate requiring all .gov subdomains to be signed by December 2009, and PCI DSS references, and ICANN's requirement that new TLDs support DNSSEC, are all examples of **time-bound, jurisdiction-specific facts** — treat these as illustrative of *the kind of pressure* compliance creates, not as current, actionable compliance guidance. Verify current standards directly rather than relying on any guide's snapshot.

The teaching takeaway: compliance-driven DNSSEC deployment is common in practice, and sometimes the *decision* to deploy DNSSEC is made above your head, by legal/regulatory/contractual obligation rather than by a security team's risk assessment.

## 3. Enhanced Security (C.Y.A. — Cover Your Ass)

> "in case of a DNS-based security breach, such as cache poisoning or domain hijacking... you might be placed under scrutiny for any preventative measure that could have been put in place."

This is a risk-management argument: DNSSEC is a **known, standardized, well-documented mitigation** against specific, well-understood attack classes — cache poisoning (like the Kaminsky attack) and domain hijacking. If such an incident occurs and DNSSEC was *not* deployed, that absence becomes a legitimate line item in any post-incident review.

> "Think of this like having your web site only available via HTTP but not HTTPS."

It's not that unsigned DNS is instantly and constantly under attack — it's that the *absence of a well-known, low-friction protection* is increasingly hard to justify once the attack techniques it defends against are public knowledge and the tooling to deploy the defense is mature. It's a baseline-hygiene argument, not a "you will definitely be attacked" argument.

## 4. New Features

> "Once DNS can be trusted completely, it becomes possible to publish SSL certificates in DNS, or PGP keys for fully automatic cross-platform email encryption, or SSH fingerprints..."

DNSSEC **upgrades what DNS is trusted to carry**. Once an answer from DNS can be cryptographically proven authentic, DNS becomes a viable place to publish *other* security-sensitive material. Concrete examples worth knowing by name:

- **TLSA records (DANE)** — publishes which TLS certificate/CA is legitimate for a domain, checkable in addition to or instead of Web PKI. This is the "SSL certificates in DNS" reference.
- **OPENPGPKEY records** — publishing PGP public keys in DNS so mail clients can automatically discover a recipient's key.
- **SSHFP records** — publishing SSH host key fingerprints in DNS, so an SSH client can automatically verify a server's host key against DNS instead of trust-on-first-use.

The unifying idea: **all of these mechanisms only make sense if DNS itself is trustworthy** — DNSSEC is the prerequisite infrastructure that makes this entire category of "store security material in DNS" features meaningful rather than a false sense of security.

---

## Mental Model to Carry Forward

| Reason | Best audience | Core argument |
|---|---|---|
| Good netizen | Security-minded engineers | Network-effect: signing + validating both add value, and value compounds as more zones/resolvers adopt it |
| Compliance | Management / legal | Sometimes not optional — mandated by regulation or contract |
| C.Y.A. | Risk / incident response | Known mitigation for known attack classes; absence is defensible-in-hindsight liability |
| New features | Forward-looking architects | DNSSEC upgrades DNS into a trustworthy place to publish other security material (TLSA, OPENPGPKEY, SSHFP) |

# 5. Getting Started

This chapter is the "before you touch a config file" checklist — software, hardware, network, and operational prerequisites. **Note:** this guide contains specific version numbers and adoption statistics that are now quite old (BIND 9.10.x, .gov/.com signing percentages "as of late 2014"). The underlying *principles* are still fully valid; flag explicitly where you should go verify current numbers.

## 5.1 Software Requirements

### 5.1.1 BIND Version

`named -v` checks the installed version. The broader teaching point: **DNSSEC support and DNSSEC feature completeness varies by DNS software version**, whichever software you run (BIND, PowerDNS, Knot DNS, NSD, etc.). Always check your specific software's DNSSEC support maturity.

**Currency note:** BIND 9.10 is a genuinely old release lineage — check ISC's current supported BIND versions. The *concepts* (signing, validation, key rollover) transfer directly.

### 5.1.2 DNSSEC Support in BIND

> "Run the command named -V to see what flags it was built with. If it was built with OpenSSL (--with-openssl), then it supports DNSSEC."

**DNSSEC signing/validation requires a cryptographic library underneath the DNS server software.** BIND delegates crypto math to a library (historically OpenSSL). `named -V` shows compile-time configuration — a binary built without crypto library linkage simply cannot do the cryptographic operations DNSSEC needs.

General principle for any DNS software: verify your binary was actually **compiled with cryptographic support**.

### 5.1.3 System Entropy

Generating cryptographic key pairs requires a source of high-quality randomness — the OS's entropy pool. **Why VMs specifically struggle:** virtual machines often lack the physical hardware entropy sources (disk I/O timing jitter, keyboard/mouse interrupts, hardware RNG) that physical machines use to seed their entropy pools. A freshly-booted, headless, cloud VM might have a nearly empty entropy pool, and key generation can hang.

The fix — `haveged` — is a userspace daemon that generates entropy through CPU jitter/timing techniques. **Currency note:** modern Linux kernels (5.6+) significantly improved `/dev/urandom`/`getrandom()` behavior, and many cloud platforms now provide virtio-rng passthrough — this was a much bigger headache circa this guide's writing than it typically is today, but verify your entropy source before generating production keys.

## 5.2 Hardware Requirements

### 5.2.1 Recursive Server (Validating Resolver) Hardware

Each hardware consideration corresponds to extra work introduced by validation:

1. **CPU** — every RRSIG verification is a cryptographic operation. Scales with *how much of your query traffic touches signed zones*.
2. **Memory** — DNSSEC answers are bigger, consuming more resolver memory when cached.
3. **Network** — packets get bigger, but rarely requires NIC upgrades on modern hardware.

**The most important teaching point:** validation overhead on a resolver is driven by **what fraction of the domains your users actually query are DNSSEC-signed**, not some fixed universal overhead.

**Currency note:** the guide's cited .gov (80%+ signed) and .com (0.5% signed) figures are badly out of date. Look up current DNSSEC deployment statistics (ICANN, APNIC publish this) rather than using frozen-in-time numbers — the *method* of reasoning is what to keep.

### 5.2.2 Authoritative Server Hardware

From the *signing* side:

1. **CPU** — periodic **re-signing** is the operative cost, since RRSIGs expire and must be regenerated.
2. **Storage** — a signed zone file is larger (RRSIG, DNSKEY, NSEC/NSEC3 records for essentially every name). "Expect at least 3x growth" is a reasonable anchor, though actual growth depends on algorithm and NSEC vs. NSEC3.
3. **Memory** — follows from storage.
4. **Network** — again, rarely requires hardware upgrades.

**Currency note:** the guide's 2014 client-side validation-awareness statistics (10-30%) are outdated. Major public resolvers (Google, Cloudflare, Quad9) validate by default today. The principle to retain: **the value of signing your zone is proportional to how many of your visitors' resolvers actually validate.**

## 5.3 Network Requirements

> "DNS and DNSSEC packets are very similar, DNSSEC packets are just bigger, which means DNS is more likely to use TCP."

Two checks map to two distinct failure modes:

1. **DNS over TCP (port 53)** — when a UDP response would be too large, DNS falls back to TCP. If firewalls only permit UDP port 53, DNSSEC-sized responses requiring TCP fallback simply fail.
2. **Large UDP packets / EDNS0** — DNSSEC relies on EDNS0 to advertise larger-than-512-byte UDP payload sizes. Some older/overly-strict middleboxes drop unusually large DNS UDP packets.

**DNSSEC failures are sometimes network infrastructure problems, not DNS configuration problems.** Checking TCP/53 reachability and large-UDP-packet handling should be an early diagnostic step.

## 5.4 Operational Requirements

### 5.4.1 Parent Zone

> "check with your parent zone administrators to make sure they support DNSSEC... a crucial step in DNSSEC deployment is to establish the parent-child trust relationship."

Signing your own zone is necessary but **not sufficient** — the DS record has to be published *by your parent zone*, pointing at your KSK, or there's no link from the trusted root down to your zone. Your zone can be perfectly signed and still be **cryptographically unreachable** from the chain of trust.

### 5.4.2 Security Requirements

> "The examples presented in this document are not intended for high value zones."

Key strength, key rotation frequency, and key storage/handling practices should scale with **what's actually at stake if a key is compromised.** A personal blog's DNSSEC keys and a bank's DNSSEC keys should not be managed with the same rigor. Treat the guide's default examples as a *learning baseline*, not a security posture appropriate for critical infrastructure.

---

## Mental Model to Carry Forward

| Requirement | Core question |
|---|---|
| **Software** | Does my DNS daemon actually support DNSSEC (crypto library linked in), and can it generate keys efficiently (sufficient entropy)? |
| **Hardware** | Have I budgeted CPU/memory for the crypto and size overhead, proportional to how much signed traffic I'll actually see? |
| **Network** | Can DNSSEC's larger packets actually get through — TCP/53 open, large UDP not dropped? |
| **Operational** | Does my parent zone support receiving a DS record, and does my key-security posture match my zone's actual value? |

**Whenever this guide (or any DNSSEC material from this era) states a specific adoption percentage or software version, treat it as illustrative of the reasoning method, and look up current figures before using them to make real decisions.**

# 6. Validation

This chapter covers: turning validation on, proving it's actually working, understanding what's happening under the hood, trust anchors, and the EDNS/network plumbing that makes it all possible.

## 6.1 Easy Start Guide for Recursive Servers

> "Once DNSSEC validation is enabled, any DNS response that does not pass the validation checks will result in the domain name not getting resolved (often a SERVFAIL status seen by the client)."

**Validation is fail-closed, not fail-open.** Unlike a lot of security features that just log-and-alert, a validating resolver actively *refuses to hand back an answer* it can't cryptographically confirm. This is a feature, but it also means **someone else's misconfiguration becomes your outage.**

> "if root name servers were misconfigured with the wrong information about .org, it could cause all validation for .org domains to fail."

DNSSEC failures aren't limited to attacks — they include honest **operational mistakes** anywhere in the chain of trust. Because the chain of trust means *every* level between root and the queried name must check out, a mistake at any single level can cascade to make the entire subtree unreachable for everyone using a validating resolver.

> "You may not need to reconfigure your name server at all, since recent versions of BIND packages... have been shipped with DNSSEC validation enabled by default."

**Currency note:** always confirm using the methods in 3.2 rather than trusting a default.

```
options {
    dnssec-validation auto;
};
```

This single line does two things: (1) turns on validation, and (2) handles **trust anchor management automatically** — meaning you don't have to manually obtain, install, and keep updating the root zone's public key yourself.

## 6.2 How to Test Your Recursive Server

A methodology — layered verification, from easiest/least-trustworthy to most rigorous/most-informative.

### 6.2.1 Web-Based Tools

Simplest tier: point your test client's DNS at your resolver and load a validation-testing website. Good for a fast sanity check, but depends on JavaScript running client-side. **Currency note:** verify these URLs are still live and maintained.

### 6.2.2 Using `dig`

**Before (DNSSEC not validating):**
```
;; flags: qr rd ra; QUERY: 1, ANSWER: 1, AUTHORITY: 0, ADDITIONAL: 0
```
Just the plain A record. No signature, no `ad` flag.

**After (DNSSEC validating):**
```
;; flags: qr rd ra ad; QUERY: 1, ANSWER: 2, AUTHORITY: 0, ADDITIONAL: 1
;; OPT PSEUDOSECTION:
; EDNS: version: 0, flags: do; udp: 4096
```

Three things to specifically train your eye to spot:

1. **`ad` flag in the header flags line** — "Authenticated Data." The resolver *telling the client*, "I checked this, and it's cryptographically verified." The single most important bit in the whole response.
2. **`do` flag in the EDNS pseudosection** — "DNSSEC OK." Confirms the *query itself* requested DNSSEC data.
3. **ANSWER count goes from 1 to 2, and an RRSIG record appears** — the actual signature riding alongside the A record.

**Subtlety worth flagging:** **`do` tells you the resolver is DNSSEC-*aware*; `ad` tells you it actually *validated this specific answer*.** You can have `do` without `ad` — don't conflate the two.

Reading the RRSIG line:
```
www.isc.org. 4 IN RRSIG A 5 3 60 ( 20141029233238 20140929233238 4521 isc.org. ... )
```
Left to right: the RRset this signs (`A` record), the signing algorithm (`5` = RSA/SHA-1), number of labels (`3`), original TTL (`60`), signature expiration, signature inception, the key tag (`4521` — identifies *which* DNSKEY to use), the signing zone name (`isc.org.`), and the base64-encoded signature bytes.

**Currency note:** algorithm `5` (RSASHA1) is now considered legacy/weak. Modern best practice favors algorithm 8 (RSASHA256) or algorithm 13 (ECDSAP256SHA256).

### 6.2.3 Using `delv`

`delv` is purpose-built for DNSSEC — its failure output is far more diagnostic:
```
;; no valid RRSIG resolving 'org/DS/IN': 192.168.1.7#53
;; no valid DS resolving 'www.isc.org/A/IN': 192.168.1.7#53
;; resolution failed: no valid DS
```
This tells you exactly *where in the chain* things broke.

`+rtrace` prints every single fetch the resolver performed:
```
;; fetch: www.isc.org/A
;; fetch: isc.org/DNSKEY
;; fetch: isc.org/DS
;; fetch: org/DNSKEY
;; fetch: org/DS
;; fetch: ./DNSKEY
```
Map this directly against the 12-step model — you can literally watch the chain-of-trust climb happen in real time.

### 6.2.4 Testing Against a Known-Bad Domain

`www.dnssec-failed.org` is a deliberately, permanently broken DNSSEC test domain — it lets you verify the *fail-closed* behavior specifically. Testing only "does a valid domain resolve" isn't sufficient — you need to also confirm "does an *invalid* domain correctly get *blocked*," or you might have a validator that's silently not actually checking anything.

### 6.2.5 Distinguishing Validation Failures from Other Failures

The `+cd` (Checking Disabled) flag tells the resolver "answer me even without doing DNSSEC validation." The diagnostic logic:

- Query fails (SERVFAIL) without `+cd`, succeeds with `+cd` → **the problem is specifically DNSSEC validation**
- Query fails both with and without `+cd` → the problem is something else entirely, not DNSSEC-related

This is your first branch point in any DNS troubleshooting flow once DNSSEC is involved.

## 6.3 Validation Easy Start, Explained

### 6.3.1 The `dnssec-validation` Options

- **`yes`** (historical default) — validation logic is on, but **you must manually supply a trust anchor** via `trusted-keys`.
- **`no`** — fully disabled, classic insecure DNS behavior.
- **`auto`** — validation on, **and** BIND supplies and self-maintains a copy of the root trust anchor for you.

> "Note dnssec-enable needs to be set to yes... in order for dnssec-validation to be effective."

Two separate config knobs need to agree — an easy thing to overlook when troubleshooting.

### 6.3.2 The 12 Steps, Revisited

Maps one-to-one onto the 1.5 walkthrough — same 12 steps, same isc.org example, now explicitly annotated with the DO bit and the specific record types fetched at each stage. If you can now read a `delv +rtrace` output and mentally map each `fetch:` line to a numbered step, you've genuinely internalized this material.

### 6.3.3 How Are Answers Verified? (The Actual Crypto)

**Generation (on the authoritative server, done ahead of time):**
1. Take the DNS record (RRset).
2. Run it through a **hash function** → produces a fixed-length digest.
3. "Encrypt" that hash with the **private key** → this encrypted hash *is* the RRSIG.

**Verification (on the validating resolver, done per-query):**
1. Receive both the plaintext RRset *and* the RRSIG.
2. Independently hash the plaintext RRset using the same algorithm → call this **hash X**.
3. Use the **public key** (fetched as DNSKEY) to "decrypt" the RRSIG → recover the original hash the signer produced → call this **hash Y**.
4. **Compare X and Y.** If they match, the data hasn't been altered (integrity) and could only have been signed by the private key holder (authenticity).

A technical precision worth adding: real digital signature algorithms (RSA, ECDSA) don't literally work by "encrypting a hash" in the way symmetric encryption does — but this simplified mental model is a reasonable first approximation.

> "although words like encryption and decryption are used from time to time, DNSSEC does not provide you with privacy."

**This whole process proves who sent the data and that it wasn't tampered with — it does not hide the data from anyone.** Anyone sniffing the wire sees the plaintext A record and the RRSIG in the clear. DNSSEC's crypto is entirely about *authentication*, never *confidentiality*.

**The timestamp check:**
```
Signature Inception:   20140929233238
Signature Expiration:  20141029233238
```
The resolver's current system clock must fall between these two timestamps. This is why:

- **RRSIGs expire and must be periodically regenerated** — an authoritative server that stops re-signing its zone will eventually have all its signatures silently expire, causing a classic, entirely self-inflicted DNSSEC outage.
- **Accurate system clocks matter enormously for DNSSEC** — NTP synchronization isn't optional infrastructure hygiene here — it's a hard DNSSEC dependency.
- The security reasoning: this defends against **replay attacks** — without expiration, an attacker who once captured a valid signed answer could keep replaying that old, stale-but-cryptographically-valid answer forever.

**The final rule:** hash match **and** valid time window **both** must hold for the `ad` bit to get set and the answer to be released. Fail either check → SERVFAIL.

## 6.4 Trust Anchors

### 6.4.1 What They Are and How They're Used

> "A trust anchor is a key that is placed into a validating resolver so that the validator can verify the results for a given request back to a known or trusted public key."

This resolves the "who verifies the verifier" problem — trust has to terminate *somewhere* it's simply asserted rather than proven, and that somewhere is the locally-configured trust anchor.

The `.gov` example shows trust anchors *aren't conceptually limited to just the root* — historically, before root itself was signed (2010), organizations wanting to validate `.gov` names manually configured `.gov`'s key directly as an additional, standalone trust anchor. The shortened 8-step process works precisely because the resolver, upon reaching a DNSKEY it recognizes as a locally-configured trust anchor, **stops climbing.**

General principle: **a trust anchor short-circuits the chain of trust at whatever level it's configured.**

### 6.4.2 Trusted Keys vs. Managed Keys

| | `trusted-keys` | `managed-keys` |
|---|---|---|
| Trust duration | Permanent, until manually removed from config | Only used to bootstrap trust, then self-updates |
| Maintenance | **Manual** — you must update it yourself when the key changes | **Automatic** — uses RFC 5011 key-rollover tracking to stay current |
| Used by | `dnssec-validation yes;` | `dnssec-validation auto;` |

> "Failing to do so will result in breaking nearly all DNS queries for the sub domain of the key... if you are manually managing the root key, you could break all DNS requests made to your recursive name server."

A stale `trusted-keys` entry isn't a minor misconfiguration — since the root key sits at the top of the entire chain of trust, letting it go stale after a root key rollover would break DNSSEC validation for **every single signed domain on the internet**, as seen through that resolver. This is precisely why `auto`/`managed-keys` is the strongly recommended default for virtually everyone.

## 6.5 EDNS — The Plumbing Underneath It All

### 6.5.1–6.5.2 Why EDNS Matters for DNSSEC

**DNSSEC rides on top of EDNS(0)**. Classic DNS assumed small (≤512 byte) UDP responses; DNSSEC's RRSIG/DNSKEY data routinely blows past that. EDNS is the extension mechanism that lets a DNS client advertise "I can accept a UDP response up to N bytes" (commonly 4096), and it's *also* the mechanism that carries the `do` bit itself. **No EDNS support → no way to even signal you want DNSSEC data in the first place.**

> "EDNS is a hop-by-hop extension to DNS... if one of your upstream name servers... does not support EDNS, you may experience DNS lookup failures."

EDNS support isn't a single global on/off switch — it has to work correctly at **every hop** in a resolution path. A weak link anywhere in that chain can degrade or break DNSSEC specifically, even when your own server is perfectly configured.

### 6.5.3–6.5.4 Network Equipment and TCP Fallback

- **MTU and fragmentation**: large EDNS/DNSSEC UDP responses may need IP fragmentation. Some **firewalls misidentify large or fragmented DNS UDP traffic as anomalous/attack traffic** and drop it, because their rule sets were written assuming DNS is always small.
- **TCP port 53**: when a UDP response would be too large even with EDNS, DNS falls back to TCP. DNSSEC substantially **increases how often** that fallback gets triggered. A network that's been quietly blocking TCP/53 for years will suddenly start breaking specifically for signed domains once DNSSEC validation is turned on.

---

## Mental Model to Carry Forward

1. **Turning it on** is one config line — but *proving it's working* takes real discipline: test a good domain (should validate) **and** a deliberately broken one (should SERVFAIL).
2. **`ad` bit** = this specific answer was validated. **`do` bit** = this query/response is DNSSEC-aware. Don't conflate them.
3. **`+cd`** is your fastest diagnostic branch.
4. Verification = hash match (integrity/authenticity) **+** valid inception/expiration window (anti-replay) — both must hold.
5. **Trust anchors** are where the chain-of-trust climb terminates by local configuration, not proof. `auto`/`managed-keys` self-maintains this safely (RFC 5011); manual `trusted-keys` is a high-blast-radius maintenance liability if neglected — especially at the root.
6. **EDNS is the substrate DNSSEC rides on** — it must work end-to-end, hop by hop, with TCP/53 as the necessary fallback path.

# 7. Signing

Where Chapter 3 was "how does a resolver verify," this chapter is "how does an authoritative server generate the material a resolver needs to verify."

## 7.1 Easy Start Guide for Signing Authoritative Zones

### 7.1.1 Generate Keys

```
# dnssec-keygen -a RSASHA256 -b 1024 example.com
# dnssec-keygen -a RSASHA256 -b 2048 -f KSK example.com
```

Two separate commands, two separate keys — the ZSK/KSK split made concrete:

- **First command (no `-f KSK` flag)** → generates a **ZSK** by default. Algorithm `RSASHA256`, key length `1024` bits.
- **Second command (`-f KSK`)** → explicitly flagged as a **KSK**. Same algorithm, but **2048** bits.

**Why does the KSK get a bigger key?** The KSK is used far less often (only signs the DNSKEY RRset) and sits at a more security-critical position (what the parent's DS record points to). Using a larger key size for it is a low-cost way to harden the more sensitive of the two keys.

**Currency note:** modern DNSSEC best practice increasingly favors **ECDSA algorithms (algorithm 13, ECDSAP256SHA256)** over RSA — smaller keys and signatures for equivalent security, helping with EDNS/packet-size concerns. 1024-bit RSA in particular is now considered a weak key size.

> "This generated four key files... Make sure these files are readable by named."

Four files total (two `.key`/two `.private` pairs, one pair per key).

### 7.1.2 Reconfigure BIND

```
zone "example.com" IN {
    type master;
    file "db/example.com.db";
    key-directory "keys/example.com";
    inline-signing yes;
    auto-dnssec maintain;
};
```

Four new pieces of config:
- `key-directory` — where to find the keys.
- `inline-signing yes` — turns on automatic, transparent signing of the zone.
- `auto-dnssec maintain` — turns on automatic key lifecycle management.

**You keep editing your zone file exactly as you did before DNSSEC existed.** BIND handles the transformation from unsigned to signed data transparently underneath.

### 7.1.3 Verification (of your own signing work)

This uses `delv` to **simulate being an external validating resolver**, checking your own freshly-signed zone *before* you've told the parent zone anything — a critical sanity check before committing to the parent-zone upload step, since once the parent publishes your DS record, any signing mistakes become visible to the entire internet.

```
$ delv @192.168.1.13 -a /tmp/example.key +root=example.com example.com. SOA +multiline
; fully validated
```

The `-a /tmp/example.key` flag is doing exactly what a real trust anchor does — manually telling `delv` "trust this specific key," letting you validate in isolation without needing the full chain of trust already in place. **You can always test a signed zone locally by manually trusting its KSK, independent of whether the parent delegation is wired up yet.**

### 7.1.4 Upload to Parent Zone

This is the step that actually **completes the chain of trust** — signing your zone alone does nothing for the outside world until this step happens.

> "before you complete this final step with your parent zone, your zone is still insecure... a validating resolver will... fall back to the insecure DNS."

**A signed-but-not-yet-delegated zone is not "broken" from a validating resolver's perspective — it's simply treated as unsigned.** This is sometimes called being in an "island of security." It fails open (falls back to insecure lookup), not closed (SERVFAIL) — because the parent's own signed data correctly and authentically states "this child has no DS record," which is a valid, provable statement, not an error. Contrast this with a *broken* DS-to-DNSKEY match — that scenario *does* fail closed with SERVFAIL, because the parent asserts a DS record that no longer matches, a genuine inconsistency.

> "verify that your newly signed zone has propagated to all of your name servers... before uploading to your parent zone."

**Sign → verify propagation to all your own authoritative servers → then tell the parent.** If you upload the DS record before all your slaves have the signed zone, a validating resolver querying a slave still serving old unsigned data, while the parent tells the world "this zone is signed," produces validation failures.

Three upload formats — DS, DNSKEY, Trusted Key — are just different serializations of the same underlying information, because different registrars have standardized on different submission formats. The end result, regardless of format: **the parent zone ends up publishing a DS record about your zone.**

### 7.1.6 Your Zone, Before and After

Notice **the DNSKEY RRset is signed by both the ZSK and the KSK.** Why both? Because either key alone could plausibly validate it — but more fundamentally, per the ZSK/KSK design, **the KSK is specifically the key whose signature over the DNSKEY RRset is the one the parent's DS record ultimately anchors trust to.** The ZSK's signature over its own DNSKEY RRset is largely incidental bookkeeping; the KSK's signature is the one that matters for chain-of-trust purposes.

## 7.2 How to Test Authoritative Zones

Same "verify, don't assume" methodology as Chapter 3, applied to the signing side.

### 7.2.1–7.2.2 Checking for Keys and Signatures

**Querying your own authoritative server directly, the same way an external resolver would, is the most trustworthy way to confirm what you're actually publishing** — don't just trust that your config *should* be producing signed output; verify the wire output directly.

### 7.2.2's serial number footnote

> "Named keeps track of the serial number of the signed version of the zone independently of the unsigned version."

With `inline-signing`, you effectively have **two zone serial numbers in play** — your hand-edited unsigned zone's serial, and the automatically-managed signed zone's serial (which BIND bumps independently whenever it re-signs, e.g., due to signature expiration, even if you haven't touched the underlying zone data). Don't be alarmed if the signed zone's serial has incremented more than expected.

### 7.2.3 Examining the Files

Your source-of-truth zone file is genuinely unmodified; BIND maintains a separate `.signed` copy. Your version control / editing workflow for zone data doesn't need to change at all.

### 7.2.4 Checking the Parent

```
$ dig example.com. DS
```

This is the single most important post-deployment check, because it's confirming the *one* piece of this whole process that's outside your direct control — that the parent zone actually published what you asked it to.

### 7.2.5 External Tools

Verisign's DNSSEC Debugger and DNSViz check the **full chain**, not just your own zone in isolation — they'll walk from root down through every delegation point exactly like a real validating resolver would.

## 7.3 Signing Easy Start, Explained

### 7.3.1 Keys, Explained

File-naming convention: `Kexample.com.+008+06817.key` breaks down as `K<zone name>+<algorithm number, zero-padded>+<key tag>`. `008` = algorithm 8 (RSASHA256); `06817` is this specific key's key tag (the same key tag embedded in RRSIG records — exactly how a resolver knows *which* DNSKEY was used to produce a given signature).

**Distinguishing ZSK from KSK by the DNSKEY flags field:** `256` = ZSK, `257` = KSK. There's no separate "record type" for KSK vs ZSK; they're both DNSKEY records, distinguished by this flags value.

**Table 4.1 (Key Usage/Frequency) summary:**
- ZSK private key: used often (signs routine zone changes) → should be easier to rotate.
- ZSK public key: used *very* often (every validation touching zone data) → but its trust derives from the KSK.
- KSK private key: used *rarely* (only when keys themselves change) → can afford more ceremony/protection.
- KSK public key: used very often (every DNSKEY RRset validation) → but changing it is expensive, requiring a parent DS update.

This frequency/cost asymmetry is exactly why the two-tier design exists — it isolates the expensive, parent-coordinated operation (KSK rollover) from the routine, frequent operation (ZSK rollover).

### 7.3.2 BIND Config Options, Explained

**`dnssec-enable`** — the master on/off switch; without it, `named` can't serve signed zones at all.

**`key-directory`** — pure bookkeeping, tells `named` where to find the key files.

**`inline-signing`** — automates the *signing operation itself*. Without it, you'd need to manually re-run signing tools every time zone data changes or signatures approach expiration. With it, BIND transparently maintains an internally-signed copy.

**`auto-dnssec`** — automates *key lifecycle management*, a distinct concern from signing itself. Three levels:
- `off` — fully manual key management
- `allow` — keys can be updated, but only when you explicitly run `rndc sign [zonename]`
- `maintain` — BIND actively watches key timing metadata and automatically adds/removes DNSKEY records on schedule

**The distinction:** `inline-signing` automates turning zone *data* into signed output; `auto-dnssec` automates *key* lifecycle transitions. You generally want both together for a low-maintenance setup.

## 7.4 Working with the Parent Zone

**Whichever of the three formats (DS / DNSKEY / Trusted Key) your registrar wants, the underlying cryptographic content is equivalent** — they're just different serializations of "here's the KSK (or its hash) you should trust for this child zone."

```
# dnssec-dsfromkey -a SHA-1 Kexample.com.+008+06817.key
example.com. IN DS 6817 8 1 59194A835ACD78D25D538D5F35CA043A8F3F4446
```

Reading this DS record left to right: key tag (`6817`), algorithm (`8` = RSASHA256), digest type (`1` = SHA-1, `2` = SHA-256 in the second example), and the digest itself. **Currency note:** SHA-1 as a DS digest type is now considered weak — prefer SHA-256 (digest type 2) or better.

## 7.5 Using NSEC3

By default BIND signs with NSEC. NSEC3 is an alternative with different privacy properties (NSEC3 uses hashed names specifically to resist zone enumeration, whereas plain NSEC's sorted plaintext name chain makes it possible to "walk" a zone and discover every name in it just by querying repeatedly). NSEC is the default; switching to NSEC3 is a deliberate additional step.

## 7.6 Maintenance Tasks — Key Rollovers

> "Just like passwords and underwear, keys should be changed periodically."

> "we recommend changing your ZSK pair annually, and your KSK pair every five years"

This asymmetric cadence directly follows from Table 4.1: ZSK rotation is cheap (self-contained, no parent coordination) so it can happen often; KSK rotation is expensive (requires parent coordination via a new DS upload) so it happens rarely. **Currency note:** these specific cadences are the guide's own suggested defaults for a low-stakes zone, not a universal standard.

### 7.6.1 ZSK Rollover Timeline

Three-phase, month-spaced timeline:

1. **Month before (Dec 1):** generate the new ZSK and **publish** it (add its DNSKEY record) *before* using it to sign anything. Gives validating resolvers time to pick up and cache the new key.
2. **Rollover day (Jan 1):** start using the new ZSK to actually sign zone data.
3. **Month after (Feb 1):** only *now* remove the old DNSKEY from the zone.

**The reasoning:** DNS records have TTLs, and resolvers around the internet may have cached the *old* DNSKEY or *old* RRSIGs. Removing the old key too soon — before caches everywhere have naturally expired — would cause validation failures. This is the general pattern for *any* safe DNSSEC key rollover: publish new material well ahead of using it, and don't retire old material until you're confident nothing in the wild still depends on it.

> "with inline-signing and auto-dnssec, most of these are automated. The only things that need to be done manually are... Change timer on current ZSK, Generate new ZSK."

### 7.6.2 KSK Rollover Timeline

Structurally identical three-phase pattern, but with extra parent-coordination baked in (generate new KSK **and DS record**, then **upload to parent** in phase 1; **remove old DS from parent** in phase 3).

> "as of this writing, KSK rollover involves a lot of manual steps... the only automated tasks are the ones that occur on the day of the rollover... everything else needs to be done manually."

**Unlike ZSK rollover, KSK rollover cannot be fully automated by your own DNS software alone**, because it inherently depends on a manual (or semi-automated via registrar APIs) interaction with a third party. **Currency note:** some registrars now offer more automated DS-record-update mechanisms (and mechanisms like CDS/CDNSKEY records, which allow a child zone to signal its new key to the parent semi-automatically) than existed when this guide was written.

---

## Mental Model to Carry Forward

1. **Two keys, two purposes:** ZSK signs zone data (frequent, cheap to rotate), KSK signs the DNSKEY RRset (rare, expensive to rotate because the parent must be updated).
2. **Signing your zone alone accomplishes nothing externally** until the parent publishes a matching DS record — before that, your zone is simply treated as unsigned (fails open), not broken (fails closed).
3. **`inline-signing`** automates turning unsigned data into signed output; **`auto-dnssec maintain`** automates key lifecycle transitions on schedule.
4. **Verify constantly, both at your own servers and at the parent** — `delv` with a manually-trusted key for your own zone before delegation, `dig ... DS` against the parent afterward, and full-chain tools like DNSViz for the end-to-end picture.
5. **Key rollovers follow a publish-early/retire-late pattern** to respect DNS caching — never swap a key instantly; overlap old and new to let caches naturally converge.
6. ZSK rollover: mostly automatable, self-contained. KSK rollover: same shape, but requires manual/semi-manual coordination with your parent zone every time.

# 8. End-to-End DNSSEC Implementation Guide:

## example.com with Primary + Secondary Name Servers

This pulls together everything from Chapters 1–4 into one practical runbook. We'll use a realistic two-server topology:

- **ns1.example.com** (192.168.1.13) — **primary/master** — this is where we sign the zone
- **ns2.example.com** (192.168.1.14) — **secondary/slave** — receives the signed zone via zone transfer

## Phase 0: Pre-Flight Checklist

Before touching any config, confirm your environment is ready (Chapter 2):

1. **BIND version** — `named -v` on both servers, confirm a currently-supported version with DNSSEC support.
2. **Crypto support** — `named -V` on both, confirm built with a crypto library (OpenSSL or equivalent).
3. **Entropy** — especially if `ns1` is a VM, confirm you have adequate entropy for key generation (install `haveged` if needed).
4. **Network** — confirm TCP/53 is open between your servers and the outside world, and that firewalls aren't dropping large UDP DNS packets (Section 2.3/3.5).
5. **Parent zone support** — confirm your registrar for `example.com` supports DNSSEC and know what format they want for the DS record submission (Section 2.4.1).
6. **Confirm existing zone transfer (AXFR/IXFR) between ns1 → ns2 already works correctly**, since signed zone data will ride over the exact same mechanism. If replication is flaky *before* DNSSEC, it'll be flakier *after* — fix that first.

## Phase 1: Generate Keys (on the Primary Only)

Keys are only ever generated and used for signing on the **primary**. The secondary never touches private keys — it just receives already-signed zone data via transfer.

```bash
# On ns1 (primary)
mkdir -p /etc/bind/keys/example.com
cd /etc/bind/keys/example.com

# Zone Signing Key (ZSK)
dnssec-keygen -a RSASHA256 -b 1024 example.com

# Key Signing Key (KSK) — larger key, less frequently used
dnssec-keygen -a RSASHA256 -b 2048 -f KSK example.com
```

> **Note on algorithm choice:** RSASHA256/1024/2048 is the guide's legacy example. If starting fresh today, seriously consider **algorithm 13 (ECDSAP256SHA256)** — smaller keys and signatures, same security strength, and it directly helps avoid packet-fragmentation issues from Section 3.5.

This produces four files:
```
Kexample.com.+008+NNNNN.key       (ZSK public)
Kexample.com.+008+NNNNN.private   (ZSK private)
Kexample.com.+008+MMMMM.key       (KSK public)
Kexample.com.+008+MMMMM.private   (KSK private)
```

Lock down permissions:
```bash
chown -R bind:bind /etc/bind/keys/example.com
chmod 640 /etc/bind/keys/example.com/*.private
```

**Record the KSK's key tag now** — you'll need it repeatedly for the parent-zone upload in Phase 5.

## Phase 2: Configure the Primary (ns1) for Signing

Edit `named.conf` on **ns1 only**:

```
options {
    directory "/etc/bind";
    recursion no;
    minimal-responses yes;
    dnssec-enable yes;
};

zone "example.com" IN {
    type master;
    file "db/example.com.db";
    key-directory "keys/example.com";
    inline-signing yes;
    auto-dnssec maintain;

    // your existing transfer/notify config to ns2 stays as-is:
    also-notify { 192.168.1.14; };
    allow-transfer { 192.168.1.14; };
};
```

Nothing about your `example.com.db` zone file itself needs to change. Reload:

```bash
rndc reload
```

At this point, `named` will detect the keys in `key-directory`, sign the zone internally, and generate `example.com.db.signed` alongside your untouched original file.

## Phase 3: Configure the Secondary (ns2) — Usually No Changes Needed

**The secondary typically requires zero DNSSEC-specific configuration.** It's just receiving zone data via normal AXFR/IXFR, and that data now happens to include RRSIG/DNSKEY/NSEC records — the secondary doesn't need to know or care that DNSSEC is involved.

Your existing secondary config stays the same:

```
zone "example.com" IN {
    type slave;
    masters { 192.168.1.13; };
    file "db/example.com.db.slave";
};
```

**One thing worth double-checking:** if `ns2` runs an old BIND version with no DNSSEC awareness at all, it can still faithfully transfer and serve signed zone data — it treats RRSIG/DNSKEY/NSEC as just more resource records it doesn't need to understand to relay.

Confirm the transfer happened:
```bash
dig @192.168.1.14 example.com. DNSKEY +short
```
You should see the same two DNSKEY records that `ns1` is serving.

## Phase 4: Verify Signing Locally (Before Touching the Parent)

Simulate what a real validating resolver will eventually do, using a manually-trusted copy of your own KSK.

```bash
# Extract the KSK public key and format it for delv
cp /etc/bind/keys/example.com/Kexample.com.+008+MMMMM.key /tmp/example.key
```

Edit `/tmp/example.key` into `trusted-keys` format:
```
trusted-keys {
    example.com. 257 3 8 "AwEAAcWDps...lM3NRn/G/R";
};
```

Now validate against **both** servers independently — this confirms the secondary is serving a fully consistent, correctly-signed copy:

```bash
delv @192.168.1.13 -a /tmp/example.key +root=example.com example.com. SOA +multiline
delv @192.168.1.14 -a /tmp/example.key +root=example.com example.com. SOA +multiline
```

Both should report `; fully validated`. If `ns2` doesn't validate but `ns1` does, that's a strong signal of a **replication problem**, not a signing problem.

Also confirm signatures exist for ordinary records on both servers:
```bash
dig @192.168.1.13 www.example.com. A +dnssec +multiline
dig @192.168.1.14 www.example.com. A +dnssec +multiline
```

## Phase 5: Generate the DS Record and Upload to the Parent

Only now — after confirming both servers are correctly signed and in sync — do you touch the parent zone.

```bash
cd /etc/bind/keys/example.com
dnssec-dsfromkey -a SHA-256 Kexample.com.+008+MMMMM.key
```
```
example.com. IN DS 6817 8 2 A5F1DF55D5E64CBD7BCFE1EFA6E9586AF335FA56A2473296E975B89AFD31E11
```

Take this DS record (or whichever format your registrar requires) and submit it through your registrar's DNSSEC/DS-record submission interface.

**This is the moment your zone transitions from "signed but not part of the chain of trust" to "actually verifiable by the outside world."** Before this step, external validating resolvers fall back to insecure lookups for `example.com` (fail-open) — after this step, they'll fully validate it (and will SERVFAIL if anything is wrong).

## Phase 6: Confirm the Parent Published Correctly

Give the parent zone time to propagate, then check from outside your own infrastructure:

```bash
dig example.com. DS
```

Confirm the returned DS record's key tag, algorithm, and digest match exactly what you generated in Phase 5.

## Phase 7: End-to-End External Validation

Test as a genuine outside validating resolver would.

**Command-line check**, from a machine using a validating resolver:
```bash
dig @<some-validating-resolver> www.example.com. A +dnssec +multiline
```
Look for the `ad` flag — confirmation the full chain (root → your TLD → `example.com` → the A record) validated successfully with no manual trust anchor involved.

**Full-chain visual check** — run `example.com` through DNSViz and/or Verisign's DNSSEC Debugger to see the entire delegation path validated graphically, and to catch subtle issues.

**Negative test** — confirm a genuinely nonexistent name under your zone still resolves correctly to NXDOMAIN with authenticated denial:
```bash
dig @<validating-resolver> nonexistent.example.com. A +dnssec
```

## Phase 8: Establish Ongoing Maintenance

Deployment isn't really "done" at Phase 7:

1. **Confirm automatic re-signing is working** — spot-check periodically that signature expiration dates in your zone stay comfortably in the future (`dig example.com. SOA +dnssec` and inspect RRSIG timestamps).
2. **Set a calendar reminder for ZSK rollover** (commonly annual) — mostly automated by `auto-dnssec maintain`, but you still need to manually generate the new ZSK and set its timing metadata ahead of the rollover date.
3. **Set a calendar reminder for KSK rollover** (commonly every ~5 years) — requires manual coordination with your registrar for the DS record update, so build in extra lead time.
4. **Monitor both servers, not just the primary** — a replication failure on `ns2` after a key rollover could leave it serving stale signed data while the parent/primary have moved on, causing intermittent validation failures depending on which server a resolver happens to query.
5. **Keep an eye on parent-zone/registrar changes** — if you ever change registrars or the parent zone's DNSSEC support changes, revisit Phase 5/6.

## Quick Reference: The Full Sequence

```
1. Pre-flight checks (BIND version, entropy, network, registrar support)
2. Generate ZSK + KSK on primary (ns1) only
3. Configure named.conf on ns1 (key-directory, inline-signing, auto-dnssec)
4. Confirm ns2 replicates the signed zone with no config changes needed
5. Validate locally with delv, using a manually-trusted KSK — on BOTH servers
6. Generate DS record, submit to registrar/parent zone
7. Confirm parent published matching DS record (dig example.com. DS)
8. Confirm full external validation (ad flag, DNSViz/Verisign debugger)
9. Test negative case (nonexistent name still resolves correctly)
10. Establish ongoing ZSK/KSK rollover schedule and monitoring on both servers
```

This sequence deliberately front-loads all verification onto your own infrastructure (Phases 1–4) before you ever involve the parent zone (Phase 5+) — because once the DS record is published, mistakes become externally visible as validation failures for real users, whereas mistakes caught in Phases 1–4 are invisible to the outside world and cost you nothing to fix.
