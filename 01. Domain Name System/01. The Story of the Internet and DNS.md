# The Story of the Internet and DNS

## 1. How It All Started (late 1960s–1970s)

A long time ago, a U.S. government group called DARPA paid for a new network called the **ARPANET**. It connected a few research computers together. At first, people just wanted to share expensive computers. But very quickly, people started using it to send messages, share files, and work together — kind of like an early version of email and file sharing.

Two things about this early network matter for our story:

- It was small — only a few hundred computers for many years.
- There was one single list, called **HOSTS.TXT**, that had the name and address of every computer on the network. One organization (SRI's Network Information Center) kept this list. People emailed in their changes, and the list was updated and shared about once or twice a week. (Fun fact: your computer's `/etc/hosts` file today comes from this old list.)

## 2. The Network Explodes (early 1980s)

In the early 1980s, a new set of network rules called **TCP/IP** became standard. This really took off because it got built into a free, popular operating system called BSD Unix, which universities used a lot. Since it was free and easy, way more schools and organizations connected to the network. The ARPANET grew from a handful of computers to thousands.

This growth broke the old HOSTS.TXT system in three ways:

- **Too much traffic** — every time a computer was added, the file got bigger, and more computers had to download the new version. The system that shared this one file couldn't keep up.
- **Name clashes** — two computers could accidentally get the same name, since there was no strong rule stopping it. If someone gave a computer the same name as an important mail server, it could break things for a lot of people.
- **Outdated info** — by the time the new list reached everyone, something had already changed. The list was always a little bit behind reality.

The basic lesson here: one single file, updated by one central place, just can't keep up when lots of different people need to make changes. This is exactly the problem DNS was built to fix.

## 3. What the New System Needed to Do

The people in charge of the ARPANET asked for a replacement with these goals:

1. Let each organization manage its own piece of information, but still let everyone else see it.
2. Get rid of the single bottleneck — no one computer should have to handle everything.
3. Organize names in a tree-like structure, so names couldn't clash with each other.

## 4. The Birth of DNS (1983–1987)

A person named **Paul Mockapetris**, working at USC, designed this new system. He first described it in 1984 in two documents (RFC 882 and 883). These were later replaced by two documents that are still the foundation of DNS today: **RFC 1034** and **RFC 1035** (from 1987). Everything added to DNS since then — like security features — builds on top of these two documents.

The first version of the software was called **JEEVES**, made by Mockapetris himself. Later, someone named Kevin Dunlap wrote a new version called **BIND** (Berkeley Internet Name Domain) for a Unix system at Berkeley. BIND became — and still is — the most popular DNS software in the world.

## 5. How DNS Is Built

Think of DNS like an upside-down tree, or like the folders on your computer:

| DNS word | Same idea in a computer's folders |
|---|---|
| Root (written as `.`) | The main drive, like `/` |
| A label (like "com") | A folder name |
| A domain | A folder (and everything inside it) |
| A subdomain | A folder inside another folder |
| A domain name (e.g. `www.example.com`) | A full folder path |

DNS has two main jobs, done by two kinds of programs:

- **Nameservers** — computers that hold the information for one part of the tree and answer questions about it.
- **Resolvers** — the part (usually built into your computer or apps) that asks the nameservers questions and gets answers back.

The clever trick that makes this whole system work is **delegation**. One organization can hand off responsibility for part of the name tree to a different organization. For example, the group that runs `.edu` can hand off `berkeley.edu` to UC Berkeley to manage on their own. This handed-off piece is called a **zone** — basically, "the part of the tree that this specific organization is in charge of."

A "domain" and a "zone" aren't exactly the same thing: a domain includes everything below it in the tree, but a zone is only the part that hasn't been handed off to someone else. So if Berkeley later hands off `cs.berkeley.edu` to the computer science department, then `cs.berkeley.edu` is no longer part of Berkeley's main zone — it's its own zone now.

This handing-off system is also what stops names from clashing. Since each organization only needs to make sure names are unique *inside their own little piece*, and every piece ends in a unique parent name, the whole system stays clash-free without anyone needing to check the entire world at once.

## 6. Nicknames for Computers

A computer can also have a nickname, called an **alias**, which just points to its real, official name (called the **canonical name**). The nickname doesn't hold any real information itself — it just points to where the real information lives.

## 7. Why Any of This Matters Today

- The idea of one organization trusting another with a piece of the tree (delegation) is the same idea behind DNS security features used today — trust flows down the tree, step by step, just like the tree itself.
- Every time you add a record to hand off a subdomain to someone else, you're doing the exact same thing Berkeley did with `.edu` back in the day — carving out a new zone.
- The old problem of "the list is always a little out of date" is why DNS today uses caching and expiration times (TTLs) — it's built to handle the fact that information changes and needs to spread out gradually, instead of relying on one perfect, always-up-to-date master list.
