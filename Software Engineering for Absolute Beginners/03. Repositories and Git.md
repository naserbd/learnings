# Repositories and Git

Docker alone isn't enough to build great software. You still need a reliable way to **distribute** your code, and to **track changes** to it over time. That's what this chapter is about.

## How Things Used to Work

It's worth understanding the problem source control actually solves, because it wasn't always around.

In less technically mature workplaces, developers used to send code to a server using **FTP** (a simple file-transfer method) — no history, no tracking, nothing automatic. If you wanted a backup, you made one yourself, manually.

This created real problems the moment more than one person touched the same code:

- Two developers couldn't safely work on the same file independently. Whoever uploaded second would overwrite whoever uploaded first.
- On larger systems, two people's work would eventually "cross paths" — editing files that depended on each other — with no safe way to combine the results.

In fact, in the author's first software job, *"Are you working on file so-and-so?"* was a constant, necessary question — because there was no other way to know.

Source control existed back then too — it just wasn't widely used. Today, it's the opposite: it would be unusual for *any* professional software team not to use it. Modern source control platforms go even further, offering automatic deployment of your code — a feature we'll return to later in this book.

## A Quick Note for Windows Users: Hidden Files

As you work through this chapter, you'll come across **hidden files** — files whose names start with a period (for example, `.gitignore`). Standard file browsers hide these by default, so you won't see them unless you know to look.

To reveal them, you'll need either:
- A file browser or editor configured to show hidden files, or
- A command-line command to list them directly

If you're working inside a Linux-based terminal (for example, a Docker quickstart terminal), this command will show you every file, hidden or not:

```
ls -ltra
```

## What Is Source Control?

**Source control** is the practice of managing and tracking every change made to your codebase. In this book, we'll use **Git** — currently the most widely used source control system in the industry.

Two ideas sit at the core of it:

1. **Managing changes** — allowing multiple developers to safely work on the same codebase at the same time.
2. **Tracking changes** — keeping a full history of what changed, when, and who changed it.

### Local vs. Remote

Git works with two locations for your code:

- A **remote** location — a shared, central copy of the code, typically hosted online (for example, on GitHub or GitLab).
- A **local** location — the copy of the code living on your own computer, where you actually write and test your changes.

Interestingly, Git itself doesn't technically treat these as different kinds of things — both are just **repositories** (or "repos" for short). One happens to be shared and central; the other happens to be on your machine.

### Branches

Inside a repository, you'll find one or more **branches**. A branch is simply a snapshot of your code at a particular point of work — think of it as your own working copy, isolated from everyone else's.

Only one branch is *active* on your local machine at a time, but you can push any branch — active or not — up to the remote repository.

**A typical setup looks like this:**

- There's usually one "main" branch — often called `master` or `main` — which represents the code that actually gets deployed to your live server.
- From that main branch, developers create new branches to build individual features.
- Once a feature is finished and reviewed, its branch gets merged back into the main branch.

**Example:** Say you're adding credit card payments to the system. You create a new branch — maybe called `add_cc_payments` — and do all your work there. The `master` branch stays completely untouched until you're ready to merge your finished feature into it. Meanwhile, a teammate can be working on an entirely separate branch, for an entirely separate feature, without any conflict.

### Why This Branching Model Is Worth It

Keeping feature work in separate branches gives you three real benefits:

1. **`master` stays stable**, so anyone on the team can safely branch off it at any time to start new work.
2. **`master` stays deployable.** You can push urgent bug fixes through it without worrying that someone else's half-finished feature will accidentally go live too.
3. **You can share work for review safely.** Push your feature branch to the remote repo, and a colleague can pull it down to review — without touching or risking their own in-progress work.

### Merging — and Merge Conflicts

When it's time to combine two people's work into one file, Git handles most of this automatically. It's able to do this intelligently because it tracks the history of *every individual line* of code, including when each change was made.

Sometimes, though, two people edit the *exact same line* of code. When that happens, Git can't safely guess which version should win — this is called a **merge conflict**, and it has to be resolved by a person, manually. (In rare, careless situations, an automatic merge can silently overwrite someone's work without flagging it as a conflict at all — one more reason to merge carefully.) We'll cover how to resolve merge conflicts later in this chapter.

### A Note on `git blame`

Git includes a feature — unfortunately named `blame` — that shows you exactly who last changed any given line of code, and when.

Despite the name, this feature isn't for assigning fault. Its real purpose is simple: when you're confused about *why* a piece of code exists, it tells you *who* to ask. Publicly calling someone out for code you find in a `blame` search is bad practice and damages team trust. Keep this tool aimed at curiosity, not blame — good teams practice **egoless programming**, where the code is reviewed on its own merits, not tied to anyone's ego.

## What Else Source Control Gives You

Beyond tracking changes, a modern source control platform typically offers:

- **Automated testing** — running your unit tests and style checks automatically whenever code is pushed to the remote repository.
- **Automated deployment** — pushing tested code straight to staging or production without manual steps. This combination is known as **continuous integration and deployment (CI/CD)**, which we'll cover in Chapter 11.
- **Pull requests** — a formal way to request that a teammate review your code before it's merged.

> **Note:** Testing and code review are never about proving code is "bug-free" — that's not something you can prove. They're about confirming the code does what it's *meant* to do, and hopefully catching real bugs along the way.

### Pull Requests and Peer Review

Good software teams build **peer review** into their process: before code gets merged, another developer reviews it for accuracy and quality.

A **pull request** is how this review gets organized. It assigns your finished feature branch to a specific reviewer, viewable through whatever platform hosts your Git repository (GitHub, GitLab, Bitbucket, etc.). From there, the reviewer can:

- Leave comments directly inline, next to the exact lines of code in question
- Approve the merge from the platform itself, rather than requiring it to happen on your local machine

We'll walk through this whole process — branching, pushing, pull requests, and merging — later in this chapter. Continuous deployment specifically is saved for Chapter 11.

## Quick Glossary

A few terms worth having in one place before you start using Git hands-on:

| Term | Meaning |
|---|---|
| **Repository (repo)** | A project's full set of code plus its complete change history |
| **Clone** | Making a local copy of a remote repository |
| **Commit** | Saving a snapshot of your changes, with a message describing what changed |
| **Push** | Sending your local commits up to the remote repository |
| **Pull** | Downloading the latest changes from the remote repository to your local copy |
| **Branch** | An independent line of development, isolated from other branches |
| **Merge** | Combining the changes from one branch into another |
| **Merge conflict** | What happens when Git can't automatically decide how to combine two conflicting changes to the same line |
| **Pull request** | A formal request to review and merge one branch into another |
