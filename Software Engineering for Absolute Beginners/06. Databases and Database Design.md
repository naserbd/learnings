# Databases and Database Design

This chapter covers more than just creating databases, tables, and queries — it also walks through setting up a database inside your Docker environment, which is a practical skill every developer should have.

## Data: The Blood of Your Application

Data gives your software purpose. It shapes what state your application is in, lets you log in and out of systems, and provides the information people use to make business decisions. Every credit card transaction, every "like" on a social media post — all of it involves reading from and writing to a database.

You'll hear a lot of buzz around "big data," data visualization, and technologies promising faster reads and writes. All of that is genuinely exciting — but before any of it is useful, you need a solid grip on the fundamentals: how to design a database properly and query it with confidence.

Database design deserves real care. Get it wrong, and you may find yourself **refactoring a database** six months down the line — restructuring how data is stored *and* rewriting the backend code that depends on it. That's messy, time-consuming work. (The alternative — leaving a flawed design as-is and just working around it — is sometimes the path of least resistance, but it usually costs you more in the long run.)

This chapter will get you comfortable with:

- Designing a normalized database
- Writing SQL queries against it
- Setting it all up inside Docker

Later in the book, you'll build on this by having a Python application read from and write to a database directly.

## Three Things You Can Do With Data

Broadly speaking, there are three ways to handle data once your system receives it:

1. **Discard it immediately after use.**
   Some data should never be stored at all. A password, for example, arrives at your system as plain text — it should be used to verify identity and then discarded, never saved anywhere in that raw form.

2. **Store it short-term.**
   Some data is only useful briefly. Shopping baskets in some e-commerce systems used to disappear after a few weeks (though many modern systems now remember them much longer). **Cached data** falls into this category too — data stored temporarily because it's faster to retrieve than pulling it fresh from long-term storage each time.

3. **Store it long-term.**
   This is the main focus of this chapter: usernames, addresses, purchase histories — anything with lasting value to your system or business.

A good rule of thumb: **only store what you'll actually need.** Storage is cheap these days, but every piece of data you keep is still something you're responsible for protecting and maintaining. If you won't use it, don't keep it.

## Overview of Database System Components

A **DBMS** (Database Management System) is the application that manages your databases — including users, permissions, logs, and metadata. It also handles more advanced concerns like replication and sharding (topics beyond the scope of this book).

Picking a DBMS isn't usually a difficult decision — there are many excellent, free options available, and you'd need fairly specialized requirements to rule most of them out.

This book uses a **relational database**. Relational databases are popular and powerful, storing data in a tabular, row-by-row format, and manipulated using a query language called **SQL**.

Here's how the pieces fit together:

- One **DBMS** can manage **multiple databases**.
- One **database** can contain **multiple tables**.
- A **table** stores data in rows and columns, and can be linked to other tables via **primary and foreign keys** (explained shortly).
- Each **column** in a table has a defined **data type** — integer, text, boolean, and so on.
- A single **row** spanning all the columns is called a **record**.
- Tables can also have **indexes** — hidden helper structures that store data in sorted order to speed up lookups (also explained shortly).

**A simple mental picture:** imagine one DBMS managing two databases. Database 1 has a `users` table with columns `name`, `surname`, and `age`. Database 2 has a `shopping_carts` table with columns `price`, `item`, and `quantity`. Each table's first row lists its column names, and every row below that holds one actual record.

## Setting Up Your DBMS

We'll use **MySQL** as our DBMS — a solid, well-supported choice for our purposes. (It's always worth comparing options as your needs grow, but MySQL will serve us well here.)

To add it to your project, edit your `docker-compose.yml` file. Under the `services` section — alongside the Python service you defined in Chapter 2 — add the following:

```yaml
mysql-dev:
  image: mysql:8.0
  container_name: mysql-dev-container
  ports:
    - 6603:3306
  environment:
    MYSQL_ROOT_PASSWORD: "root"
  volumes:
    - /storage/docker/mysql-datadirectory:/var/lib/mysql
```

Most of this should look familiar from Chapter 2. Here's what the three new-ish entries mean:

**`ports`** — Formatted as `A:B`. Port `A` is how you'll reach the database from *outside* Docker (e.g. from a tool on your own computer). Port `B` is the port used for communication *between* Docker containers. Here, `6603` is external, `3306` is internal.

**`environment`** — Sets the database's root password. (`root` is used here for simplicity — never use such a weak password in a real production system.)

**`volumes`** — Maps a folder on your local computer to a folder inside the container, formatted as `local-path:container-path`. This means your database's actual data lives on your computer, at `/storage/docker/mysql-datadirectory`, even though MySQL (inside the container) thinks it's writing to `/var/lib/mysql`. This mapping is what keeps your data safe even if you shut the container down completely.

### The Complete Docker Compose File

Putting it together, your full file should now look like this:

```yaml
version: "3"
services:
  python-dev:
    container_name: python-dev-container
    build:
      context: .
      dockerfile: ./Dockerfile
    volumes:
      - ./test:/home

  mysql-dev:
    image: mysql:8.0
    container_name: mysql-dev-container
    ports:
      - 6603:3306
    environment:
      MYSQL_ROOT_PASSWORD: "root"
    volumes:
      - /storage/docker/mysql-datadirectory:/var/lib/mysql
```

Now rebuild your environment:

```
docker-compose build --no-cache
```

Then start it up. The `-d` flag runs your containers in the background ("detached"), freeing up your terminal to keep typing commands:

```
docker-compose up -d
```

Run `docker-compose ps` to confirm your database container is running. From here on, you can run your Python code like this:

```
docker-compose run python-dev
```

## Preparing Your Database

Before writing any real code against your database, there's some housekeeping worth doing first.

Right now, you have one user: **root** — the all-powerful account with every possible permission. **Never use the root account for your application's day-to-day database access.** Instead, once you start writing Python code that talks to the database, create a dedicated, limited-permission user for it.

**Why bother with a second user?** Your application's database username and password will end up stored somewhere in your codebase. If that credential leaks, you want the damage capped. A limited-permission user can be revoked or changed quickly, and — because its permissions are restricted — there's only so much harm it can do even if compromised. (That said: even a "read-only" leaked credential can expose sensitive data, so the password should always be treated as a secret, no matter how limited the account's permissions are.)

When creating an application-facing username, aim for these properties:

- **Scope it to one database.** If someone steals this username, you don't want them able to reach every database on the server.
- **Limit its permissions precisely** to what your application actually needs — nothing more. If your app never deletes data, don't grant delete permissions. If it never drops tables, don't grant that either. **Never** give an application user permission to *change permissions* — that would let a compromised account escalate its own access.
- **Restrict it by IP address**, if possible — only allowing connections from a specific server, region, or network. (This adds a layer of protection, though it isn't foolproof on its own.)

## Primary Keys

Every table should have a **primary key** — a value that uniquely identifies each row, appearing exactly once per table. Most DBMSs offer an **auto-increment** integer key, which generates this value automatically as new rows are added.

This is convenient, but worth a security thought: if primary keys are simple, sequential, and *guessable* (1, 2, 3, 4...), and your application doesn't otherwise protect against it, someone could potentially guess at valid IDs to target specific rows directly. Primary keys are also automatically indexed, meaning they're significantly faster to search on than non-indexed columns.

## Indexes

You can create an **index** on any column you choose — it doesn't have to be unique (though it *can* be). Indexing builds a hidden secondary structure that stores that column's values in sorted order, alongside a pointer back to the original row. Because the data is pre-sorted, lookups on that column become dramatically faster.

### The catch with indexes

Indexes aren't free:

- They take up additional storage, and grow as your table grows.
- Every time an indexed field is **updated**, the DBMS also has to update the index — which adds overhead.

**Example of where this bites:** Imagine a table tracking vehicle locations, where `last_location` updates every 10 seconds. With 1,000 vehicles, that's a constant stream of index updates. Scale that to 10,000 vehicles, and the reindexing overhead alone could start causing real performance problems.

**The takeaway:** only index columns you'll actually search on — not every column in the table.

## Data Types

Every column needs a specific **data type**. Here are some of the most common ones:

| Type | Description |
|---|---|
| `text` | Free-form text; has smaller/larger variants like `tinytext`, `mediumtext` |
| `varchar` | Text with a maximum length you define; only uses as much storage as the actual string needs |
| `int` | Whole numbers; has variants like `tinyint`, `mediumint` for smaller/larger ranges |
| `decimal` | Numbers with a fixed decimal point |
| `datetime` | Date-and-time data; related types include `date` and `time` alone |
| `boolean` | Stored internally as `tinyint` |

**Choose the tightest-fitting type for your data.** If a field will only ever hold numbers under 100, `tinyint` is enough — no need for a full `int`. If a field holds first names, `varchar(60)` (max 60 characters) is more appropriate than an unrestricted `text` field. Picking the wrong type usually won't break your system outright, but it can cause real inefficiency as your data grows. It's worth researching your specific DBMS's data types before finalizing a table design — different systems offer different options.

## Your First SQL Queries

**SQL** (Structured Query Language) is how you interact with your database — not just to fetch data, but to insert, update, delete, and even define new tables. This chapter covers the essentials, using these keywords (by convention, always written in UPPERCASE):

| Keyword | Purpose |
|---|---|
| `SHOW` | List available tables (handy from the command line) |
| `DESCRIBE` | List a table's columns and their types |
| `SELECT` | Retrieve data |
| `INSERT` | Add new data |
| `UPDATE` | Change existing data |
| `DELETE` | Remove data |
| `JOIN` | Combine data across multiple tables |
| `CREATE` | Create new tables |
| `FROM` | Specifies which table a query pulls data from |
| `WHERE` | Filters which rows a query affects |

In Adminer, click **SQL Command** on the left sidebar — this opens a text box where you can type and run queries directly.

### `SHOW` and `DESCRIBE`

```sql
SHOW TABLES
```

Lists every table in the current database — you should see `classes`.

```sql
DESCRIBE classes
```

Lists `classes`'s columns and their types — useful any time you don't have a visual tool like Adminer handy.

### `SELECT`

```sql
SELECT * FROM classes
```

The `*` means "every column." This returns every row, with every column, from `classes`.

```sql
SELECT firstname, mobile FROM classes
```

This returns only the `firstname` and `mobile` columns, for every row.

```sql
SELECT * FROM classes WHERE firstname = 'Rogers' AND city = 'London'
```

The `WHERE` clause filters rows — here, only rows where `firstname` is `'Rogers'` **and** `city` is `'London'` are returned. Conditions can be combined with `AND` and `OR`, and compared using `=`, `<`, `>`, `!=`, `IN`, `NOT IN`, `IS`, and `IS NOT`.

```sql
SELECT * FROM classes WHERE city IN ('London', 'Cape Town', 'Beijing')
```

Returns rows where `city` matches *any* value in the list.

```sql
SELECT * FROM classes WHERE city NOT IN ('London', 'Cape Town', 'Beijing')
```

The reverse — rows where `city` matches *none* of the listed values.

```sql
SELECT * FROM classes WHERE name IS NOT null
```

Checks for the *absence* of data. `NULL` means "no value was ever recorded" — it's different from an empty string (`''`), which is a value that just happens to be empty. `IS NOT null` finds rows where the field genuinely has *something* stored in it.

### `INSERT`

Inserts don't use a `WHERE` clause — there's nothing to filter, since you're adding new data, not selecting existing data.

**A single insert:**

```sql
INSERT INTO classes (salutation, firstname, mobile, city, languages)
VALUES ('Mrs', 'Richards', '0754896325', 'London', 'German, Dutch')
```

**Multiple inserts in one statement** (much faster than running several single inserts back-to-back — though most databases cap you at around 1,000 rows per statement):

```sql
INSERT INTO classes (salutation, firstname, mobile, city, languages)
VALUES
    ('Mr', 'Eagle', '07546458745', 'Bristol', 'Dutch'),
    ('Miss', 'Schofield', '0766984752', 'London', 'English')
```

**General shape of an insert:**

```sql
INSERT INTO table_name (column_list)
VALUES (values_for_one_row), (values_for_another_row), ...
```

### `UPDATE`

Use `UPDATE` to change existing rows. **This is where caution really matters** — an `UPDATE` without a `WHERE` clause changes *every single row* in the table.

```sql
-- DANGER: this updates every salutation in the entire table to 'Mr'
UPDATE classes SET salutation = 'Mr'
```

Almost always, you'll want to narrow this down:

```sql
UPDATE classes SET salutation = 'Mr' WHERE firstname = 'Richards'
```

Even here, be aware: this still updates *every* row where `firstname` equals `'Richards'` — if that name isn't unique in your table, multiple rows will change. Always think through exactly which rows your `WHERE` clause will match before running an update.

### `DELETE`

Deleting data deserves careful thought about what should actually be removed. A common, safer alternative is a **soft delete**: instead of physically removing a row, you add a column (commonly `deleted_at`) and set it to the current timestamp when a row should be treated as "deleted." Your application code then simply filters those rows out:

```sql
SELECT * FROM classes WHERE deleted_at IS NOT null
```

Sometimes, though, a full, permanent deletion is required — for example, honoring a legal request from a customer to erase their data entirely.

```sql
DELETE FROM classes WHERE firstname = 'Rogers'
```

**Just like `UPDATE`, running `DELETE` without a `WHERE` clause removes every row in the table.** Always double-check your `WHERE` clause before executing a delete.

## Normalizing the `classes` Table

Now let's fix the problems baked into our original design. The current table repeats a lot of data, which causes two real issues:

1. **Wasted space.** Repeated text (like the same city name across hundreds of rows) adds up.
2. **Fragile updates.** Suppose "London" was misspelled somewhere. Because the same value is duplicated across many rows, fixing it means updating *every single one* — trivial with 3 rows, a nightmare with millions.
3. **Awkward multi-value fields.** The `languages` column crams multiple values into one string (`'German, English'`). Removing just `English` from that string means parsing it apart, editing it, and reconstructing it — increasingly painful as the list of languages grows.

**Normalization** is the process of breaking one messy table into several smaller, well-structured ones. We'll walk through the first three stages, called **normal forms**, then look at how to query the results using SQL `JOIN`s.

### First Normal Form (1NF)

**Rule:** Every entry should be unique, and every cell should hold a single value — no cramming multiple values into one field.

Our current `classes` table violates this (the `languages` column holds multiple comma-separated values) — so it isn't in any normal form yet.

### Second Normal Form (2NF)

**Rule:** The table must already satisfy 1NF, have a single primary key, and every non-key column must be **fully dependent on that primary key**.

**"Functional dependency"** is the key idea here: a non-key column should exist *because of* the primary key — meaning it helps describe the specific thing that key identifies, not something else entirely.

Applying this: `salutation`, `firstname`, and `mobile` are all naturally about *identifying a specific client* — they belong together in their own table, which we'll call `clients`. By contrast, a client's `city` or the `languages` they're learning aren't really about identifying *the client themselves* — they describe separate concepts (a location, a course) that just happen to be associated with that client.

At this point, though, splitting things up has cost us something: we've lost the connections. We no longer know *which* city a client lives in, or *which* languages they're taking. The next step is rebuilding those connections properly.

### One-to-Many Relationships, Joins, and Foreign Keys

A client lives in exactly one city, but a city can have many clients — this is a **one-to-many relationship**.

To represent it, add a `city_id` column to the `clients` table, replacing the old repeated city name with a reference to that city's primary key in a separate `cities` table. This reference is called a **foreign key** — a column that uniquely identifies a row in *another* table.

**To build this out yourself:**

- Every table needs an auto-increment `id` column as its primary key.
- Use `varchar` for text fields (with sensible length guesses), and `int` for `city_id`.
- Build the tables in Adminer.
- Make sure the values you put in `city_id` actually correspond to real primary keys in your `cities` table.

By the end, you should have four tables: the original `classes`, plus new `clients`, `cities`, and `languages` tables — all populated with data. (We haven't yet connected `clients` to `languages` — that comes next, since it's a trickier kind of relationship.)

### Querying with `JOIN`

To pull related data back together across tables, use `JOIN`:

```sql
SELECT * FROM clients cl JOIN cities cit ON cl.city_id = cit.id
```

Two new ideas here:

- **`cl` and `cit`** are **aliases** — short nicknames for `clients` and `cities`. Aliases save typing for long table names, and are essential when a table needs to be joined to *itself* (since you'd otherwise have no way to distinguish the two references to the same table).
- **`JOIN ... ON`** tells MySQL exactly how the two tables relate: rows in `clients` are matched to rows in `cities` wherever `cl.city_id` equals `cit.id`.

Filtering a joined query works exactly like before:

```sql
SELECT * FROM clients cl JOIN cities cit ON cl.city_id = cit.id
WHERE cit.city = 'London'
```

### Many-to-Many Relationships

Some relationships are more complex. One client can speak many languages, and one language can be spoken by many clients — a **many-to-many relationship**.

You can't solve this the same way as before: adding multiple language values to a single `clients` row would break 1NF again, and adding multiple client IDs to a `languages` row has the exact same problem in reverse.

**The solution: a pivot table** (also called an intermediate or junction table) — a table that exists purely to record the *connections* between two other tables. We'll call it `client_languages`, with two integer columns:

| Column | References |
|---|---|
| `client_id` | A client's primary key, in `clients` |
| `language_id` | A language's primary key, in `languages` |

Each row in `client_languages` represents one client-speaks-one-language fact. A client with three languages simply has three rows here — one per language.

Create this table with those two `int` columns, and populate it so that at least some clients have more than one language.

Querying across all three tables:

```sql
SELECT * FROM clients c
JOIN client_languages cl ON c.id = cl.client_id
JOIN languages l ON l.id = cl.language_id
```

This chains two joins together: `clients` → `client_languages` → `languages`, following the relationships step by step.

To find clients who speak German specifically, add a `WHERE` clause:

```sql
SELECT *
FROM clients c
JOIN client_languages cl ON c.id = cl.client_id
JOIN languages l ON l.id = cl.language_id
WHERE l.language = 'German'
```

### Third Normal Form (3NF)

**Rule:** No non-primary field should be dependent on *another non-primary field* within the same table.

This one takes a bit more thought. Consider `salutation` in the `clients` table. Technically, a salutation (`Mr`, `Mrs`, `Miss`) is tied to gender, which — in this simplified example — is implicitly tied to the person's name. If a client's name changed from `Jane` to `John` for any reason, their `salutation` would logically need to change too. That's a dependency between two *non-primary* fields (`salutation` depending on `firstname`), which 3NF says shouldn't exist within the same table.

**The fix:** pull `salutation` out into its own table (call it `salutations`, with an `id` primary key and a `salutation` text column), and reference it from `clients` via a new `salutation_id` foreign key, replacing the old `salutation` text column entirely.

Querying across everything now looks like this:

```sql
SELECT *
FROM clients c
JOIN client_languages cl ON c.id = cl.client_id
JOIN languages l ON l.id = cl.language_id
JOIN salutations s ON s.id = c.salutation_id
```

## A Final Word on Joins

**`JOIN` only applies to `SELECT` statements** — you can't join tables in an `INSERT`, `UPDATE`, or `DELETE`.

There are also different *kinds* of joins. Two worth knowing about now: `LEFT JOIN` and `RIGHT JOIN`.

Consider this query again:

```sql
SELECT * FROM clients cl JOIN cities cit ON cl.city_id = cit.id
WHERE cit.city = 'London'
```

If a client's `city_id` doesn't match *any* row in `cities` (for example, it's missing or invalid), that client is silently **excluded** from the results entirely — a plain `JOIN` only returns matches found on *both* sides.

If you want every client to appear regardless — even ones with no matching city — use a **`LEFT JOIN`**: it keeps every row from the *left*-hand table (`clients`, in this case), filling in empty/`NULL` values wherever no match exists on the right. A **`RIGHT JOIN`** works the same way in reverse, preserving every row from the *right*-hand table instead.

## Cheat Sheet and Checklist

**Logical operators:**
`>`, `<`, `=`, `!=`, `IN`, `NOT IN`, `IS`, `IS NOT`

**Select data:**
```sql
SELECT [columns] FROM table_name alias
WHERE field [operator] value
```

**Multiple joins:**
```sql

```

**Insert data:**
```sql
INSERT INTO table_name (columns) VALUES (values)
```

**Update data:**
```sql
UPDATE table_name SET field = value
WHERE field [operator] value
```

**Delete data:**
```sql
DELETE FROM table_name
WHERE field [operator] value
```

**Normal forms, at a glance:**

| Form | Requirement |
|---|---|
| **1NF** | Every entry is unique; every field holds one value only |
| **2NF** | Every non-primary field is fully dependent on the primary key |
| **3NF** | No non-primary field is dependent on another non-primary field |

**Key concepts:**

- A **primary key** uniquely identifies a row within its own table.
- A **foreign key** identifies a relationship with another table — typically, a foreign key in Table A is a primary key in Table B.
- **Indexing** stores a column's data in a separate, ordered structure to speed up searches on that column.
