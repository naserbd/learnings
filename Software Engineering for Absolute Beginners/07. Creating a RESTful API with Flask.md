# Creating a RESTful API: Flask

In this chapter, you'll build a small, complete project using everything covered so far — Docker, GitLab, and database design — and add one new piece: a **software framework**.

## Why Use a Framework?

When building software, you don't need to write everything from scratch. A **framework** provides the foundational tools most systems need — routing incoming requests to the right code, basic database access patterns, error handling, and more — so you can focus on the part that actually matters: solving your business's specific problem, not reinventing plumbing that thousands of other developers have already built and tested.

A widely used, well-established framework also tends to be more reliable — more people using it means more people finding and fixing its bugs.

We'll use **Flask** — a *microframework*, meaning it's deliberately lightweight, giving you just the essentials without much extra baggage. That makes it a great fit for the small project ahead.

Before any code gets written, this chapter spends real time on setup — dependencies, environment configuration, and **migrations** (a way of managing database structure changes). That groundwork matters as much as the code itself.

## The Project

Modern systems are often split into two parts:

- The **front end** — the interface a person interacts with, sending requests for data
- The **back end** — the system that receives those requests, processes them, and sends data back

These two commonly communicate using an architectural style called **REST**. Communication is strictly one-directional: the front end initiates a request, and the back end responds. The back end never independently pushes data to the front end (that's a different pattern, outside the scope of this book).

Separating front end and back end this way — so they can even live on entirely different servers — has real benefits: two smaller, more focused codebases, and a back end that's cleanly accessible to *any* system that knows how to speak REST, not just one specific front end.

**This chapter builds only the back end** — but you'll still be able to test it directly, simulating requests as if they came from a front end.

## What Is REST?

**REST** (Representational State Transfer) is a widely used architectural pattern for exposing data to external systems in a predictable, reliable way. It's language-agnostic — a REST API built in Python can be consumed by a front end written in JavaScript, Swift, or anything else — as long as both sides follow REST's conventions.

At its simplest: a REST request is just an HTTP (or ideally HTTPS) request, similar to loading a normal web page, but following stricter, more predictable rules. A REST server exposes **endpoints**, each identified by a URL, and the *action* taken at that URL is determined by an **HTTP verb**.

### JSON

**JSON** (JavaScript Object Notation) is the plain-text data format used to send and receive data in most REST APIs. It's built from `key: value` pairs, wrapped in curly braces `{}`. Keys are always in quotes; string values are too.

```json
// A simple JSON object
{"name": "Roy", "age": 60}

// JSON containing an array (a list, marked with [ ])
{"name": "Roy", "age": 60, "hobbies": ["cooking", "reading"]}

// JSON with a nested object inside it
{
  "name": "Roy",
  "age": 60,
  "address": {
    "street_name": "old street",
    "phone_number": "0556325541"
  }
}
```

### HTTP Verbs

You use HTTP verbs constantly without thinking about it — every page load, every form submission relies on one. In REST, the verb is used *deliberately* to signal intent to the back end. Five matter most:

| Verb | Purpose |
|---|---|
| `GET` | Retrieve data |
| `POST` | Add new data |
| `PUT` | Replace an entry completely |
| `PATCH` | Update part of an entry |
| `DELETE` | Remove an entry |

These verbs are only *signals* — your code still has to actually implement the right behavior for each one. (In practice, some teams only implement `PATCH`, or only `PUT`, rather than both — how strictly you follow REST convention is a team decision.)

**The general shape of a REST call:**

```
VERB  url/resource/<optional-identifier>
{request body in JSON — only needed for verbs other than GET and DELETE}
```

**Example**, using a hypothetical `https://api-vehicles.com/vehicles` endpoint:

```
GET https://api-vehicles.com/vehicles/100
```
Retrieves the single vehicle with ID `100`.

```
GET https://api-vehicles.com/vehicles
```
Retrieves *all* vehicles (no ID specified).

```
DELETE https://api-vehicles.com/vehicles/100
```
Deletes the vehicle with ID `100`.

The next three all require a JSON body describing the data to change or add:

```
PUT https://api-vehicles.com/vehicles/100
{"car": "ford", "color": "red", "production_year": 2015}
```
Completely replaces the record at ID `100`.

```
PATCH https://api-vehicles.com/vehicles/100
{"car": "ford", "color": "red"}
```
Updates only the specified fields of the record at ID `100`.

```
POST https://api-vehicles.com/vehicles
{"car": "ford", "color": "red", "production_year": 2015}
```
Creates a brand-new record.

### REST Routes in Flask

Flask handles the details of matching a URL to the right function for you. You mark a function as an entry point using a decorator (an annotation placed just above the function):

```python
@app.route('/hello')
def hello():
    return 'Hello'
```

This tells Flask: "when a request comes in for `/hello`, run the `hello()` function."

### HTTP Status Codes

Every HTTP request returns a **status code** — a number telling the requester whether things went well. In REST, this code should reflect the outcome of the *underlying operation*, not just whether the server received the request successfully.

For example: if you request data that simply doesn't exist, the request itself reached your system fine — but the *result* is "not found," so the correct response is a `404`, even though nothing technically went "wrong" with the request itself. A `200` means the request was received and handled successfully — it doesn't guarantee the data you wanted was actually found.

### HATEOAS

**HATEOAS** — "Hypermedia As The Engine Of Application State" — is a core REST principle. It means your responses shouldn't just contain raw data; they should also describe what actions are available *on* that data. This decouples the client from needing to hard-code knowledge of your API's URL structure — the server tells the client what it can do next.

One common use is **pagination** — when a request returns part of a larger dataset (say, 10 of 100 results), the response can include links to the next and previous pages, so the client doesn't need to construct those URLs itself.

**Example response, including HATEOAS links and pagination:**

```json
{
  "id": 100,
  "car": "ford",
  "color": "red",
  "production_year": 2015,
  "links": [
    {
      "rel": "remove",
      "href": "https://mysite/vehicles/100",
      "method": "DELETE"
    },
    {
      "rel": "self",
      "href": "https://mysite/vehicles/100",
      "method": "GET"
    }
  ],
  "pagination": {
    "current_page": 2,
    "next_page": "https://mysite/vehicles?page=3",
    "previous_page": "https://mysite/vehicles?page=1"
  }
}
```

*(In our case, the Dockerfile below runs this step automatically when the image is built — you won't need to run it by hand.)*

**`app/Dockerfile`:**

```dockerfile
FROM python:3.7.5-slim

RUN mkdir -p /var/app/flask_app
WORKDIR /var/app/flask_app

COPY requirements.txt /var/app/flask_app
RUN pip install --no-cache-dir -r requirements.txt

COPY . /var/app/flask_app

ENV FLASK_APP ../app.py

CMD exec gunicorn -w 1 -b :8000 app:app --reload
```

**Walking through what each line does:**

1. `FROM python:3.7.5-slim` — start from a minimal Python 3.7.5 base image.
2. `RUN mkdir -p ...` and `WORKDIR ...` — create and switch to the working directory that matches the volume mount in `docker-compose.yml`.
3. `COPY requirements.txt ...` then `RUN pip install ...` — install dependencies *before* copying the rest of the code. (This ordering matters for build speed: Docker only re-runs this install step when `requirements.txt` itself changes, not every time you edit your actual code.)
4. `COPY . ...` — copy the rest of your local project files into the container.
5. `ENV FLASK_APP ../app.py` — tells Flask where the app entry point lives. It's only needed when running migration commands (which live one directory below `app.py`, hence the `../`).
6. `CMD exec gunicorn -w 1 -b :8000 app:app --reload` — starts Gunicorn, listening on port `8000`, running the Flask `app` object found in `app.py`. The `--reload` flag automatically restarts the server whenever your code changes — **convenient for development, but not something you'd want enabled in production**, where reloads should be triggered deliberately.

## Migrations

**Migrations** automate the creation and modification of your database structure. Rather than manually building tables in Adminer (as in the previous chapter), you'll define your tables in code — this makes your schema portable to any new environment and trackable over time, right alongside your application code. The same objects you define here will also become your ORM models later.

**Step 1: Create the database.** Log into Adminer (`http://127.0.0.1:8080`, or the equivalent IP-based URL on Windows) and create a database named `users`.

**Step 2: Write the migration file.**

**`app/database/migration.py`:**

```python
from flask import Flask
from flask_sqlalchemy import SQLAlchemy
from flask_migrate import Migrate, MigrateCommand
from flask_script import Manager

app = Flask(__name__)
app.config['SQLALCHEMY_DATABASE_URI'] = 'mysql+mysqlconnector://root:root@flask-db/users'

db = SQLAlchemy(app)
migrate = Migrate(app, db)
handler = Manager(app)
handler.add_command('db', MigrateCommand)


class Users(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    name = db.Column(db.String(128))
    surname = db.Column(db.String(128))
    identity_number = db.Column(db.Integer())


if __name__ == '__main__':
    handler.run()
```

**Breaking this down:**

- The first four lines import the libraries this file needs.
- The **database connection string** — `mysql+mysqlconnector://root:root@flask-db/users` — tells SQLAlchemy how to reach your database:

  | Part | Meaning |
  |---|---|
  | `mysql+mysqlconnector` | The connection protocol/driver |
  | `root:root` | `username:password` |
  | `flask-db` | The database container's name |
  | `users` | The database name |

- **SQLAlchemy** is the **ORM** (Object-Relational Mapper) you'll use — it lets you interact with your database using Python classes instead of writing SQL directly. Generally a good default choice, though for very complex databases, raw SQL sometimes remains preferable — you'll see both approaches in this chapter.
- The `Users` class *is* both your migration definition **and** your ORM model, used later to actually query the table. It defines an auto-incrementing primary key `id`, two string columns (`name`, `surname`), and one integer column (`identity_number`).

### Step 1: A Reusable Database Connection (`initialise.py`)

Right now, the database connection string lives directly inside `migration.py` — but `app.py` will need the exact same connection logic. Rather than duplicating it, pull it into a shared, reusable class.

**`app/initialise.py`:**

```python
class Initialise():
    def db(self, app):
        app.config.from_object("config.Config")
        app.config['SQLALCHEMY_DATABASE_URI'] = 'mysql+mysqlconnector://{}:{}@{}/{}'.format(
            app.config["DB_USERNAME"],
            app.config["DB_PASSWORD"],
            app.config["DB_LOCATION"],
            app.config["DB_DATABASE"]
        )
        app.config['SQLALCHEMY_TRACK_MODIFICATIONS'] = True
        return app
```

**What's happening here:** `app.config.from_object("config.Config")` loads settings from a `Config` class — the lowercase `config` refers to the *filename* (`config.py`), and the capitalized `Config` refers to the *class* inside it. This gives you one single, central place where database credentials live, instead of scattering them across multiple files.

### Step 5: The Anatomy of an Endpoint

Before writing the first real endpoint, here's the general pattern every endpoint follows — whether raw SQL or ORM:

```python
@app.route('route/to/endpoint/<identifier_variable>', methods=['GET'])
def function(identifier_variable):
    # ... logic here, typically returning JSON
```

- The `@app.route(...)` line defines the URL path and which HTTP verb(s) it responds to.
- `<identifier_variable>` in the route captures part of the URL (like a specific user's ID) and passes it into the function as an argument. `POST` requests never use this, since they're creating something new rather than referencing an existing one.

**A critical detail for raw SQL: preventing SQL injection.**

Whenever you write raw SQL yourself, you must guard against **SQL injection attacks** — a security topic covered in more depth later in the book. For now, the key defense is **parameter binding**: never insert user input directly into a SQL string. Instead:

```python
sql = text('SELECT * FROM banking WHERE id = :id AND code = :some_code')
result = db.engine.execute(sql, id=10, some_code=111).first()
```

The `text()` function wraps your SQL string and enables `:placeholder` syntax. Each `:name` is a safe placeholder — the actual values are passed separately when the query executes, so user-supplied data is never directly concatenated into the SQL string itself (which is what makes injection attacks possible in the first place).

Now, add the `POST` endpoint along with a shared `hateoas()` helper function:

**Add to `app.py`:**

```python
def hateoas(id):
    return [
        {
            "rel": "self",
            "resource": "http://127.0.0.1:8000/v1/users/" + str(id),
            "method": "GET"
        },
        {
            "rel": "update",
            "resource": "http://127.0.0.1:8000/v1/users/" + str(id),
            "method": "PATCH"
        },
        {
            "rel": "delete",
            "resource": "http://127.0.0.1:8000/v1/users/" + str(id),
            "method": "DELETE"
        }
    ]


@app.route('/v1/users', methods=['POST'])
def post_user_details():
    try:
        data = request.get_json()
        sql = text('INSERT INTO users (name, surname, identity_number) values (:name, :surname, :id_num)')
        result = db.engine.execute(
            sql,
            name=data['name'],
            surname=data['surname'],
            id_num=data['identity_number']
        )
        return json.dumps({"id": result.lastrowid, "links": hateoas(result.lastrowid)})
    except Exception as e:
        return json.dumps('Failed. ' + str(e)), HTTPStatus.NOT_FOUND
```

**Walking through it:**

- `hateoas(id)` builds the list of related actions (per the HATEOAS principle discussed earlier) — you'll attach this to every response.
- `@app.route('/v1/users', methods=['POST'])` — this endpoint responds to `POST` requests at `/v1/users`. The `v1` prefix is deliberate: it leaves room for a `v2` version later, using the ORM instead of raw SQL, without the two implementations colliding.
- `request.get_json()` reads the JSON body sent with the request.
- The `INSERT` query runs with bound parameters, exactly as described above.
- `json.dumps(...)` builds the JSON response; the returned tuple's second element (`HTTPStatus.NOT_FOUND` in the error path) sets the actual HTTP status code returned to the client.

**Testing this endpoint in Postman:**

1. Open a new request tab, set the verb to `POST`.
2. Click **Body** (only available once the verb isn't `GET`), select **raw**, then choose **JSON (application/json)** from the format dropdown.
3. Enter a JSON body, for example:

   ```json
   {"name": "John", "surname": "Connor", "identity_number": "902308"}
   ```

4. Click **Send** — you should get back a `200` status with the new record's `id` and its HATEOAS links.

### Step 7: The PATCH Endpoint

`PATCH` is slightly more involved, because the set of fields being updated varies from request to request — the update query has to be built dynamically:

```python
@app.route('/v1/users/<user_id>', methods=['PATCH'])
def patch_user_details(user_id):
    data = request.get_json()

    # Because the number of fields being updated can vary, we build
    # the "SET column = :column, ..." portion of the query dynamically.
    update_string = ''
    for key in data:
        update_string += key + '=:' + key + ','
    update_string = update_string[:-1]  # remove the trailing comma

    try:
        data['id'] = user_id
        sql = text('UPDATE users SET ' + update_string + ' WHERE id = :id')
        result = db.engine.execute(sql, data)
        return json.dumps({
            "id": user_id,
            "links": hateoas(user_id)
        })
    except Exception as e:
        return json.dumps('Failed. ' + str(e)), HTTPStatus.NOT_FOUND
```

**What's happening in the loop:** for each field in the incoming JSON, we append `fieldname=:fieldname,` to a string, then trim the final trailing comma. So a body of `{"name": "newname"}` produces the SQL fragment `name=:name`, while `{"name": "x", "surname": "y"}` produces `name=:name,surname=:surname`. The `data` dictionary itself is then passed straight into `execute()` as the source of bound values — including the `id` we added to it just before running the query.

**Testing:** open a new Postman tab, set the verb to `PATCH`, target a specific user's URL (e.g. `/v1/users/1`), and send a body containing just the fields you want to change:

```json
{"name": "newname"}
```

Confirm the change worked by re-running the `GET` endpoint for that same ID.

### Step 8: The DELETE Endpoint

The simplest of the four — like `GET`, it only needs an ID from the URL, with no request body:

```python
@app.route('/v1/users/<user_id>', methods=['DELETE'])
def delete_user_details(user_id):
    try:
        sql = text('DELETE FROM users WHERE id=:id_num')
        result = db.engine.execute(sql, id_num=user_id)
        return json.dumps('Deleted'), HTTPStatus.OK
    except Exception as e:
        return json.dumps('Failed. ' + str(e)), HTTPStatus.NOT_FOUND
```

*(No `hateoas()` call here — once a record is deleted, there's nothing left to `GET`, `PATCH`, or `DELETE` again, so there's nothing meaningful to link to.)*

**Test it** the same way as the others — set the verb to `DELETE` in Postman, target the ID you want removed, and send. Confirm with a follow-up `GET` request (it should now return a "not found" result).

You now have a fully working REST API supporting Create, Read, Update, and Delete.

## The ORM Version

Now let's build the same four endpoints again — this time using SQLAlchemy's ORM instead of raw SQL. Three things change overall:

1. Routes move from `/v1/...` to `/v2/...`, so both versions can coexist.
2. The `PATCH` endpoint gets *simpler* — no manual query-building needed; you just hand the ORM your data and let it handle the update.
3. All raw SQL is replaced with ORM method calls.

**Add to the bottom of `app.py`:**

```python
@app.route('/v2/users/<user_id>', methods=['GET'])
def get_user_details_orm(user_id):
    try:
        result = db.session.query(Users).filter_by(id=user_id).first()
        return json.dumps({
            "name": result.name,
            "surname": result.surname,
            "identity_number": result.identity_number,
            "links": hateoas(user_id)
        }), HTTPStatus.OK
    except Exception as e:
        return json.dumps('Failed to retrieve record. ' + str(e)), HTTPStatus.NOT_FOUND


@app.route('/v2/users', methods=['POST'])
def post_user_details_orm():
    try:
        data = request.get_json()
        user = Users(name=data['name'], surname=data['surname'], identity_number=data['identity_number'])
        db.session.add(user)
        db.session.commit()
        return json.dumps({
            "id": user.id,
            "links": hateoas(user.id)
        }), HTTPStatus.OK
    except Exception as e:
        return json.dumps('Failed to add record. ' + str(e)), HTTPStatus.NOT_FOUND


@app.route('/v2/users/<user_id>', methods=['PATCH'])
def patch_user_details_orm(user_id):
    try:
        data = request.get_json()
        db.session.query(Users).filter_by(id=user_id).update(data)
        db.session.commit()
        return json.dumps({
            "id": user_id,
            "links": hateoas(user_id)
        })
    except Exception as e:
        return json.dumps('Failed to update record. ' + str(e)), HTTPStatus.NOT_FOUND


@app.route('/v2/users/<user_id>', methods=['DELETE'])
def delete_user_details_orm(user_id):
    try:
        user = db.session.query(Users).filter_by(id=user_id).first()
        db.session.delete(user)
        db.session.commit()
        return json.dumps('Deleted'), HTTPStatus.OK
    except Exception as e:
        return json.dumps('Failed. ' + str(e)), HTTPStatus.NOT_FOUND
```

*(Note: `Users` here refers to the ORM model class defined in `migration.py` — make sure it's imported or otherwise accessible from `app.py`.)*

**What changed in each endpoint, and why:**

**GET:**
```python
result = db.session.query(Users).filter_by(id=user_id).first()
```
`Users` is your ORM model. `filter_by(id=user_id)` narrows the query, and `.first()` takes just the first matching row. The result is a Python *object* — you access its fields with dot notation (`result.name`), rather than dictionary-style keys.

**POST:**
```python
user = Users(name=data['name'], surname=data['surname'], identity_number=data['identity_number'])
db.session.add(user)
db.session.commit()
```
You construct a `Users` object directly from the incoming data, `add()` it to the current database session, then `commit()` to actually persist it to the database. Nothing is saved until `commit()` runs.

**PATCH:**
```python
db.session.query(Users).filter_by(id=user_id).update(data)
db.session.commit()
```
Just two lines — compare this to the manual string-building the raw-SQL version needed. The ORM's `update()` method accepts the incoming data dictionary directly and figures out the rest.

**DELETE:**
```python
user = db.session.query(Users).filter_by(id=user_id).first()
db.session.delete(user)
db.session.commit()
```
Fetch the record, mark it for deletion, then commit the change.

### ORM vs. Raw SQL: Which Should You Use?

The ORM version is noticeably shorter and easier to read — especially for `PATCH`. But it isn't automatically the better choice in every situation. For complex queries involving many joins or specialized SQL features, raw SQL can end up being clearer and more efficient than fighting an ORM's abstractions to express the same thing. As with most technology choices in this book, it comes down to planning and matching the tool to the problem.
