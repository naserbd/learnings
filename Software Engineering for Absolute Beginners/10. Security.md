# Security

Securing your code is one of the most important parts of being a software developer. This chapter isn't about teaching you to break into systems — it's about making you aware of the common attack angles, so you can defend against them.

A quick terminology note: people who break into systems for malicious purposes are technically called **crackers**, not hackers ("hacker" originally just meant a skilled, curious programmer). Their motives vary — financial gain, defacing a site they dislike, stealing data, or simply bragging rights. **No target is too small.** Don't assume your system is too obscure or unimportant to be worth attacking.

Attackers benefit enormously from the internet's anonymity, and they often use automated tools that can probe your system around the clock, for months, without you noticing anything. On a small site the author built a few years back, installing a web application firewall (**ModSecurity**, or "modsec") revealed malicious access attempts arriving like clockwork — one every 5 minutes, clearly automated — from day one, despite the site running quietly for a year beforehand without any visible incidents.

**A sobering mindset to carry into this chapter:** your job is to build software. A cracker's job is to understand systems and find their weaknesses. Never assume you know more about attack techniques than someone dedicated to finding them. If your system has 1,000 input fields and just one is left unprotected, a patient attacker will likely find it eventually — so **everything** needs to be locked down, not just the obvious entry points.

For a deeper, continuously updated resource on this topic, **owasp.org** publishes an annual list of the top ten security threats — well worth bookmarking.

## Code-Level Security

### SQL Injection

**SQL injection** is one of the easiest ways to compromise a database — and, fortunately, one of the easiest to defend against. In fact, you already protected against it back in the Flask chapter, without necessarily realizing why it mattered so much.

**How the attack works:** an attacker submits a carefully crafted string — through a form field like a username or password — designed to alter the *structure* of your SQL query, rather than just supplying an ordinary value.

Imagine a `users` table with `username` and `password` columns, queried like this:

```sql
SELECT * FROM users WHERE username = 'peter' AND password = 'h2*K64e';
```

In your actual application code, both values come from user-submitted input:

```python
user_name = 'peter'
password = 'h2*K64e'
# The query gets built using these variables
```

**The key exploit tool: SQL comments.** In SQL, `--` comments out everything after it on that line — the database engine simply ignores it. An attacker can use this to "cut off" the rest of a query.

**What happens if the `user_name` value itself is crafted like this:**

```
user_name = " ' AND password != 'dfasdf' ;-- "
```

Assume this gets inserted directly into the query without protection. The resulting SQL becomes:

```sql
SELECT * FROM users WHERE username = '' AND password != 'dfasdf' ;-- ' AND password = password;
```

Because everything after `--` is now a comment, the query that actually *executes* is just:

```sql
SELECT * FROM users WHERE username = '' AND password != 'dfasdf'
```

**Walking through the exploit:** the injected string starts with a `'` to close off the username value early, then appends new SQL of the attacker's own choosing (`AND password != 'dfasdf'`), and finishes with `--` to comment out the rest of the original query — including the part that would have actually checked the password properly. The odds of a real password *accidentally* matching the string `'dfasdf'` are astronomically low, so `password != 'dfasdf'` is essentially always true — meaning this crafted input can return a real user record without ever knowing the correct password.

**How to protect against it: parameter binding.** You've already used this technique — it's the strongest defense against SQL injection.

```python
sql = text('SELECT * FROM users WHERE id=:id_num')
result = db.engine.execute(sql, id_num=user_id).fetchone()
```

**What makes this safe:** instead of directly inserting `user_id` into the SQL string, the query uses a **placeholder** (`:id_num`), and the actual value is passed in separately as a parameter. The database driver handles this value strictly as *data* — never as executable SQL syntax — so no matter how cleverly an attacker crafts their input, it can never restructure the query the way the raw-string example above did.

**One more piece of good news:** when using an **ORM** (as in the second, SQLAlchemy-based half of your REST API), this kind of escaping and binding happens automatically, behind the scenes.

> **Rule to live by: always use parameter binding, or escape your SQL properly, for every single field — no exceptions.**

### Cleaning (Validating) Variables

Validating incoming data isn't just a security measure — it also protects your system from crashing on malformed input in the first place.

**Example:** imagine a column storing a 3-character office phone extension. If your application doesn't enforce that constraint *before* the data reaches your database query, an attacker could submit something like:

```
ext = " 123'; DROP TABLE table_name -- "
```

If this somehow bypasses your SQL injection protections, it could drop an entire table — or worse, an entire database. But if your *application code* enforces "this value must be exactly 3 digits" **before** it's ever used in a query, malicious input like this never even reaches the database layer at all.

**A good practice:** keep your database's own size/type constraints and your application-level validation rules perfectly in sync. If a database column only accepts strings up to 10 characters, your validation logic should enforce that exact same limit — not a looser or mismatched one.

### Implementing Validation with Marshmallow

Your Flask app already includes **Marshmallow**, a library built for exactly this kind of validation.

**Setup:** create a `validation/` directory in your project root, with a `validation.py` file inside it (plus an `__init__.py`, to make it a proper Python package).

**`validation/validation.py`:**

```python
from marshmallow import Schema, fields
# Import built-in validators
from marshmallow.validate import Length, Range


class UserSchema(Schema):
    # Required, and must be shorter than 50 characters
    name = fields.Str(required=True, validate=Length(max=50))
    # Required, and must be shorter than 50 characters
    surname = fields.Str(required=True, validate=Length(max=50))
    # Required, and must be a positive integer
    identity_number = fields.Int(required=True, validate=Range(min=1))


class IDSchema(Schema):
    identity_number = fields.Int(required=True, validate=Range(min=1))
```

**What's happening here:** `Length` and `Range` are just two of Marshmallow's many built-in validators — plenty more exist for different needs. `UserSchema` validates a full set of user fields (`name`, `surname`, `identity_number`); `IDSchema` validates just an ID on its own, for endpoints that only need that.

**Using a schema to validate incoming data:**

```python
errors = id_schema.validate({'identity_number': user_id})
if errors:
    return json.dumps(str(errors)), HTTPStatus.BAD_REQUEST

errors = user_schema.validate(data)
if errors:
    return json.dumps(str(errors)), HTTPStatus.BAD_REQUEST
```

`validate()` is inherited from Marshmallow's `Schema` base class (which both `UserSchema` and `IDSchema` extend). You pass it a plain dictionary; it returns a dictionary of validation errors — empty if everything checks out, populated if something failed.

**Putting it all together, inside a real endpoint:**

```python
@app.route('/v1/user', methods=['POST'])
def post_user_details():
    try:
        data = request.get_json()

        errors = user_schema.validate(data)
        if errors:
            return json.dumps(str(errors)), HTTPStatus.BAD_REQUEST

        sql = text('INSERT INTO users (name, surname, identity_number) values (:name, :surname, :id_num)')
        result = db.engine.execute(
            sql,
            name=data['name'],
            surname=data['surname'],
            id_num=data['identity_number']
        )
        return json.dumps('Added'), HTTPStatus.OK
    except Exception as e:
        return json.dumps('Failed to add record. ' + str(e)), HTTPStatus.NOT_FOUND
```

Notice the order: **validate first, before the data ever touches the database.** If validation fails, the function returns immediately with a `400 Bad Request` — the insert never even gets attempted.

> **Rule to live by: every field should be validated, without exception.**

### Keeping Error Messages Private

Be careful about what your system reveals when something goes wrong. Attackers actively probe systems looking for ways to break them — and error messages are a goldmine of information: what web server you're running, what language or framework powers your backend, what CMS you're using. Each of these details is a small stepping stone toward a more targeted, effective attack.

**A real example of this going badly wrong:** a colleague once entered a perfectly legitimate password containing special characters into a login form. The system rejected some of those characters — and displayed an error message containing the **exact raw SQL query** the database had tried (and failed) to run. That single mistake revealed two things at once: the query wasn't using parameter binding at all, *and* exactly how it was structured — practically an instruction manual for crafting a working SQL injection attack, no guesswork required.

*(In this real case, the flaw was reported responsibly to the site's owners rather than exploited — a good reminder that finding a security hole doesn't give you permission to walk through it. Just because someone left their front door unlocked doesn't mean it's fine to let yourself in.)*

**The practical takeaway:** always return clean, human-readable, generic error messages to the user — never the raw underlying error your system actually generated internally.

### XSS (Cross-Site Scripting)

**XSS** is one of the most common attack vectors today. It happens when malicious code (typically JavaScript) gets injected into your system, and your system unintentionally lets a victim's browser execute it. A successful XSS attack can steal login credentials, redirect users to malicious sites, or worse.

**How it can happen:** imagine a form where users submit their name, and that name later gets displayed back on a page. If there's no protection in place, and someone submits this as their "name":

```html
<script>alert('hi there')</script>
```

...then instead of displaying that text as a name, the browser could actually **execute it as real JavaScript** — popping up an alert box. That specific example is harmless, but the same mechanism can be used to run code that quietly steals a visitor's session cookie and sends it to an attacker's server, effectively hijacking their login session.

**A real-world case:** at a previous company using a popular CMS that loaded visual page components from a database, an attacker managed to inject JavaScript into one of those components. From then on, every visitor loading that page got silently redirected to a different website — because the browser treated the injected script as legitimate, executable code belonging to the page.

### Defending Against XSS

There are two general schools of thought:

1. **Sanitize incoming data** — strip out anything that looks like malicious code before it's ever stored.
2. **Escape outgoing data** — treat all displayed data as plain text, never as executable code, at the moment it's rendered.

Because XSS attacks are triggered at the point where data gets *displayed*, escaping output is the more reliable defense of the two. **Escaping** means that no matter how cleverly crafted a piece of injected JavaScript is, it gets rendered on-screen as harmless, inert text — never actually executed.

**Sanitizing input, by contrast, is trickier to get fully right.** It requires understanding exactly how each piece of data will be used and stripping out anything potentially harmful — but overly aggressive sanitization can corrupt legitimate data, and there are countless variations of XSS attack strings, making it genuinely hard to claim you've caught every single one. If your sanitization logic misses even one obscure variant, it may quietly let something through — and you likely won't know until it's already been exploited.

*(Quick distinction: **sanitizing** actively strips out potentially malicious content from incoming data — or rejects it outright. **Validation** just confirms a value conforms to an expected format. They're related but different tools.)*

Many modern frameworks now escape output by default — which is a big help. Still, the responsibility ultimately falls on you to understand exactly how your framework handles this, and to make sure nothing slips through unescaped.

> **Rule to live by: escape every single field displayed in a browser. Know exactly how your framework handles output escaping.**

### CSRF (Cross-Site Request Forgery)

**CSRF** exploits the fact that your browser automatically attaches your login session (via cookies) to *any* request sent to a site you're logged into — even if that request didn't originate from the site itself.

**How the attack plays out:** imagine you're logged into a social network — call it "MyFriends" — and it's vulnerable to CSRF. You receive a tempting email with a link (say, promising adorable kitten photos). Clicking it opens a page that, instead of showing kittens, silently submits a hidden form directly to MyFriends — perhaps requesting a password change, a profile update, or a payment. Since your browser still holds an active MyFriends session cookie, it automatically attaches that cookie to the request, and MyFriends has no way to tell the request didn't genuinely come from you, on its own site.

**The defense: CSRF tokens.** A CSRF token is a unique value shared between the backend and the legitimate front end, regenerated with every request. Every real request from your own front end includes the current token; the backend verifies it matches, then issues a fresh one for next time. An attacker's forged page has no way to obtain this token in advance — it was never served to them — so their forged request fails the check and gets rejected outright.

**In short:** a CSRF attack relies on you being logged into the target system while visiting a completely unrelated, malicious page that quietly fires a request at that target system on your behalf, letting your own browser's session handle the "authentication" for the attacker.

> **Rule to live by: every form submission should include a CSRF token.**

### Session Management

Most frameworks come with session management built in — but "built in" doesn't automatically mean "airtight." A few practical safeguards worth checking on (with OWASP's site as a deeper resource):

- **Set a session timeout.** A session shouldn't stay valid indefinitely, or for days at a time.
- **Generate sessions server-side only** — never let a client dictate its own session identifier.
- **Prevent multiple simultaneous logins** under the same user ID, where it makes sense for your system.
- **Never put session data in the URL's query string (the `GET` parameters).** URLs are human-readable, often logged in plain text by servers and proxies along the way, and aren't protected the way HTTPS protects the actual request body. You never know who might end up with access to those log files down the line.
- **Provide a real logout function** that properly destroys the session or access token — not just one that hides the "logged in" UI while leaving the session itself valid.

At the core of most session systems: browsers use cookies to identify a logged-in user. If an attacker manages to steal that cookie, they effectively gain that user's login session — which is exactly why proper session controls matter so much.

## System-Level Security

Beyond your code, several system-level practices meaningfully reduce your overall risk of compromise.

### Keep Your Systems Up to Date

Every piece of your stack — Python, PHP, your web server, your OS, your CMS — should be tracked, and updated promptly whenever a security patch is released. Outdated, unpatched software with known vulnerabilities is one of the most tempting and straightforward targets for an attacker.

### Database Users

Database security protects against more than just malicious attackers — it also protects against your own developers accidentally causing damage.

- The **root** (master) database user's credentials must stay tightly restricted to a small, trusted group.
- **Developers should ideally work against a copy of the production database**, not the live one — protecting against an accidental destructive query (a dropped table, a mistaken bulk delete).
- **Every developer should have their own individual login** — never shared credentials — with permissions limited to exactly what they need, and no ability to create new users.
- **Application/system accounts** used by your code need to be tightly restricted too. They generally should **not** be able to:
  - Create, delete, or update users
  - Drop an entire database
  - Drop individual tables (though some frameworks legitimately need this during migrations — use your judgment)
- Where possible, **tie a system's database credentials to the specific IP range** it connects from.
- **Use a separate username/password pair per database.** If one credential set ever leaks, you don't want that to hand an attacker access to *every* database you run.

### Ports

Every networked service is reachable via an IP address **and** a specific port. Common defaults: HTTP uses port `80`, HTTPS uses `443`, SSH uses `22` — and databases, FTP, and other services each have their own conventional ports too.

**The rule: only expose the ports you actually need.** The exact steps for opening and closing ports depend on your hosting setup — cloud providers like AWS each have their own tools and terminology for this, worth researching specifically for whatever platform you use.

### Docker Images

If you're not building an image entirely yourself, you're trusting whoever *did* build it. Docker images with hidden backdoors or serious security flaws genuinely exist in the wild. **Only use images from a trusted, reputable vendor** — not just any image someone happened to upload to Docker Hub, regardless of how many downloads it has.

### HTTPS

Any web application — REST APIs included — should use **HTTPS**, never plain HTTP. HTTPS encrypts communication between the front end and back end, protecting it from anyone eavesdropping on the connection in between.

### Password Policy

Passwords guard every entry point into your system, so don't treat them casually:

- Use a **different password for every system**.
- Require a real mix: uppercase, lowercase, digits, and non-alphabetic characters.
- Enforce this policy consistently, for everyone — and make credential-sharing an explicit violation of company policy.
- Consider **passphrases** — memorable strings of unrelated words — as a modern, easier-to-remember alternative to traditional complex passwords.

**Two real stories that underline why this matters:**

**Story one:** Investigating a feature for another department, the author noticed a colleague could view clearly sensitive information they shouldn't have had access to. It turned out they were logged in under the finance manager's own account — because the finance manager had simply *given* multiple people his password so they could get their work done. This was an enormous, entirely avoidable exposure of sensitive data, and it was shut down that same day, with proper individual permissions set up instead.

**Story two:** During a job interview, a candidate shared a website he'd built as a work sample. Out of curiosity, the interviewer tried logging into its admin panel with the username `admin` and a password like `1234` — and it worked immediately. This was inadvertently done from a work computer and office IP address (against company policy for unauthorized access, even accidentally), so it had to be disclosed to a manager afterward. The flaw wasn't held against the candidate — the site hadn't been built to the interviewing company's own standards, so it wouldn't have been fair to penalize him for it — but it's a vivid reminder of how catastrophic a weak default password can be.

**Passwords matter — don't skimp on them.**

## Social Engineering

The final topic isn't a coding technique at all — it's **social engineering**: exploiting human trust, rather than a technical flaw, to extract sensitive information.

This can take many forms: a phone call or email impersonating a company's head of IT, asking for a password under a plausible-sounding pretext; a spoofed email appearing to come from the CEO, requesting a building access code. Attackers running these schemes often do real homework beforehand — sometimes literally digging through discarded paperwork for clues.

**A vivid hypothetical:** suppose someone jots down a server's name, IP address, and open ports during an internal meeting, and that note ends up in the trash rather than being shredded. An attacker who finds it now has enough real, verifiable detail to convincingly pose as an auditor from your hosting provider — each true fact they can casually mention makes their pretext more believable, and gets them one step closer to convincing someone to hand over real access.

**A real internal test that illustrates this well:** a company's IT team once ran a planned exercise — creating a convincing-looking (but fake) email account for the system administrator, and a lookalike (but subtly wrong) website URL, then emailing the whole company asking everyone to "test" a new login system. No real credentials were ever captured — only a count of how many people attempted to log in was recorded. The number of people who fell for it was, in the author's words, staggering — a clear demonstration of how effective these attacks can be even against an otherwise security-conscious organization.

**How to defend against this:** since these attacks target people rather than code, the defenses are different too — regularly educating your team on what these attempts look like, and treating *all* sensitive information carefully, including physical notes (shred them, don't just bin them).

## Summary

This chapter covered a lot of ground quickly. Following these practices will meaningfully reduce your risk — but no system can ever be called *fully* secure with total confidence. Attackers can be remarkably creative, and it only takes one overlooked weak point to give them an opening.

**Quick reference:**

| Topic | Key Practice |
|---|---|
| **SQL injection** | Always use parameter binding — never insert raw input directly into a SQL string. |
| **Cleaning variables** | Validate every variable's type and size; reject anything that doesn't conform. |
| **Error messages** | Return generic, human-readable errors — never the system's raw internal error output. |
| **XSS** | Escape every piece of output displayed in a browser. |
| **CSRF** | Require a CSRF token on every form submission. |
| **Session management** | Enforce session expiry; never pass session IDs via the URL's `GET` string; follow OWASP's session guidance. |
| **System updates** | Patch every part of your stack promptly when security updates are released. |
| **Database users** | Grant only the exact permissions each user or system needs — nothing more — with unique credentials per user and per database. |
| **Ports** | Close every port you don't actively need; only expose what's in real use. |
| **Docker images** | Use trusted, vetted images — never choose one based on download count alone. |
| **HTTPS** | Always use HTTPS, never plain HTTP, for data in transit. |
| **Password policy** | Enforce strong, unique passwords per system; never share credentials. |
| **Social engineering** | Educate your team, and never carelessly discard sensitive information — even internal notes. |
