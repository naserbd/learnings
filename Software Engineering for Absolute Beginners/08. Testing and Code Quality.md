# Testing and Code Quality

No matter how big or small your system is, the integrity of your codebase matters. Basic habits — clear naming, tidy structure, careful coding — directly affect how easy the code is to maintain. Readable code is harder to accidentally break, and easier to safely change. Proper naming, in particular, prevents an entire category of bugs that badly named variables, functions, and classes tend to invite.

## A Cautionary Story

At a previous job, the author was asked to add a new feature to a system that had two main processing steps: **importing** and **ingesting**. Everything seemed fine — except a function named something like `file_import` actually performed *ingesting*. Not only was the name wrong, but the function was called from the *ingesting* section of the code, despite what its name implied.

The bug was caught quickly this time. But imagine a different scenario: someone working on the *import* section sees a function called `file_import`, assumes — reasonably — that it's the right tool for the job, reuses it, and then edits it because it "doesn't quite work" for their case. The knock-on effect: every other place that correctly relied on `file_import` (for ingesting) now silently breaks.

So — how do you prevent disasters like this? That's what this chapter covers.

## Overview of Code Quality Steps

There's no single silver bullet, but a handful of practices, used together, go a long way toward catching problems before they reach production:

1. **Be careful, and know your codebase.** This has no formal process — it's simply discipline. Don't change something you don't understand. If you're new to a codebase and asked to work on it, ask for time to actually get familiar with it first.

2. **Run automated tests.** Done well, automated tests cover your codebase broadly enough that a breaking change gets caught immediately. The catch: a lot of real-world codebases have thin or nonexistent test coverage — testing is often the first thing sacrificed under deadline pressure, and retrofitting tests onto a finished project is slow and painful.

3. **Peer review.** A second developer reviews your code for design quality, obvious errors (code is never *provably* bug-free), and — most importantly — whether it actually does what the business needs. This costs another developer's time, but it meaningfully raises confidence in what got built.

4. **User Acceptance Testing (UAT).** Typically run on a staging server that closely mirrors production, this simply asks: does the actual user agree this does what they needed?

Most of these steps assume you have at least one other person to work with — a teammate for reviews, a stakeholder for UAT. But it's worth understanding all of them, even if you're currently working solo, since you'll need them the moment you're not.

## Automated Testing

Automated tests set up a range of scenarios, run your code against them, and compare the actual output to the expected output. With solid coverage, you can make changes to your codebase, run the tests, and deploy with real confidence.

A useful side effect: **if you can't easily write a test for a piece of code, that's often a sign the code itself is poorly designed.** As covered in earlier chapters, functions and classes should have a single responsibility — the moment something does too much, it becomes hard to test, because there are too many paths through it to reasonably cover. Testing quietly enforces good design.

This chapter covers two kinds: **unit tests** and **integration tests**.

## Unit Tests

A **unit test** checks one specific, isolated piece of code — it doesn't care how that piece interacts with the rest of the system.

**Example scenario:** imagine a function `valid_age(date_of_birth)` that returns `True` or `False` depending on whether someone is old enough (say, 20+) for a credit card. To test it properly, you'd check:

- **Scenario 1:** a date of birth from December 12, 1978 → expect `True`
- **Scenario 2:** a date of birth from December 12, 2010 (assuming "today" is 2020) → expect `False`

If the function returns `True` and `False` respectively for these two inputs, the test passes.

### Are unit tests always worth writing?

Opinions differ. Some developers unit-test everything; others argue a lot of unit tests are a waste of time. Take a trivial function that just subtracts two dates — how likely is *that* to break?

The honest answer: not very likely to break through its own simple logic — but there's still a real risk that someone calls it with a badly formatted date, and the function has no validation to catch that.

**A practical rule of thumb:** the deciding factor isn't how complex a function is — it's **how important its correctness is to the business.** "Don't issue a credit card to a minor" sounds like exactly the kind of rule you want locked down with a test, however simple the underlying logic looks.

### Writing a Unit Test

Python's built-in `unittest` library is what we'll use. You create a class that inherits from `unittest.TestCase`, which gives you access to a set of **assertion** methods — functions that check whether an actual result matches an expected one.

```python
import unittest

"""
This is the function we want to test
"""
def is_even(number):
    if number % 2 == 0:
        return True
    return False

"""
This is our unit test
"""
class TestEven(unittest.TestCase):
    def test_is_even(self):
        self.assertEqual(is_even(2), True)

    def test_is_uneven(self):
        self.assertEqual(is_even(3), False)


if __name__ == '__main__':
    unittest.main()
```

*(In practice, your test code and the code being tested would usually live in separate files — they're combined here purely to keep the example simple.)*

### Breaking Down the Unit Test

- `is_even(number)` is the function under test. It uses Python's **modulus operator** (`%`), which returns the *remainder* of a division — `number % 2` is `0` for even numbers, and nonzero for odd ones.
- `class TestEven(unittest.TestCase):` — this class inherits from `unittest.TestCase`, which is what unlocks all the built-in assertion methods.
- Each individual test lives in its own method, and — as a rule of thumb — each test method should check just **one** thing.
- **Test method names must start with `test`.** This is how `unittest` knows which methods to actually run as tests, versus which are just helper methods you've added for your own use.
- `self.assertEqual(is_even(2), True)` — because `self` gives you access to everything `TestCase` provides, `assertEqual` compares the function's actual output against the value you expect. `2` was chosen deliberately here because we already know exactly what `is_even(2)` *should* return — that known-correct expectation is what makes the test meaningful.

A couple of more descriptive alternatives exist for boolean checks specifically:

```python
self.assertTrue(is_even(2))
self.assertFalse(is_even(3))
```

A few other useful assertions worth knowing about: `assertGreater`, `assertLess`, `assertGreaterEqual`, `assertIn`, `assertNotIn`. You don't need to memorize the full list — in most editors, typing `self.assert` and triggering autocomplete will show you everything available.


## Integration Tests

Where a unit test checks one isolated function, an **integration test** checks how several components work together — for example, how the database layer interacts with your business logic, or whether an email actually gets sent as part of a larger flow.

Integration tests are particularly good at catching **regression errors** — cases where a change to one part of the system unexpectedly breaks something else connected to it. They don't need to run the *entire* system end-to-end; just enough connected pieces to verify the integration itself works.

We'll write two integration tests, covering the `GET` and `POST` endpoints from the previous chapter — calling the underlying functions directly, rather than making real HTTP requests. After reading this section, writing similar tests for the remaining endpoints is a great exercise.

### Step 1: Making the System Testable

Right now, your application is tied to a single database — which is a problem, because your tests must **never** touch your production database. A properly set-up system should make it *impossible* for tests to accidentally reach production data. Testing and reviewing should always happen entirely before code reaches a production-ready state, and unit tests are typically stripped out of the code that actually gets deployed to production (removed from the production deployment — not from your development codebase).

*(Quick terminology note: **staging** is a separate environment, distinct from production, where people can safely test your system and where automated tests can run.)*

For test runs specifically, we'll use **SQLite** — a lightweight, file-free database that can be created entirely **in memory**. Tables get created and populated fresh for each test run, and the whole thing disappears once testing finishes — there's no risk of it colliding with, or corrupting, any real database.

### Step 2: The Test File Skeleton

Create `test.py`:

```python
import os, sys
sys.path.append(os.path.abspath(os.path.join(os.path.dirname(__file__), '..')))
basedir = os.path.abspath(os.path.dirname(__file__))

import unittest
import app
import json
from http import HTTPStatus


class TestAPIMethods(unittest.TestCase):
    pass  # setup and test methods go here


if __name__ == '__main__':
    unittest.main()
```

The `sys.path.append(...)` line ensures Python can find modules one directory up, regardless of where the test is actually run from. `import app` pulls in your `app.py` — this gives the test access to your Flask app and its database configuration. `if __name__ == '__main__':` ensures `unittest.main()` only runs when this file is executed directly, not when it's imported elsewhere.

### Step 3: Setting Up a Fresh Test Database

Add a `setUp` method to the `TestAPIMethods` class — `unittest` automatically runs this before every single test method, giving each test a clean slate:

```python
def setUp(self):
    self.db_uri = 'sqlite:///' + os.path.join(basedir, 'test.db')
    app.app.config['TESTING'] = True
    app.app.config['WTF_CSRF_ENABLED'] = False
    app.app.config['SQLALCHEMY_DATABASE_URI'] = self.db_uri
    self.app = app.app.test_client()
    app.db.create_all()

    cmd = "DROP TABLE IF EXISTS users"
    app.db.engine.execute(cmd)

    cmd = """
        CREATE TABLE users
        (id INTEGER PRIMARY KEY AUTOINCREMENT, name TEXT,
         surname TEXT, identity_number INTEGER)
    """
    app.db.engine.execute(cmd)

    cmd = "INSERT INTO users (name, surname, identity_number) VALUES ('John', 'Connor', 200509)"
    app.db.engine.execute(cmd)
```

**What's happening here:**

- `self.db_uri = 'sqlite:///...'` builds a connection string pointing to a local SQLite file, instead of your real MySQL database.
- `app.app` refers to the Flask **app instance** created in `app.py` (the outer `app` is the *module*; `app.app` is the actual Flask object inside it). We're intercepting the app's existing database configuration and pointing it at this SQLite database instead.
- `app.db.create_all()` creates any tables SQLAlchemy knows about from your models.
- The `DROP TABLE`, `CREATE TABLE`, and `INSERT` statements guarantee a completely fresh, predictable `users` table — with exactly one known row — before every single test runs.

**Why bother rewriting the database connection at all?** Because your test code needs to run against an isolated, disposable database — not the real one — and this is the mechanism that redirects it there safely.

### Step 4: The First Real Test — `GET`

```python
def test_get(self):
    result = app.get_user_details(1)
    expected = json.dumps({
        "name": "John",
        "surname": "Connor",
        "identity_number": 200509,
        "links": [
            {"rel": "self", "resource": "http://127.0.0.1:8000/v1/users/1", "method": "GET"},
            {"rel": "update", "resource": "http://127.0.0.1:8000/v1/users/1", "method": "PATCH"},
            {"rel": "delete", "resource": "http://127.0.0.1:8000/v1/users/1", "method": "DELETE"}
        ]
    }), HTTPStatus.OK
    self.assertEqual(result, expected)
```

We call `get_user_details(1)` **directly** — not through an actual HTTP request — and compare the result to what we already know is in the (freshly reset) test database. We can safely assume the test row has primary key `1`, since `setUp` recreates the table from scratch before every test. This is a clean example of testing integrated components (the endpoint function plus the database) without needing to spin up a full HTTP request/response cycle.

## A Design Problem, and Some Refactoring

So far, so good — but what about testing `POST` or `PATCH`? Looking at the existing `post_user_details` function (and its ORM counterpart) reveals a problem called **tight coupling** — different parts of the system depending directly on each other in ways they don't actually need to.

```python
@app.route('/v1/users', methods=['POST'])
def post_user_details():
    try:
        data = request.get_json()
        sql = text(
            'INSERT INTO users (name, surname, identity_number) values (:name, :surname, :identity_number)'
        )
        result = db.engine.execute(sql, data)
        return json.dumps('Added'), HTTPStatus.OK
    except Exception as e:
        return json.dumps('Failed. ' + str(e)), HTTPStatus.NOT_FOUND
```

This single function is bound directly to three separate concerns: Flask's `request` object, SQLAlchemy's `db.engine`, and the `json` formatting of the response. To test it as-is, you'd have to fake (a technique called using **test doubles**) the return value of `request.get_json()`, just to exercise the SQL logic buried inside it. That's a strong signal the design itself needs improvement — not just the tests.

### Splitting It Into Three Focused Functions

**1. `create(post_data)`** — handles just the SQL insert logic, taking data as a plain parameter instead of pulling it from the Flask request directly. This makes it reusable *and* directly testable, with no need to fake anything Flask-related.

**2. `execute(sql, data)`** — wraps the actual database call. Every raw-SQL function in your app should route through this. The payoff: if you ever switch database engines or libraries, there's exactly **one** place to update, instead of hunting down every direct `db.engine.execute(...)` call in your codebase.

**3. `return_message(message)`** — formats the outgoing response. JSON isn't the only valid REST response format (XML is another example) — if that ever needs to change, there's one function to touch, and every endpoint using it benefits automatically.

```python
def create(post_data):
    sql = text(
        'INSERT INTO users (name, surname, identity_number) values (:name, :surname, :identity_number)'
    )
    return execute(sql, post_data)


# The return value here is useful for retrieving metadata about the
# operation — for example, the id of the row that was just inserted.
def execute(sql, data):
    return db.engine.execute(sql, data)


# Returns data in JSON format
def return_message(message):
    return json.dumps(message), HTTPStatus.OK
```

**The refactored endpoint**, now much thinner:

```python
def post_user_details():
    try:
        # Get the details from the request object as usual
        data = request.get_json()

        # Delegate the actual insert logic to create() —
        # we don't need its return value here
        create(data)

        return return_message('Added')
    except Exception as e:
        return json.dumps('Failed. ' + str(e)), HTTPStatus.NOT_FOUND
```

Notice that `post_user_details` no longer knows or cares *how* `create()` talks to the database, or *how* `return_message()` formats its output — it just orchestrates the pieces. That's the real benefit of decoupling: each function has exactly one job, and each is testable on its own terms.

### Testing the Refactored Code

```python
def test_post(self):
    result = app.create({"name": "Peter", "surname": "Watson", "identity_number": 511247})
    result = app.get_user_details(result.lastrowid)
    expected = json.dumps({
        "name": "Peter",
        "surname": "Watson",
        "identity_number": 511247,
        "links": [
            {"rel": "self", "resource": "http://127.0.0.1:8000/v1/users/2", "method": "GET"},
            {"rel": "update", "resource": "http://127.0.0.1:8000/v1/users/2", "method": "PATCH"},
            {"rel": "delete", "resource": "http://127.0.0.1:8000/v1/users/2", "method": "DELETE"}
        ]
    }), HTTPStatus.OK
    self.assertEqual(result, expected)
```

`create(...)` returns an object with a `.lastrowid` attribute — the primary key of the row that was just inserted. We use that ID to immediately fetch the row back via `get_user_details`, and confirm the result matches what we expect. (Note that this test's inserted row ends up with ID `2`, not `1` — the setup row from `test_get`'s scenario already occupies ID `1` in a freshly reset table, and this insert is the second row added.)

That's the whole refactor: `execute()` can now be reused by *every* raw-SQL function in the app, and `return_message()` can format the response for *every* endpoint — and both are now fully testable in isolation.

## The Downsides of Automated Testing

### Tests Can Create a False Sense of Security

Automated tests never prove your code is bug-free — only that current functionality behaves as the tests expect. Here's a real example of how that gap can bite you.

Around 2015, a piece of code generated **GUIDs** — unique identifier strings, typically formatted as groups of 8-4-4-4-12 hexadecimal characters, like `8711a51c-f18c-42c3-8d4b-85d485d5d33d`. This code was unit tested, since GUIDs mattered for the system.

At some point, the GUID format broke — instead of `8-4-4-4-12`, it started producing `8-4-4-12`, missing one group entirely. On investigation, it turned out the *unit test itself* had been "fixed" to match the broken output, rather than the underlying bug being fixed. A developer had mistakenly assumed the *original*, correct GUID format was the bug, and "corrected" both the code and the test to match their incorrect assumption. The test then passed reliably — while quietly validating the wrong thing.

This particular case wasn't catastrophic, but it easily could have been. **The lesson: a passing test only tells you the code matches the test's expectations — not that those expectations were ever right in the first place.**

**A related point: test your edge cases, not just the easy path.** A hundred tests using clean, convenient data tell you almost nothing if none of them touch the awkward cases — how your payment logic handles a leap year, or what happens when a name longer than your database column allows gets submitted. Without edge-case coverage, a sense of security from "100 passing tests" can be entirely false.

### Time Pressure Is Real

Writing tests takes time — and as deadlines close in, it's often the first thing developers are tempted to cut. If a deadline is genuinely at risk, a reasonable compromise: write **one test that must pass, and one that must fail**, for each aspect you care about most. That's a much thinner safety net than full coverage, but it's still far better than nothing — and it gives you *something* to catch obvious regressions. Just make sure to schedule real time afterward to properly finish the test suite.

Tests also take time to *run*. In-memory databases like SQLite help keep this fast, but even so, large test suites against realistic data volumes can take a while — one system the author worked on had ~100 tests each taking around 2 seconds, adding up to over 3 minutes per full run. It can feel slow in the moment, but the confidence it buys is generally worth the wait.

## Peer Reviews

Peer review is invaluable for code quality — though it does require having at least a small team. Two common approaches: **peer review** (reviewing code directly) and **walk-throughs** (a more formal presentation-style review).

### Peer Review

In its simplest form: a colleague reviews a discrete, focused piece of your work — checking coding standards, obvious errors, algorithm quality, whether the code actually runs, and — critically — **whether it solves the actual problem it was meant to solve.**

That last point matters more than it might seem. A reviewer who wasn't involved when the work was assigned may genuinely not know what the feature is *supposed* to do — and if the original requirements were even slightly ambiguous, the developer and reviewer can walk away with two different mental models of "done." If a reviewer ever doubts whether even flawless-looking code actually solves the intended problem, that's worth raising immediately — a review that passes without catching this kind of misunderstanding can let a real problem slip all the way to production.

**What the developer submitting work should prepare:**

- A clear description of what the change is supposed to achieve
- A reasonably small, focused scope — a 1,000-line review is unlikely to get a genuinely thorough pass
- A note on any install steps, system updates, or database changes the reviewer needs to be aware of
- Basic usage instructions, especially for a newer or less familiar system

**What the reviewer should check:**

- Adherence to coding style
- Whether automated tests were written
- Whether the algorithms are reasonably efficient and free of obvious, serious errors
- Any potential security weaknesses
- Whether the code actually solves the business problem it was meant to

### Walk-Throughs

A more formal process: schedule a meeting with relevant stakeholders, share the necessary context beforehand, and walk through the feature step by step — live — while stakeholders critique and suggest changes, and you take notes.

## Staging Environment and UAT

This is the last stop before production. By this point, the code has been written, automated tests have run, and a peer review has happened. Now it moves to a **staging environment** — a setup that closely mirrors production, running against its own staging database where data can be freely created, edited, and deleted without any risk to real systems.

On staging, stakeholders — anyone with a genuine interest in the feature, whether that's coworkers, a product owner, or the feature's actual end users — can try the software directly and confirm it meets their expectations. Once they're satisfied, the code is cleared for production.

**One informal but important rule: the developer should not be the one performing the User Acceptance Test.** Developers are notoriously bad at objectively testing their own work — it's easy to unconsciously test only the paths you already know work. UAT should be conducted by the actual people the feature was built for. It's a valuable, often mandatory step — but keep in mind it mostly won't catch *regression* errors (things that broke elsewhere) unless the tester happens to stumble onto them directly; that's still primarily the job of your automated tests.

## Wrapping Up

You now have a working toolkit for building confidence in a codebase before it ships: careful, deliberate coding habits; unit and integration tests; a design mindset that favors decoupled, testable functions; peer review; and staging/UAT as a final real-world check. Each of these topics goes considerably deeper than what's covered here — your understanding of them will keep growing the more you encounter them in practice.
