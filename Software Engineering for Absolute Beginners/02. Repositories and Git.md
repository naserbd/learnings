# Containerizing Your Environment

This chapter is a required stepping stone — the next chapter builds directly on it, so it's worth taking your time here.

## Why Containers Changed Everything

Think about a few problems that used to plague software teams:

- Code that works fine on your machine but breaks the moment it's deployed somewhere else.
- Rigging your own laptop with extra software just to *approximately* recreate what staging or production looks like.
- A new team member spending their first few days just installing the development environment, piece by piece.
- Deploying code, only to find a dependency is missing on the server.

Virtualization — and especially **containerization** — solved all of these. In a modern setup, a new hire mostly needs two things: access to your **container orchestration file** (which describes how your containers fit together) and access to the code repository. That's it.

This chapter covers containerization. The next one will cover repositories.

## What Are Containers?

Here's a simple way to picture it: a container lets you run a mini, isolated "server" on your own computer, dedicated to one job. You could have one container running your database and a separate one running your web server, with the two set up to talk to each other. Put a few of these together, and you effectively have a small network of servers running on your laptop — without eating up much in the way of resources.

A more precise way to describe it: a container is an **isolated environment for running your code**. It isolates the application and its dependencies, but *not* the operating system's kernel — that part is shared across every container on the machine. This shared-kernel design is exactly why containers are so lightweight and can start in under a second.

A couple of things worth knowing about how containers behave:

- They run **headless** — no graphical interface, just the command line. (The *application* running inside might have its own user interface, like a website. The container itself never does.)
- Because they don't need to boot an entire operating system, they start almost instantly.

### Containers vs. Virtual Machines

It's useful to know roughly how containers differ from virtual machines (VMs), even though you don't need to master the internals right now.

| | Virtual Machine | Container |
|---|---|---|
| **Isolation** | Complete — includes its own OS kernel | Partial — shares the host's kernel |
| **Size** | Large (a full simulated computer) | Small |
| **Startup time** | Slow (booting an OS) | Near-instant |
| **How many can run at once** | Few, due to resource cost | Many |

Because containers are so much smaller and share the host's kernel, you can pack far more of them onto a single machine, and spin them up or shut them down quickly, without the overhead a VM requires.

## Docker

The container technology we'll use throughout this book is **Docker** — by far the most widely adopted tool of its kind, and one that has fundamentally changed how software gets built, shipped, and run. (Other tools exist too, such as **Podman** and **containerd**, but Docker remains the standard starting point for most developers.)

When you package an application using Docker, developers call this **"dockerizing"** the application.

### Why Dockerize Your Application?

- **Better staging/production parity.** Your local environment can match your production environment far more closely than any manual setup could.
- **Dependencies travel with the code.** You never have to wonder whether a production server has the right libraries installed — they're baked into the container.
- **Try tools without installing them.** Want to experiment with MongoDB? Run it in a container instead of installing it directly on your machine.
- **Effortless setup on new machines.** Handing your whole system to someone else — a teammate, a new server, a friend's laptop — becomes dramatically simpler.

### One Word of Caution

Containers trade off some security compared to virtual machines, precisely *because* they share the host's kernel rather than being fully isolated. Because of this:

- Build your own Docker images carefully and deliberately.
- Don't blindly pull prebuilt images from sources you don't trust — especially in any environment where security matters.

The upside is consistency: if you configure your Docker image correctly, it will produce the *exact same* container every single time — whether you're running it on your own laptop, a friend's computer, or a cloud provider like AWS.

## The Three Core Pieces of Docker

Docker revolves around three building blocks, and you'll generally work with them in this order:

1. **Dockerfile** — the instructions
2. **Docker image** — the built result
3. **Docker container** — the running instance

```
Dockerfile  --(build)-->  Docker Image  --(run)-->  Docker Container
```

### 1. The Dockerfile

A Dockerfile is essentially a **recipe** — a plain text file listing the exact steps Docker should follow to build an image. It typically specifies:

- Which **base image** to start from (this could be an operating system, an application that already includes an operating system, or even an image you built yourself previously)
- What commands to run on top of that base
- How to bring in files from your local project
- What extra software to install

You're taking a starting point and extending it, step by step.

A few of the most common instructions you'll see inside a Dockerfile:

| Instruction | What it does |
|---|---|
| `FROM` | Sets the base image to build on top of |
| `COPY` | Copies files from your machine into the image |
| `RUN` | Executes a command during the build (e.g. installing software) |
| `CMD` | Specifies the command to run once a container starts |

### 2. The Docker Image

There are two ways to build a Docker image. We'll focus mainly on one of them, but it's worth understanding both.

**Method A: Building step by step, by hand**

You start from an existing base image — say, an Ubuntu image that already has PHP and Python installed. You spin up a container from it, and then work inside that running container directly: installing new software, removing what you don't need (for example, removing Python and installing Go instead). Once it's set up the way you want, you save that container's current state as a brand-new image.

This works, but it isn't easy to share or reproduce reliably — someone else following your steps by hand might get a slightly different result.

**Method B: Building from a Dockerfile (the method this book uses)**

Instead of manually clicking and typing your way through a container, you write every step down in a Dockerfile, then hand it to Docker's `build` command. Docker reads the file from top to bottom and constructs the image for you automatically.

The big advantage: because every step is written down, you can share the Dockerfile itself, and anyone who builds from it will get an *identical* result.

A finished image is inert — you can't run commands inside it or interact with it directly. Think of it as a packaged template: a bundle of files containing everything needed to create a container. Images can also be uploaded to and shared through a **Docker image repository** (covered later in the book).

### 3. The Docker Container

The image's whole purpose is to produce a **container** — the actual running instance of your software, executing your code and responding to input.

A container exists in one of two states: **stopped** or **running**. You can also "step inside" a running container (this is called *mounting* it) and interact with it much like you would a separate server.

**Put simply:**
- Build an **image** from a **Dockerfile**.
- Run that image to create a **container**.

### Short-Lived vs. Long-Lived Containers

Containers don't all behave the same way once they're running:

- **Long-lived containers** stay alive continuously, waiting for requests — a web server is the classic example. These need to be explicitly stopped ("killed") when you're done with them.
- **Short-lived containers** run just long enough to complete a task, then shut themselves down. They typically spin back up fresh the next time they're needed.

This distinction trips up a lot of beginners. If you're expecting to see a container running permanently, but it was actually designed to be short-lived, you might check your container list, see nothing running, and assume something's broken — when really, it just finished its job and exited exactly as intended.

You'll encounter and inspect both types later in this chapter.

## A Final Reminder

The internet has no shortage of prebuilt Docker images ready to download and use — but availability isn't the same as trustworthiness. Don't assume a prebuilt image is safe just because it's popular or convenient. In any environment where security is a real concern, build your own image from a Dockerfile you understand and control.
