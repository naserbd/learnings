# Object Calisthenics, Coding Styles, and Refactoring

This chapter is about *how* you write code — not about solving problems, but about making your solutions easy for other humans to read. It covers three related topics:

1. **Object calisthenics** — guidelines for writing more readable, maintainable code
2. **Coding styles** — formatting rules for how your code should visually look
3. **Refactoring** — the practice of improving code you've already written

## Why Readability Matters

Harold Abelson, author of *Structure and Interpretation of Computer Programs*, put it best:

> "Programs must be written for people to read, and only incidentally for machines to execute."

That's the theme of this whole chapter. Writing code isn't just about getting the right answer — it's about writing it in a way another person (including future-you) can actually understand.

Here's a useful thought experiment: imagine two developers independently write code to solve the *exact same* problem, using the *exact same* algorithm (an **algorithm** just means the sequence of steps used to solve a problem). Will their code look identical? Almost never. Sometimes the two solutions will look so different that you wouldn't guess they solve the same problem at all.

Experienced developers will always have opinions about which style they prefer — that's simply part of the job. Your code *will* be reviewed and critiqued, sometimes harshly, sometimes gently, and often positively too. That's normal, and it's not something to take personally.

The good news: you don't have to figure out "readable code" from scratch. There are established guidelines that will noticeably improve your code's clarity if you follow them — object calisthenics and standard coding styles among them. And once your code is written, **refactoring** — revisiting and improving it — is a normal, healthy part of the process, not a sign you did something wrong the first time.

One piece of career advice before we dive in: when you're starting out, take feedback seriously, no matter who it comes from — junior or senior. Try to leave your ego out of your code. Evaluate criticism on its merits, and when it's valid, apply it. This is one of the fastest ways you'll grow as a developer.

## Part 1: Object Calisthenics

**Object calisthenics** is a set of nine guidelines for writing more readable, maintainable code, proposed by Jeff Bay in *The ThoughtWorks Anthology*.

A few important framing notes before we get into the rules themselves:

- These are **guidelines**, not strict rules. Follow them closely where you can — but don't force them where they genuinely don't fit.
- If a rule *doesn't* seem to make sense for a piece of code, that's sometimes worth a second look: is the rule wrong here, or is your code just too tangled to apply it cleanly? Often, it's a sign the code itself could use a rethink.
- This chapter covers **8 of the 9 rules**. The ninth is skipped here, as it tends to cause more confusion than clarity for beginners.

### Rule 1: Limit indentation to one level per method

Every extra level of indentation inside a method usually means another nested control structure (an `if` inside an `if`, for example) — and every nested structure multiplies the number of possible paths your code can take. Fewer indentation levels means a simpler, more predictable, more readable method.

**Example — comparing two versions of the same logic:**

```python
def more_indentation(number, name):
    if number > 10:
        if name == "John":
            print("Number is bigger than 10 and Name is John")
        if name != "John":
            print("Number is bigger than 10 and Name is not John")
        if name == "Peter":
            print("Number is bigger than 10 and Name is Peter")


def less_indentation(number, name):
    # Exit early. If number is 10 or lower, there's nothing left to do here,
    # so we return immediately instead of nesting everything below inside
    # an "if number > 10:" block.
    if number <= 10:
        return

    # Start with a default message, then override it only if needed.
    is_john = "Number is bigger than 10 and Name is not John"
    if name == "John":
        is_john = "Number is bigger than 10 and Name is John"
    print(is_john)

    if name == "Peter":
        print("Number is bigger than 10 and Name is Peter")


# Both functions produce identical output
more_indentation(11, "John")
less_indentation(11, "John")
```

**What's actually happening here:** `more_indentation` nests three `if` statements inside one outer `if`, so you have to mentally track *four* layers of conditions just to know whether a line will run. `less_indentation` uses an **early return** to eliminate the outermost layer entirely — the moment we know `number <= 10` doesn't matter, we exit and never think about it again. Everything after that point only has to consider one condition at a time, at one consistent indentation level.

### Rule 2: Avoid the `else` keyword

This might feel strange right after learning `else` — but the problem isn't that `else` is *wrong*, it's that it doesn't do much for readability. There are two common techniques to avoid it:

**Technique A — Return early.** Not everyone likes this approach, since critics argue that multiple return statements can make long functions harder to follow. But the counter-argument is just as strong: without early returns, you end up wrapping the "otherwise" code in extra `if` blocks anyway, adding complexity rather than removing it.

**Technique B — Default value, then override.** Set a variable to a sensible default, then only change it if a condition is met.

```python
number = 10

# The original version, using else
if number == 10:
    print(10)
else:
    print('not 10')


# Refactor 1: default value, overridden only when needed
message = 'not 10'
if number == 10:
    message = '10'
print(message)


# Refactor 2: early return — works especially well inside a function
def check_ten():
    if number == 10:
        return '10'
    return 'not 10'

print(check_ten())
```

**What's actually happening here:** In the original, you have to hold *both* branches (`if` and `else`) in your head to understand the full behavior. In Refactor 1, there's only one condition to reason about — the default already covers the "otherwise" case. In Refactor 2, the function exits as soon as its answer is known, so there's no need for an `else` at all.

### Rule 3: Wrap all primitives and strings

**Primitives** are basic built-in data types — booleans, integers, floats, strings. When a primitive represents a real business concept (not just a raw value) and has behavior of its own, it deserves to be wrapped in its own class rather than passed around as a plain string or number.

**Example — a password shouldn't just be a string:**

A password isn't just text — it has rules attached to it (must be complex enough, must not have been used before). If you store it as a plain string on a `User` class, that validation logic ends up scattered inside `User`, where it doesn't belong.

```python
class Password:
    """
    This class has one piece of data (the password string) and two
    behaviors: checking whether it's valid, and whether it's been
    reused. The actual validation logic is omitted here for brevity.
    """
    password = ""

    def __init__(self, password):
        self.password = password

    def is_valid(self) -> bool:
        # Password complexity check goes here
        return True

    def is_reused(self) -> bool:
        # Reuse-detection logic goes here
        return False
```

**What's actually happening here:** Any class that needs password validation — `User`, `AdminAccount`, whatever comes next — can now just use `Password` and rely on its `is_valid()` and `is_reused()` methods, instead of re-implementing (or worse, duplicating) that logic every time. The knowledge of "what makes a password valid" lives in exactly one place.

### Rule 4: Use only one dot per line

This isn't really about the `.` character itself — it's about *how deep* your code is allowed to reach into other objects. A class should only talk to:

- Its own methods and properties
- The objects it directly owns (its own "members")

It should **not** reach *through* one of its members to grab something that member's own member owns. If class `A` uses class `B`, and class `B` uses class `C`, then `A` should never talk to `C` directly — only to `B`, and let `B` decide how it uses `C` internally.

```python
class TotalCalculator:
    def get_value(self) -> int:
        return 100


# MemberFees has a TotalCalculator.
class MemberFees:
    def __init__(self):
        self.calculator_object = TotalCalculator()


# Club has a MemberFees.
class Club:
    def __init__(self):
        self.member_object = MemberFees()

    # Problem: Club is reaching straight through MemberFees to touch
    # TotalCalculator, a class it has no direct relationship with.
    def get_total(self) -> str:
        return self.member_object.calculator_object.get_value()


c = Club()
print(c.get_total())
```

**The fix** — give `MemberFees` its own method that hides `TotalCalculator` from the outside world entirely:

```python
class MemberFees:
    def __init__(self):
        self.calculator_object = TotalCalculator()

    def total_fees(self) -> str:
        return self.calculator_object.get_value()


class Club:
    def __init__(self):
        self.member_object = MemberFees()

    # Better: Club only ever talks to MemberFees. It has no idea
    # TotalCalculator even exists — and it doesn't need to.
    # (Note: `self.member_object.total_fees()` is technically two dots,
    # but the first dot — `self.` — just accesses the object's own
    # scope, so it doesn't count against this rule.)
    def get_total(self) -> str:
        return self.member_object.total_fees()

    # A worse alternative: creating MemberFees fresh here, instead of
    # in the constructor, technically uses only one dot — but it makes
    # it much harder to see, at a glance, which objects belong to Club.
    # Following the letter of the rule here actually hurts readability,
    # which is exactly the kind of situation where a guideline should
    # bend to common sense.
    def get_total_one_dot(self) -> str:
        member_object = MemberFees()
        return member_object.total_fees()


c = Club()
print(c.get_total())
print(c.get_total_one_dot())
```

**What's actually happening here:** `get_total()` (the good version) means `Club` only needs to know about `MemberFees` — if `TotalCalculator` is ever redesigned or removed, `Club` doesn't need to change at all. That's the real payoff of this rule: it's not about counting dots, it's about limiting how much any one class needs to know about the rest of the system.

### Rule 5: Don't abbreviate

Keep names to one or two clear words — no cryptic shortenings. If a name feels like it needs to be longer than that, treat that as a signal to reconsider what the class or function is actually responsible for.

Also, don't repeat the class name inside its own method names. If your class is called `Email` and it has a method for sending emails, don't call it `send_email()` — just call it `send()`. In context, it's already obvious what's being sent:

```python
email = Email()
email.send()
```

### Rule 6: Keep entities small

Classes and methods should do **one thing** (plus closely related sub-tasks). If an object is doing two unrelated jobs — say, watering plants *and* painting the house — split it into two separate objects.

As a rough guideline, object calisthenics suggests keeping classes under **50 lines**. That's not a hard limit — sometimes it's genuinely not possible — but the underlying principle (keep things small and focused) is always worth holding onto.

### Rule 7: Limit classes to two instance variables

Quick refresher: **instance variables** are the values a class receives when it's created (instantiated):

```python
def __init__(self, town, country):
    self.town = town
    self.country = country
```

Here, `town` and `country` are instance variables — set once at creation, and accessible on the object from then on.

This is one of the hardest rules to follow, and sometimes it's simply not achievable, even with good design. It connects to a broader principle called **low coupling** — keeping objects as independent from other objects as possible, so that a change to one doesn't ripple out and break several others. If your object depends on, say, six different instance variables (and therefore six other objects), it's worth asking: is this object trying to do too much?

### Rule 8: Use first-class collections

If a class contains a collection (like a list or tuple) *and* that collection has its own behavior (like a search or validation function), pull that collection out into its own dedicated class — one that holds nothing else.

```python
class WinningSequence:
    winning_sequence = (1, 3, 12, 14, 15)

    def validate(self, number) -> bool:
        for in_sequence in self.winning_sequence:
            if in_sequence == number:
                return True
        return False


# Lotto doesn't need to know *how* numbers are validated — it just
# asks WinningSequence to do it.
class Lotto:
    def __init__(self):
        self.sequence = WinningSequence()

    def is_winner(self, number: int) -> bool:
        return self.sequence.validate(number)


lotto = Lotto()

lotto_message = 'You do not have a winning number'
if lotto.is_winner(3):
    lotto_message = 'You have a winning number'
print(lotto_message)
```

**What's actually happening here:** `WinningSequence` is now a fully self-contained, reusable object — it can be validated, tested, and reused anywhere in the codebase, completely independent of `Lotto`. Meanwhile, `Lotto` is simpler: it doesn't need to know or care *how* a sequence is stored or checked, only that it can ask `WinningSequence` for an answer.

## Part 2: Refactoring

**Refactoring** pairs naturally with object calisthenics — and with good software design in general. It's a skill that improves with practice and with your growing understanding of design principles covered elsewhere in this book.

**Refactoring means rewriting parts of your existing, working code to improve it** — *without* changing what it actually does. This is an important distinction:

- **Refactoring** ≠ fixing a bug. Refactored code already produces the *correct* results — it might just be inefficient, hard to read, or hard to extend.
- **Bug fixing** deals with code that's producing *wrong* results or breaking under certain conditions. That needs fixing immediately, and it's usually obvious when it's happening.

Refactoring, by contrast, isn't always obvious. You typically only recognize the need for it by stepping back and honestly assessing what you've written — asking whether it's as clear and efficient as it could be. **Scalability** — how easily new features can be layered onto your code later — is one of the key things you're evaluating.

### Why and when to refactor

Refactoring is a routine, expected part of the job — not a rare event. Common reasons to do it:

- You wrote something quickly just to prove a concept works, or to pass a basic test.
- You've realized, in hindsight, that your first approach wasn't optimal.

**Refactor early, and refactor often.** It's much easier to clean up code you wrote an hour ago than code you wrote two months ago — the reasoning behind it is still fresh in your memory. Don't let refactoring debt pile up.

That said, you'll inevitably encounter old code that needs refactoring too — your own, a colleague's, or code that's been sitting untouched for a decade. When that happens:

- Schedule dedicated time for it — don't try to squeeze it in as an afterthought.
- Make sure you fully understand what the old code is *supposed* to do before you start changing it.

### What refactoring should improve

Refactoring should aim to improve three things:

1. **Readability** — how easily someone can understand the code
2. **Extendibility** — how easily new features can be added on top of it
3. **Reusability** — whether the code (or parts of it) can be reused elsewhere in the system

Of these three, **readability matters most**. Never underestimate it.

### The basic workflow

1. Plan your code.
2. Write your code.
3. Refactor your code.

When you get to step 3, look at readability, class design, algorithm efficiency, and any style-guide violations (covered next). And remember the golden rule: **refactoring changes how code is written, never what it actually does.**

## Part 3: Coding Styles

**Coding styles** govern the visual formatting of your code — spacing, capitalization, naming conventions, and so on. Unlike object calisthenics, these are treated as **rules**, not guidelines, within most teams and communities.

It's fair to feel skeptical about this at first. Does it really matter whether variable names are lowercase, or whether class names start with a capital letter? In practice — yes. Most developers find that within a few weeks of working under a consistent style, they start noticing immediately when *they themselves* deviate from it — and so do their teammates. Coding style adds nothing to how your code *runs*, but it adds a great deal to how easily it can be *read*.

Python's official style guide is **PEP 8**. It's worth keeping a PEP 8 cheat sheet handy as a quick reference while you're learning. The single most important thing about any coding style — whether it's PEP 8 or something you invent yourself — is **consistency**. Sticking with a widely recognized standard like PEP 8 also pays off beyond your own project: it makes it far easier to read other people's Python code, and to fit in quickly if you join a new team.

### Linting

**Linters** are tools that automatically detect style violations in your code. In VS Code, a linter called **Pylint** is typically available by default.

To access it: press `Ctrl+Shift+P` to open the Command Palette, then type `lint`. You'll see options like *Select Linter*, *Enable Linting*, and *Run Linting*.

You don't need to dive into configuring linters right away — it's more useful to first internalize the style rules yourself. But it's good to know linters exist, and that they can catch style issues (and sometimes fix them) automatically once you're ready to use them.

### Commenting Your Code

Comments explain *what* your code does and *why* — describing classes, methods, and variables in plain language, so other developers (and future-you) can understand how to use them. Comments should be written as complete sentences, using normal capitalization.

There are three main kinds: **block comments**, **inline comments**, and **docstrings**.

#### Block comments

Block comments sit *before* the code they describe, can span multiple lines, and should match the indentation level of the code beneath them.

```python
if is_member:
    # This sends an email to the member.
    # The configuration for this function can be found in the config directory.
    send_email()

if is_member:
    """This sends an email to the member.
    The configuration for this function can be found in the config directory.
    """
    send_email()
```

#### Inline comments

Inline comments sit on the *same line* as the code they refer to, and apply only to that one line. Use them sparingly — PEP 8's 79-character line limit leaves very little room for both code and a meaningful comment on one line.

```python
if is_member:
    send_email()  # This sends an email to the member
```

#### Docstrings

Docstrings live inside classes and methods, explaining their purpose in more depth. Several formatting standards exist:

- EpyText
- reST
- Google-style docstrings
- Numpydoc

Whichever you pick, prioritize readability and stay consistent throughout your project. It's also worth choosing a format your editor understands — most editors can display docstring content automatically as you use a class, which is a nice productivity boost. (Google-style docstrings tend to be considered especially readable, though the "right" choice ultimately comes down to preference and what your team already uses.)

A good docstring should explain the class or function's purpose, its parameters, its return value, and any relevant member variables.

### Maximum Line Length

PEP 8 recommends a limit of **79 characters per line**. This is usually one of the first rules a company relaxes — but it exists for good reasons:

- It lets you comfortably view multiple files side by side.
- It keeps code readable even in constrained environments, like a terminal-only server connection.
- It naturally discourages deep nesting, since heavily nested code tends to run long.

### Indentation

Use **spaces**, four per indent level. In VS Code, pressing Tab inserts four spaces automatically (some other editors need this configured manually).

You *can* use tabs instead of spaces — but never mix the two within the same file. Python 3 will refuse to run code that mixes tabs and spaces for indentation.

### Indenting Line Wraps

When a line exceeds the character limit, it needs to **wrap** onto the next line for readability. This applies to strings, comments, function declarations — really, anything that would otherwise run too long.

**Hanging parameters** — where the first parameter starts on the line below the function declaration — should be indented two extra spaces, to visually separate them from the function body:

```python
def this_is_a_function(
        variable1, variable2,
        variable3):
    function_body_code
    function_body_code
```

If your parameters start on the *same* line as the declaration, align any wrapped parameters with the first one:

```python
func = this_is_a_function(variable1, variable2,
                           variable3, variable4)
```

For a long `if` statement that needs to wrap, add extra indentation to visually separate the condition from the body that follows it:

```python
# Correct — the wrapped condition is indented, separating it
# clearly from the code block below
if (a == True and b == True
        and c == True):
    do_this()
```

```python
# Incorrect — the wrapped condition lines up with the code block,
# making it easy to mistake the condition for part of the body
if (a == True and b == True
    and c == True):
    do_this()
    do_that()
```

Arrays/lists can also wrap — both of these are considered correct:

```python
fruit_list = [
    'apples',
    'pears',
    'strawberries'
]

fruit_list = [
    'apples',
    'pears',
    'strawberries'
    ]
```

Long expressions with multiple operators can wrap between parentheses, with the operator starting the new line, aligned with the first value:

```python
total = (one
         + two
         + three
         )
```

Wrapping with consistent, intentional indentation is a habit worth building early — it pays off constantly in day-to-day readability.

### Blank Lines

Add one blank line before and after each function definition. You can also use blank lines to separate logically distinct sections within a function.

### Encoding

Your source files should be encoded in **UTF-8**. In VS Code: **File → Preferences → Settings**, then search for `Files: Encoding`. VS Code defaults to UTF-8, so you usually won't need to change anything.

### Imports

Imports belong at the **top of the file**, after any file-level docstrings or comments. Group them by origin, in this order:

1. **Standard library imports** — Python's built-in modules
2. **Related third-party imports** — packages you installed from outside sources
3. **Local application imports** — modules you wrote yourself, within this project

### Whitespace

A little extra whitespace might feel like it improves readability, but PEP 8 recommends against it in most cases:

- **Avoid** whitespace directly inside square, round, or curly braces.
- **Allow** whitespace after a comma.

```python
# Correct
my_data = [1, 2, 3, 4]
my_function(param1, param2)
```

Surround assignment, binary, and logical operators with whitespace:

```python
amount = 10 - 4 * 4
```

**Exception:** when `=` is used to set a named parameter or a default value inside a function signature or call, skip the surrounding spaces:

```python
# Correct — no spaces around '=' in a default parameter value
def email(email_address, from_address='server@server.com'):
    ...

# Correct — no spaces around '=' when calling with named parameters
email(from_address='do-not-reply@server.com',
      email_address='client@mail.com')
```

### Naming Conventions

There are several common naming styles used across programming languages:

| Style | Example |
|---|---|
| **PascalCase** | `MyClassName` |
| **camelCase** | `myVariableName` |
| **kebab-case** | `my-variable-name` |
| **snake_case** | `my_variable_name` |

Python primarily uses **PascalCase** (for classes) and **snake_case** (for almost everything else — always lowercase, with one exception: constants, covered below).

**Above all, choose readable names.** `banking_details` tells you far more than `details`. Vague names like `the_data` mean nothing to whoever reads the code next — including future-you. Avoid single-character variable names, especially `l`, `i`, and `o` — depending on the font, `l` can look like `1`, and `o` can look like `0`, creating easy-to-miss bugs. Clear, descriptive names make your code partly "self-documenting" — understandable to some degree even without comments or docstrings.

#### Packages and modules — `snake_case`

Prefer single-word names where possible. When a name needs two words, join them with an underscore, all lowercase.

#### Classes — `PascalCase`

Capitalize every word, with no underscores. When a class name includes an abbreviation, capitalize the *entire* abbreviation: `SSHKey` is correct; `SshKey` is not.

#### Methods — `snake_case`

All lowercase, words separated by underscores. Don't wrap method names in double leading/trailing underscores (`__like_this__`) — that pattern is reserved for Python's built-in "magic methods" (like `__init__`).

**Internal-use-only methods** get a *single* leading underscore (`_my_method`). This is a signal to other developers: "this method is for use inside this class only — don't build external code that depends on it, because it can change or disappear without warning."

**Preventing a method from being overridden:** prefixing a method with *two* leading underscores triggers Python's **name mangling** — Python internally renames it to include the class name, making it much harder for subclasses to accidentally override.

```python
class Security:
    pin_number = 12345

    def __print_pin(self):
        return self.pin_number


secure = Security()
print(secure._Security__print_pin())
```

Notice that from outside the class, you now have to call it as `_Security__print_pin()` — Python has mangled the name behind the scenes specifically to discourage external or subclass access.

#### Variables — `snake_case`

Same rules as methods: all lowercase, words joined with underscores, and never wrapped in double underscores (that's reserved for Python's magic methods — special methods the *language itself* calls automatically at specific moments, rather than ones you call directly. `__init__()`, the constructor, is the classic example: you never call it yourself, but Python calls it automatically every time you create a new object).

**Internal-use-only variables** get a single leading underscore, for the same reason as internal methods — signaling "don't rely on this from outside the class."

**Preventing a variable from being overwritten by a subclass:** prefix it with two leading underscores, which triggers the same name-mangling behavior as with methods:

```python
class Security:
    __pin_number = 12345


secure = Security()
print(secure._Security__pin_number)
```

*(Note: name mangling is triggered by two* leading *underscores — not trailing slashes or trailing underscores.)*

#### Constants — `SNAKE_CASE` (all caps)

Constants use underscores to join words, but in **uppercase**, distinguishing them at a glance from regular variables:

```python
TAX_RATE = 17
```

## Chapter Summary

**Readability is king.** Every time you write code, assume someone else will eventually need to build on top of it — and design it accordingly. Achieving that takes consistent, deliberate practice: applying object calisthenics, refactoring regularly, and following a consistent coding style, starting from your very first lines of code and tests.

Here's a scenario worth keeping in mind: a year from now, you might look back at code you wrote today. In that year, you may have written a hundred thousand new lines across parts of the system you haven't touched since. When you return to that old code, you might not remember it at all — even though you wrote every line yourself. You'll pick it back up quickly enough, but the clearer and more concisely it was written, the less relearning you'll have to do.

So write clean, well-commented, readable code — not just as a courtesy to your teammates, but as a gift to your future self. As Damian Conway put it:

> "Documentation is a love letter that you write to your future self."
