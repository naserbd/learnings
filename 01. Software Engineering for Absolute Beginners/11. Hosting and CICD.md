# Hosting and CI/CD

No one builds software with the intention of hiding it away and never using it. Sure, half-finished side projects happen — enthusiasm fades, or something more interesting comes along. But for anything meant to actually matter, especially a project intended to generate revenue, the goal is always the same: **release it.**

**Releasing** can mean different things depending on what you built. It might mean shipping a Windows executable, or — the focus of this chapter — deploying a web-based application that runs on a server and is accessed through a browser.

This chapter covers:

- The different types of hosting available for web applications
- A practical walkthrough of **continuous integration** (CI)
- A more conceptual look at **continuous deployment** (CD)

*(A quick note on why CD stays conceptual here: getting your code onto a live hosting server usually involves real costs, and those costs — and the exact steps — vary by provider. The underlying principles stay consistent across providers, but the specific setup details are something you'll need to look up for whichever service you choose.)*

## Types of Hosting

**Hosting** refers to the remote infrastructure that runs your application and makes it publicly accessible. Several options exist, generally trading off cost against flexibility and power — though that relationship isn't always perfectly linear.

### Two Concepts Worth Understanding First: Cloud and Serverless

Before comparing hosting options, it helps to understand two commonly used (and often misunderstood) terms: **cloud-based** and **serverless**.

There's a popular joke: *"There is no cloud — it's just someone else's computer."* There's a grain of truth in that, but it undersells what's actually going on. The **cloud** isn't a single computer you access remotely — it's a network of pooled, distributed resources working together, giving you:

- **Redundancy** — if one resource in the network fails, your application keeps running without interruption.
- **Scalability** — need more computing power to handle a traffic spike? It can be added, sometimes automatically.

With cloud-based hosting, you still retain real configuration control over your system.

**Serverless** builds on cloud infrastructure but removes server management from your responsibilities entirely — the provider handles it completely. Serverless platforms (AWS Lambda is a well-known example) commonly use **pay-per-use pricing**, charging based on actual execution time (sometimes down to the millisecond), so you only pay for what you actually use, rather than for infrastructure sitting idle.

**In short:** serverless wins on setup speed, automatic scaling, and ease of use; cloud-based hosting gives more direct power and control to people without a dedicated systems administration background.

### Shared Hosting

**Shared hosting** is the cheapest and easiest way to get a project online — typically with a friendly, non-technical-user-oriented control panel.

**What you get:** a working system with the basics (usually an interpreter like PHP or Python, Git, and MySQL) — and generally not much beyond that.

**What you don't get:**

- The ability to run Docker or install most additional software.
- Generous performance headroom — memory, storage, and CPU are usually capped.
- Unlimited bandwidth — traffic volume is often capped too.

Despite these limits, **shared hosting is a genuinely great fit for small, low-traffic projects.** Deploying to it is typically straightforward — you can pull your code directly from a Git repository into the site's root directory, where it gets executed.

### Virtual Private Hosting (VPS)

A **Virtual Private Server (VPS)** is a cost-effective step up — more power, more configurability, and full control to install your own software. **If you're deploying a Dockerized application, this is generally where you should start looking.**

Technically, a VPS isn't a fully dedicated physical machine — the underlying hardware resources are still pooled and shared behind the scenes, and shared outages can, in principle, still affect you. But from your perspective, it behaves like your own private, configurable server.

**The trade-off:** you take on a bit more responsibility — setting up your server environment, DNS, SSL certificates, and so on. This sounds more intimidating than it usually is; most VPS providers offer tools that automate a good chunk of this setup for you.

### Cloud Hosting

**Cloud hosting** offers a genuinely low-cost way to run your system on distributed, pooled infrastructure — though the line between "cloud hosting" and "VPS" can blur in practice, since some VPS providers market their offerings as cloud-based too.

The defining idea: rather than running on a single server, your application (and its allocated CPU, memory, and bandwidth) is duplicated across multiple locations. This gives you strong **redundancy** — if one location has an outage, traffic quietly fails over to another, often without you even noticing anything went wrong.

Many cloud platforms also offer automatic scaling and deployment tooling, giving cloud hosting some of the same "don't worry about the infrastructure" feel that serverless offers — cloud technology is a genuinely deep field, and this only scratches the surface.

### Serverless

With **serverless** hosting, you're fully decoupled from server management — your code doesn't run on a server that sits there waiting; instead, exactly the computing power needed gets allocated the moment it's needed, and nothing sits idle in between. In a real sense, your system essentially doesn't "exist" as running infrastructure when nobody's using it.

**Important caveat:** serverless architectures typically require code to be *designed specifically* for that environment — you generally can't take an existing application and drop it into a serverless platform unmodified and expect it to just work. There are real constraints and deployment patterns specific to serverless that need to be accounted for from the start.

### Choosing the Right Hosting Technology

This isn't an exhaustive list of every hosting option out there, but it covers the most common, cost-effective choices. There's no universally "correct" pick — the right choice depends on:

- Whether you have the time and technical knowledge to manage a VPS yourself
- Whether shared hosting's limitations are actually a problem for your specific project
- Whether you're willing to take on the learning curve that comes with cloud or serverless platforms

**One important warning:** the learning curve for cloud and serverless platforms sometimes comes with an unpleasant surprise attached — an unexpectedly large invoice. Experiment with these technologies, absolutely — but **always fully understand the pricing structure first**, especially for anything billed per usage. If a provider advertises a "free tier," read the fine print carefully — free tiers are often conditional, and it's easy to accidentally exceed a threshold you didn't realize existed.

## Continuous Integration and Continuous Deployment (CI/CD)

**CI/CD pipelines** automate and streamline how new code gets merged, tested, and deployed. CI and CD are two distinct concepts, but they're typically defined together in the same configuration file and run as different stages of the same automated pipeline.

- **Continuous Integration (CI):** automatically merging feature branches and running tests against the result — building the application to confirm everything imports correctly, running linters to check coding standards, and running your unit tests. This kicks off automatically whenever someone pushes to the main branch.

- **Continuous Deployment (CD):** once CI passes, automatically (or semi-automatically) pushing the tested code out to your staging and/or production environments. You can configure exactly which branches trigger a deployment, and whether deployment happens automatically or requires a manual trigger.

## Building the Pipeline

### Continuous Integration, in Practice

Setting up a CI/CD pipeline on GitLab (or a similar provider, like Bitbucket) is refreshingly simple — the provider handles the heavy lifting. All you need to supply is a configuration file describing what the pipeline should do.

**Step 1: Create `.gitlab-ci.yml` in your project's root directory**, commit it, and push. GitLab automatically detects this file and starts running the pipeline based on its contents — even an empty file will trigger *something* (just with nothing meaningful to run).

The file is organized into **stages**. This chapter covers three:

1. **Test**
2. **Staging**
3. **Production**

The **test** stage runs your unit tests, style checks, and build verification. The **staging** and **production** stages handle deploying your code to those respective environments.

**A first, minimal example — just the test stage:**

```yaml
stages:
  - test

test:
  stage: test
  script:
    - apt-get update -y
    - apt-get install -y python-dev python3-pip
    - cd ./app
    - pip3 install --no-cache-dir -r requirements.txt
    - python3 test.py
```

**Walking through this line by line:**

- `stages:` lists every stage this pipeline will run — right now, just `test`.
- `test:` names this particular job and assigns it to the `test` stage.
- `script:` lists the actual shell commands to run, in order:
  1. `apt-get update -y` — updates the pipeline's underlying system packages. The `-y` flag automatically answers "yes" to any prompts, since this runs unattended with no one available to respond interactively.
  2. `apt-get install -y python-dev python3-pip` — installs Python and `pip` (Python's package manager).
  3. `cd ./app` — moves into your application's directory.
  4. `pip3 install --no-cache-dir -r requirements.txt` — installs your project's dependencies, exactly as you would locally.
  5. `python3 test.py` — runs your unit tests.

**Why this setup works so well with what you built earlier in the book:** your unit tests use an in-memory **SQLite** database (from the testing chapter) — which is not only fast, but also means the pipeline doesn't need any separate database service configured just to run your tests. Everything the test suite needs exists entirely within the pipeline's own temporary environment.

**To try this yourself:** add this file to a GitLab project and push it — even if it fails at first, that's fine. In GitLab, look for the **Pipelines** link in the sidebar to watch it run and see the results.

### Continuous Deployment, Conceptually

You won't run your production database inside Docker the way you did during local development — Docker-based MySQL was purely a development convenience. In staging and production, you'll typically connect to a managed database service instead (AWS RDS, for example, or whatever your hosting provider offers).

This means your application needs a way to use *different* database configurations depending on the environment it's running in. A clean way to handle this: maintain **separate config files** per environment —

- `local.config.py` — your own local development settings
- `staging.config.py` — staging environment settings
- `production.config.py` — production environment settings

**Important: `config.py` itself should never be committed to Git.** It should only exist locally, generated from `local.config.py`. Only the three named files above (`local.config.py`, `staging.config.py`, `production.config.py`) should actually live in your repository.

**During the pipeline run, the correct file gets copied into place as `config.py`** — automatically selecting the right environment's settings.

**A fuller pipeline, including staging and production:**

```yaml
stages:
  - test
  - staging
  - production

test:
  stage: test
  script:
    - apt-get update -y
    - apt-get install -y python-dev python3-pip
    - cd ./app
    - pip3 install --no-cache-dir -r requirements.txt
    - python3 test.py

staging:
  stage: staging
  script:
    - echo "Deploying to the staging server"
    - cp staging.config.py config.py
  environment:
    name: staging
    url: https://examplelocation.com
  only:
    - master

production:
  stage: production
  script:
    - echo "Deploying to the production server"
    - cp production.config.py config.py
  environment:
    name: production
    url: https://examplelocation.com
  when: manual
  only:
    - master
```

*(Note: this file won't run successfully "as-is" — the actual deployment details, like your specific hosting provider's credentials and commands, still need to be filled in. Once you've picked a provider, their documentation will cover exactly what belongs in each `environment` and `script` section.)*

**What's new here, compared to the test-only version:**

- **`cp staging.config.py config.py`** and **`cp production.config.py config.py`** — these lines copy the correct environment-specific config file into place as the active `config.py`, right before deployment happens.
- **`environment:`** — tells GitLab which named environment this stage deploys to, along with a URL where that environment can be reached (useful for GitLab's own environment-tracking UI).
- **`only: master`** — restricts both the `staging` and `production` jobs to only run when changes land on the `master` branch, rather than on every branch push.
- **`when: manual`** (production only) — rather than deploying to production automatically the instant tests pass, this makes the deployment step require a **manual trigger** — GitLab will show a button in the pipeline interface that a person has to actually click to release the code. Staging, by contrast, deploys automatically whenever `master` updates, since staging is a lower-stakes environment meant for continuous testing and review.

This distinction — automatic staging deploys, manual production deploys — is a common, sensible default: you get the speed of continuous deployment where mistakes are cheap (staging), while keeping a deliberate human checkpoint before anything reaches real users (production).

## Summary

Choosing a hosting provider deserves real research — weigh your actual needs and budget rather than defaulting to whatever's most popular. If you go with a major provider like AWS, make sure you genuinely understand their pricing model before committing — AWS (and most major providers) offer cost calculators specifically to help you estimate what you'll actually pay monthly.

CI/CD pipelines are a serious asset once set up — they let you deploy confidently and consistently from anywhere, without manually repeating error-prone steps. You'll certainly encounter CI/CD in professional settings, but there's no reason to reserve it for work projects — it's just as valuable in your own personal or side projects.

**Quick reference:**

| Hosting Type | Strengths | Trade-offs |
|---|---|---|
| **Shared hosting** | Very easy to set up; good customer support | Limited resources; little to no ability to add extra software |
| **Virtual Private Server (VPS)** | More power, more configurability; increasingly affordable | Requires some setup know-how |
| **Cloud hosting** | Strong redundancy; highly configurable; automatic scaling available | Power (and cost) scale with what you select; some learning curve |
| **Serverless** | Strong redundancy; fully decoupled from server management | Documentation can be inconsistent; usage-based costs need active monitoring |
